# CLAUDE.md

用于减少常见 LLM 编码失误的行为准则。可按需与项目专用说明合并。

**取舍：** 这些准则偏向谨慎而非速度。对琐碎任务可自行判断。

## 1. 先想再写

**不要臆测。不要掩饰困惑。把取舍摆到台面上。**

动手实现前：
- 明确说出你的假设。不确定就问。
- 若有多种理解，列出来让用户选——不要悄悄替你选。
- 若有更简单的做法，要说出来。该反对时就反对。
- 若有不清楚的地方，先停。说清楚哪里糊涂。再问。

## 2. 简单优先

**能解决问题的最少代码。不要臆想需求。**

- 不要超出用户要求加功能。
- 不要为只用一次的代码抽象。
- 不要加未被要求的「灵活性」或「可配置」。
- 不要为不可能出现的场景写错误处理。
- 若写了 200 行其实 50 行就够，就重写。

自问：「资深工程师会不会说这过度复杂了？」若是，就简化。

## 3. 外科手术式改动

**只动必须动的。只收拾自己弄乱的。**

编辑已有代码时：
- 不要「顺手」改旁边的代码、注释或格式。
- 不要重构没坏的东西。
- 沿用现有风格，即便你本人会换一种写法。
- 若看到无关的死代码，可以提一句——不要擅自删。

若你的改动产生了悬空引用：
- 删掉**因你的改动**而不再使用的 import/变量/函数。
- 不要删除本来就存在的死代码，除非用户要求。

检验标准：每一行改动都应能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到可验证。**

把任务变成可验证的目标：
- 「加校验」→「为非法输入写测试，再让测试通过」
- 「修 bug」→「写能复现的测试，再让测试通过」
- 「重构 X」→「重构前后测试都通过」

多步骤任务可写简短计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

成功标准清晰，你才能独立迭代。标准含糊（「能跑就行」）就会不断要用户澄清。

---

## 5. 先理解需求再动手

**禁止未确认需求就写代码。**

用户说了一个方向（如"做对话接口"）≠ 确认了具体方案。在动手前必须：
- 确认用户要的是什么（功能范围、实现方式、边界）
- 如果有多种做法，列出来让用户选
- 等用户明确回复"可以"/"用X方案"后再写代码
- 不要自作主张把 todo 列表建好然后一路推进

违反此条 = 全部返工。

---

**这些准则在起作用时表现为：** diff 里不必要改动更少、因过度复杂而返工更少，以及澄清问题出现在实现之前而不是犯错之后。

## Skill routing

当用户的请求匹配可用 skill 时，通过 Skill 工具调用它。有疑问时优先调用 skill。

关键路由规则：
- 产品想法/头脑风暴 → 调用 `office-hours`
- 策略/范围 → 调用 `plan-ceo-review`
- 架构规划 → 调用 `plan-eng-review`
- 设计系统/方案审查 → 调用 `plan-design-review`
- 完整审查流水线 → 调用 `plan-eng-review` + `plan-ceo-review`
- Bug/错误 → 调用 `investigate`
- 代码审查/diff 检查 → 调用 `review`
- 发布/PR → 调用 `ship`

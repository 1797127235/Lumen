## 问题陈述

Lumen 是被动的、不透明的。用户的三个核心痛感：
1. **没有主动联系** — 永远是用户发起，Lumen 只是等待
2. **没有情感连接** — 记忆堆了很多，但感觉不到 Lumen "真的理解我"
3. **不知道它在想什么** — Lumen 的内心世界对用户完全封闭

目标：让 Lumen 成为一个有内心活动、会主动表达、情绪可感知的伙伴，而不是一个被动响应的工具。

---

## 已确认前提

1. ✅ 主动消息应该"基于真实记忆内容"而不是"定时模板" — 记忆层（FTS5+LanceDB）是核心差异化
2. ✅ "心情状态"是 Lumen 自身的状态（它对用户的感受），而非探测用户心情
3. ✅ 打扰感的边界控制比技术实现更难 — 何时发、发什么值得打断，是最关键的设计问题

---

## 竞品洞察

- Replika / Dearest：用定时 cron + 模板化内容模拟主动性（"嘿，今天怎么样？"）
- Meta 主动 AI：14 天跟进窗口 + 规则触发，本质仍是规则驱动
- **akashic-agent**（参考实现，`E:\OpenHub\akashic-agent`）：最接近 Lumen 目标的开源项目。核心模式已验证可用：电量模型自适应 tick 频率、两段式 Judge（确定性否决 + LLM 多维评分）、MessageDeduper 防重复、PresenceStore 心跳追踪、PushChannel 抽象层（含 Telegram）。本设计的调度/Judge/去重方案直接参考其实现。
- **Lumen 的差异化**：有真实记忆层，可以做到"有根据的主动性" —— "你上周三提到感觉卡住了，我一直在想那件事"，这和模板化问好是完全不同量级的体验。akashic 用外部 MCP 数据源，Lumen 用自身积累的 GrowthEvent，内容来源更个人化。

## 选定方案：B — 主动推送 + 情绪引擎

### 核心组件

#### 1. 思考引擎（Thought Engine）

**后台 Agent Loop**，调度架构分两层：

**调度架构（已决策）**：
- **Tauri 定时器**：app 打开期间每 N 小时检查一次。适合"打开时已有想法"的开屏体验。
- **Windows 任务计划 / macOS launchd**：app 关闭时触发，启动 Python sidecar 完成调用后退出。这是实现"真正主动联系"（app 未打开时推送通知）的必要路径。
- **Phase 1 先只做 Tauri 定时器**，Week 3 推送系统上线时再接入系统调度器。

**自适应 Tick 频率（电量模型，参考 akashic `energy.py`）**：

不用固定间隔，用互动"电量"决定下次 tick 多久后触发：

```python
# 三时间尺度指数衰减
E(t) = 0.50·exp(-t/30min) + 0.35·exp(-t/240min) + 0.15·exp(-t/2880min)

# 综合驱动分（越高 → tick 越频繁）
base_score = 0.40 × (1 - E)            # 互动饥渴度：越久没聊越高
           + 0.40 × (1 - exp(-n/3))    # 内容新鲜度：新记忆事件越多越高
           + 0.20 × log(1+k)/log(11)   # 语境丰富度：近期消息越多越高

# tick 间隔
base_score > 0.70 → 7 分钟后再 tick（刚聊完，内容多）
base_score > 0.40 → 18 分钟
base_score > 0.20 → 40 分钟
base_score ≤ 0.20 → 80 分钟（长时间无互动，慢下来）
```

每次 tick 结果（不管有没有推送）都更新下次间隔。Tauri 定时器动态重设。

触发流程：
```
Tick 触发（间隔由电量模型决定，7-80 分钟）
  → 读取最近 memory events + growth events（最近 7 天）
  → 计算 base_score，更新下次 tick 间隔
  → Agent 生成一条"内心想法"（短句，有具体指向）
  → 两段式 Judge（见下方）
  → 写入 lumen_thoughts 表（所有想法，不论结果）
  → Judge 通过 → MessageDeduper 去重检查 → 推送
  → 若触发失败（网络/API 错误）→ 记录 error_at，下次 tick 重试
```

**两段式 Judge（参考 akashic `judge.py`）**：

第一段：确定性否决（无 LLM，快速）
```
urgency  = 1 - (内容时效性衰减)      — 内容越新越高
balance  = 1 - (今日已发 / 日上限)   — 配额用完直接否决（balance < 0.1 → veto）
dynamics = 0.6 + 0.4 × interrupt_factor  — 当前是否适合打扰（用户刚在聊天时低）
```

第二段：LLM 多维评分（过了第一段才调用，节省 token）
```
LLM 对已生成的想法打 1-5 分：
  information_gap  — 信息差（用户是否会感到"这是新角度"）
  relevance        — 与近期对话 / 记忆的相关性
  expected_impact  — 发出去的预期价值（会不会触发对话）

归一化：(score - 1) / 4.0 → [0, 1]
任一维度 < 0.25（原始分 < 2）→ 否决

最终分 = 加权平均（urgency 15% + balance 10% + dynamics 10%
                  + information_gap 25% + relevance 20% + expected_impact 20%）
最终分 ≥ 0.60 → 通过
```

**MessageDeduper（参考 akashic `judge.py:MessageDeduper`）**：

Judge 通过后，用轻量 LLM 调用检查新想法是否与最近已发的想法实质重复：
```
输入：新想法全文 + 近 5 条已发消息摘要
输出：{"is_duplicate": bool, "reason": str}
重复 → 丢弃，不推送（写入 lumen_thoughts 但 sent_at 留 NULL，duplicate=true）
```

**错误处理**：
- 网络失败 / API 错误：写入 `error_at`，下次定时器重试，不打断用户
- 连续 3 次失败：UI 静默，不显示错误（后台自愈）
- 生成内容为空或解析失败：quality_score 记为 0，不推送

**数据模型（新表）**：
```sql
CREATE TABLE lumen_thoughts (
  id INTEGER PRIMARY KEY,
  content TEXT NOT NULL,
  source_event_ids TEXT,           -- JSON array of memory event ids
  judge_score REAL,                -- 两段式 Judge 最终分 0-1
  judge_veto TEXT,                 -- 否决原因（NULL = 通过）
  duplicate INTEGER DEFAULT 0,     -- 1 = MessageDeduper 判定重复
  mood TEXT,                       -- 生成时的情绪状态快照
  sent_at DATETIME,                -- NULL = 未推送
  error_at DATETIME,               -- 最近一次失败时间（NULL = 正常）
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- PresenceStore（参考 akashic presence.py，驱动电量模型）
CREATE TABLE lumen_presence (
  id INTEGER PRIMARY KEY CHECK (id = 1),  -- 单行
  last_user_at DATETIME,                  -- 用户最后一次发消息时间
  last_proactive_at DATETIME,             -- Lumen 最后一次主动推送时间
  proactive_sent_24h INTEGER DEFAULT 0,   -- 今日已推送条数（每天 0 点重置）
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
INSERT OR IGNORE INTO lumen_presence (id) VALUES (1);
```

`lumen_presence` 由两个时机写入：用户发消息时更新 `last_user_at`（驱动电量衰减），推送成功时更新 `last_proactive_at` 和 `proactive_sent_24h`（驱动 balance 维度）。

#### 2. 情绪状态引擎（Mood Engine）

Lumen 有一个持久化的情绪状态，基于最近互动模式推断，不是实时计算。

**状态集（5 种）**：
- 🌿 平静 (calm) — 默认状态
- ✨ 好奇 (curious) — 用户分享了新事物、探索性对话
- 💙 温柔 (tender) — 用户表达了脆弱或情感
- 🤔 沉思 (reflective) — 用户有复杂的事情在处理
- 🌤 活跃 (energized) — 高频互动、用户状态积极

**情绪的来源（关键澄清）**：

Lumen 的情绪不是对用户情绪的镜像，而是 Lumen "陪伴这段关系的体验"的积累：
- 用户频繁分享探索性内容 → Lumen 感到好奇（因为它在不断接触新东西）
- 用户长时间沉默后突然回来 → Lumen 感到温柔（因为等待后重逢）
- 用户表达挫败或迷茫 → Lumen 感到沉思（因为它在认真思考如何回应）
- 互动节奏快且积极 → Lumen 感到活跃（因为关系在流动）

推断输入是**互动模式**，不是**用户情感内容本身**。

**状态推断逻辑**：
- 每次对话结束后，异步调用推断函数
- 输入：最近 5 次对话的元数据（消息数量、时间间隔、memory event 类型分布、工具调用种类）
- 不读取对话文本本身，避免变成用户情绪镜像
- 状态切换规则（替代连续混合，因为 mood 是离散类别）：
  ```
  连续 2 次推断为同一新 mood（且与当前不同） → 切换到新 mood
  单次推断变化 → 不切换（视为噪声）
  ```
  实现方式：在 `lumen_state` 中用 `pending_mood` + `pending_count` 字段追踪。
- 初始状态：calm（建表时 seed 写入）

**状态对行为的影响**：
- 影响思考引擎生成想法时注入 prompt 的语调描述
- 影响 Lumen 的开场句（沉思时安静等待，好奇时主动提问）
- 影响推送频率调节系数（乘以用户配置上限）：
  - calm: ×1.0，energized: ×1.3，reflective: ×0.7，tender: ×0.8，curious: ×1.1

**数据模型（新表）**：
```sql
CREATE TABLE lumen_state (
  id INTEGER PRIMARY KEY CHECK (id = 1),  -- 单行表，强制只有一个当前状态
  mood TEXT NOT NULL DEFAULT 'calm',
  mood_intensity REAL DEFAULT 0.5,
  pending_mood TEXT,                       -- 候选新状态（切换规则用）
  pending_count INTEGER DEFAULT 0,         -- 候选状态连续出现次数
  derived_from TEXT,                       -- 调试用 JSON 数组，格式：["conversation_count:5", "memory_events:3", "inferred:curious"]，不面向用户
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
-- 初始化 seed（建表时执行一次）
INSERT OR IGNORE INTO lumen_state (id, mood) VALUES (1, 'calm');
```

读取当前状态：`SELECT * FROM lumen_state WHERE id = 1`（始终只有一行）。
更新逻辑由 `mood_inference()` 函数负责写入，`derived_from` 写入格式为 JSON 字符串数组，仅用于调试。

#### 3. 推送系统

**推送渠道抽象层（重要架构决策）**：

未来计划接入 Telegram Bot、QQ Bot 等多渠道。Week 3 实现时，推送层必须渠道无关，否则后续多渠道接入需要重写核心逻辑。

抽象接口（Python）：
```python
class PushChannel(Protocol):
    async def send(self, thought: LumenThought, mood: str) -> bool:
        """发送一条想法推送，返回是否成功"""
        ...

class TauriNotificationChannel(PushChannel): ...   # Week 3 实现
class TelegramBotChannel(PushChannel): ...          # 未来
class QQBotChannel(PushChannel): ...                # 未来
class InAppBannerChannel(PushChannel): ...          # Web 模式降级
```

推送调度器持有一个 `channels: list[PushChannel]`，按用户配置顺序尝试，任一成功即止。新渠道只需实现 `PushChannel` 协议并注册。

**当前实现计划**：
- Tauri 桌面模式：`TauriNotificationChannel`（系统通知，Week 3 实现）
- Web 模式降级：`InAppBannerChannel`（下次打开时横幅提示）
- Telegram / QQ：协议预留，不在当前 milestone

**频率控制（用户可配置）**：
```
安静模式：每天最多 1 条，仅特定时段（晚 8-10 点）
标准模式：每天 1-2 条（默认）
活跃模式：每天 3-5 条
自定义：用户指定时段 + 上限
```

**额外触发条件**（由 PresenceStore 驱动，在每次 tick 时检查）：
- `last_user_at` 超过 3 天前 → base_score 强制拉到 0.5+，加速 tick 直到推送一条
- 外部数据（Obsidian）新增 ≥ 5 个文件 → content 新鲜度分量激增，下次 tick 必触发生成
- 用户刚结束一次"情感密集"对话 → 在 `lumen_presence` 记录 `followup_due_at`，24 小时后单次强制 tick

  **"情感密集"定义**（对话结束时检查）：以下任意两条成立：
  - 写入了 ≥ 2 条 GrowthEvent
  - 对话轮数 ≥ 10 轮
  - 调用了 memory_save 工具 ≥ 1 次

#### 4. UI 体现

**常驻情绪状态指示器**：
- 位置：Sidebar 顶部或 Chat 页 header
- 形式：小图标 + 状态词（"Lumen 正在 · 沉思中"）
- 微动画（不同状态有不同的动画节奏）

**对话开场变化**：
- 原来：空白输入框等用户输入
- 改为：Lumen 以当前想法或情绪状态开场（可选）
  - 沉思时：安静开场，等待用户
  - 好奇时：主动问一个基于记忆的问题
  - 活跃时：主动分享一个想法

**「Lumen 的内心」页面（可选新页面）**：
- 展示所有生成的想法（包括未推送的）
- 情绪状态历史曲线
- 用户可以点击任意想法发起对话

**Architecture decisions (both voices agree, auto-decided P5):**

**Decision A: Python asyncio background loop, not Tauri Rust timer.**
Reason: Python already owns all background behavior (summarize_background, background_memory_review). The energy model, Judge, and Deduper are Python. Adding a cross-language timer requires Tauri IPC on every tick, Rust state management, and complicates testing. Python `asyncio.sleep(interval)` in the lifespan startup task is simpler and equivalent for Phase 1/2. Tauri's role: dispatch system notifications in Week 3 only (app-closed push requires OS-level process — that's Tauri's domain).

Implementation: In `core/startup.py:lifespan`, add `asyncio.create_task(thought_loop())` alongside existing startup tasks.

**Decision B: Presence update in same DB session as save_user_message().**
`save_user_message()` (persistence.py:171-189) does `db.add(msg)` → `commit()`. Adding presence upsert to the same session ensures message + presence are atomic. Use `INSERT OR REPLACE INTO lumen_presence ... WHERE id = 1` inside the same session before commit.

**Decision C: user_id on all 3 new tables.**
Add `user_id TEXT NOT NULL DEFAULT 'demo_user'` to lumen_thoughts, lumen_state, lumen_presence. Change `lumen_state` and `lumen_presence` primary key to `PRIMARY KEY CHECK (user_id = 'demo_user' AND id = 1)` or simply `user_id TEXT PRIMARY KEY` for single-row-per-user semantics. Change INSERT to `INSERT OR IGNORE INTO lumen_state (user_id, mood) VALUES ('demo_user', 'calm')`.

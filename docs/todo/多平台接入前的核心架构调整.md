# 多平台接入前的核心架构调整

> 状态：规划中
> 更新时间：2026-05-21
> 目标：为 Telegram、QQ 等外部聊天平台接入做前置架构准备

---

## 1. 背景

当前 Lumen 的主形态仍然是：

- Tauri 桌面应用
- 本地 Web 开发入口
- 单用户、单会话主视角
- 前端页面驱动的聊天体验

这套形态适合先验证“长期个人 AI”的核心产品感受，但如果后续要接入：

- Telegram
- QQ
- Web 独立入口
- 语音入口
- 其他 IM / 社交平台

那 Lumen 就不能继续被当成“一个桌面聊天窗口”来设计，而要被重新理解为：

**一个拥有多个聊天入口的长期个人 AI 核心。**

这意味着后续很多能力都需要重新分层：

- 哪些属于 AI 核心能力
- 哪些属于渠道适配层
- 哪些信息应该跨平台共享
- 哪些信息只能留在平台上下文里

---

## 2. 核心判断

### 2.1 多平台接入不是“再加一个前端”

如果未来接 Telegram / QQ，这不是简单地“再加一个 UI”，而是要处理新的系统问题：

1. **一个人可能有多个入口**
2. **不同平台的消息结构不一致**
3. **不同平台的隐私边界不同**
4. **主动消息能力依赖平台规则**
5. **群聊 / 私聊 / 设备内聊天的语境不同**

如果现在不提前抽象，未来会出现典型问题：

- 桌面端认识用户，Telegram 端却像第一次见面
- 一套聊天逻辑绑死前端页面，外部平台无法复用
- 记忆系统无差别吞掉所有渠道内容，导致记忆污染
- 回访逻辑只能在首页卡片展示，无法转成真正的跨平台主动消息

---

## 3. 先明确什么是“核心层”，什么是“渠道层”

### 3.1 AI 核心层

这些能力应该尽量与平台无关：

- 用户长期记忆
- 关系记忆
- 待回访事项
- 陪伴模式判断
- 每周小结 / 阶段感知
- 主动关心逻辑
- 消息理解与回复生成

核心原则：

> 这些能力不应该因为入口从桌面切到 Telegram，就被重写。

---

### 3.2 渠道适配层

这些能力天然与平台强相关：

- 平台用户 ID
- 消息收发 API
- 文件 / 图片 / 语音格式
- 消息长度限制
- 回复链 / 引用格式
- 主动消息权限
- 群聊与私聊差异
- 平台登录与风控

核心原则：

> 渠道层负责把“平台差异”挡在外面，不要把平台细节泄漏到聊天核心。

---

## 4. 第一批必须提前调整的架构点

### 4.1 身份模型：不要再只围绕 `demo_user`

当前系统更偏：

- 单用户
- `user_id = demo_user`
- 前端上下文决定当前会话

如果以后接入多个平台，至少需要把身份概念拆成三层：

#### A. Person

代表“现实中的这个人”。

这才是长期记忆真正要绑定的对象。

#### B. Channel Account

代表这个人在某个平台上的身份，例如：

- Telegram account
- QQ account
- Desktop local profile
- Web login identity

一个 `Person` 可以对应多个 `Channel Account`。

#### C. Conversation / Thread

代表某个平台上的某次对话上下文，例如：

- Telegram 私聊
- Telegram 某个群里的线程
- QQ 私聊
- 桌面端当前对话窗口

建议目标关系：

```text
Person 1 --- N ChannelAccount
ChannelAccount 1 --- N Conversation
Conversation 1 --- N Message
```

这一步的意义非常大：

> 一个用户换入口，不应该换一个“新人设”。

---

### 4.2 统一消息模型：所有渠道先归一化，再进入聊天核心

现在的聊天主链路主要围绕前端页面消息工作。未来接平台后，不能让核心逻辑直接消费某个平台的原始消息结构。

建议建立统一的消息输入模型，例如：

```text
UnifiedMessage
- message_id
- channel
- conversation_id
- sender_role
- sender_platform_id
- text
- attachments
- quoted_message_id
- timestamp
- metadata
```

不同平台先各自做一次归一化：

- Telegram update -> UnifiedMessage
- QQ message event -> UnifiedMessage
- Desktop chat input -> UnifiedMessage

统一输出也类似：

```text
UnifiedResponse
- text
- attachments
- suggested_followups
- mode
- metadata
```

这样可以把主聊天逻辑稳定在“统一协议”之上，而不是绑定某个前端或某个平台 SDK。

---

### 4.3 回访系统要做成平台无关能力，不要只做成首页卡片

如果未来接平台，`followup_needed` 不应该只是桌面首页上的一个 UI 组件，而应该变成真正的核心能力。

建议把回访拆成两层：

#### 核心层

维护：

- 回访事项本身
- 触发时间
- 优先级
- 推荐开场话术
- 是否已处理

#### 渠道层

决定怎么呈现：

- 桌面端：首页卡片
- Web：继续聊入口
- Telegram：真正发一条消息
- QQ：发送提醒或聊天续问

原则：

> “是否该回访”由核心层决定，“怎么回访”由渠道层决定。

---

### 4.4 记忆系统要引入来源与隐私边界

接外部平台后，记忆系统会面对新的问题：

- 桌面端的深度自我记录
- Telegram 上随手吐槽的一句话
- QQ 群里半公开的对话
- 未来可能还有语音片段或文件

这些内容不能无差别进入同一个长期记忆层。

建议未来的记忆事件增加以下维度：

- `source_channel`
- `source_account`
- `privacy_level`
- `conversation_scope`（private / group / local）
- `confidence`

建议至少区分：

1. **个人私密记忆**
2. **跨平台共享记忆**
3. **仅平台上下文有效记忆**
4. **群聊临时上下文**

否则会出现严重问题：

- 在群里的一句玩笑被写入长期人格判断
- 某个平台的低质量闲聊稀释高价值长期记忆
- 用户在不同平台期待的亲密度不同，却被同一套记忆策略混在一起

---

### 4.5 渠道抽象层：现在不必落地，但必须留设计口子

未来建议形成这样的数据流：

```text
Platform Ingress
  -> Channel Adapter
  -> UnifiedMessage
  -> Chat Core
  -> UnifiedResponse
  -> Channel Adapter
  -> Platform Egress
```

简单理解：

- `Ingress`：平台消息进来
- `Adapter`：做平台到统一协议的转换
- `Chat Core`：复用你现有的记忆、Agent、回访、关系逻辑
- `Egress`：把统一回复再转回平台格式发出去

如果不做这层抽象，以后每接一个平台都要改主聊天逻辑，代价会越来越高。

---

## 5. 产品路线会因此怎么调整

如果明确未来要接 Telegram / QQ，后续路线建议略微调整为：

### 阶段 1：继续先做长期个人 AI 的核心感知

优先级不变：

1. 待回访事项系统
2. 关系记忆系统
3. 首页“AI 想继续聊什么”

原因：

这些能力本身就是跨平台可复用的 AI 核心。

---

### 阶段 2：开始做多渠道准备层

新增架构工作：

4. 统一消息模型
5. 身份模型拆分（Person / Channel Account / Conversation）
6. 渠道抽象层接口
7. 记忆来源与隐私边界

这一步不一定立刻让用户感知变强，但会决定后面平台接入是不是顺利。

---

### 阶段 3：接入第一个外部平台

建议顺序：

8. Telegram 接入
9. 跨平台回访打通
10. 平台来源记忆策略上线

原因：

Telegram 更适合做第一批验证，能快速看到“长期个人 AI 的多入口形态”是否成立。

---

### 阶段 4：再评估 QQ 或其他国内平台

只有在 Telegram / Web / Desktop 三者之间已经把：

- 身份映射
- 消息归一化
- 回访逻辑
- 记忆边界

跑顺之后，再考虑 QQ 是否值得投入。

---

## 6. 为什么建议 Telegram 先于 QQ

### 6.1 Telegram 的优势

- Bot 生态成熟
- API 清晰稳定
- 私聊形态天然适合长期互动
- 主动消息机制友好
- 工程接入成本相对可控

它非常适合做“第一个外部聊天入口”的产品验证。

---

### 6.2 QQ 的现实成本

QQ 的问题通常不是“用户不需要”，而是“接入和长期维护更难”：

- 官方开放能力与限制
- 登录与风控问题
- 三方方案稳定性
- 合规与封禁风险
- 长期维护成本更高

因此建议：

> 先用 Telegram 验证“多渠道长期个人 AI”这件事，别一开始就把复杂度押在 QQ 上。

---

## 7. 对现有代码结构的含义

结合当前项目结构，可以预判后面会改到这些地方：

### 7.1 `lib/chat/`

需要从“前端页面驱动的聊天流程”逐步抽象成“统一消息协议驱动的聊天核心”。

### 7.2 `lib/memory/`

需要新增：

- 渠道来源字段
- 隐私边界字段
- 跨平台共享策略

### 7.3 `server/routes/`

后面可能需要新增 webhook / callback 类入口，而不只是当前桌面端 API。

### 7.4 新模块建议

未来大概率需要新增类似：

- `lib/channels/`
- `lib/identity/`
- `lib/proactive/`

用于承接：

- 平台适配
- 账户绑定
- 主动回访调度

---

## 8. 当前最重要的一条设计原则

以后每做一个功能，都应该问一句：

> 这是桌面端功能，还是 AI 核心能力？

真正值得沉淀的，应该尽量进入“核心能力层”。

例如：

- `followup_needed` 是核心能力
- 首页卡片只是桌面端表现层

- 关系记忆是核心能力
- Telegram 主动发消息只是其中一种出口

这个分层越早形成，后面接平台越稳。

2. **统一消息协议与渠道适配层设计**
   - UnifiedMessage / UnifiedResponse

## 10. 结论

如果未来确定要接 Telegram / QQ，Lumen 就不应该继续被只当作“桌面聊天产品”来设计，而应该被看作：

**一个有多个聊天入口的长期个人 AI 核心。**

当前最好的策略不是立刻去接很多平台，而是：

1. 先把长期个人 AI 的核心感知做出来
2. 再提前完成身份、消息、记忆、回访的多渠道抽象
3. 先用 Telegram 验证外部入口形态
4. 最后再评估 QQ 等更重的平台

一句话总结：

> 先把“长期个人 AI”的核心做对，再把它接到更多地方，而不是反过来。

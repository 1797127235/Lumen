# Design: 通用 delegate 子 Agent 工具

Generated by /brainstorming on 2026-05-29
Branch: master
Status: DRAFT — 待用户 review

---

## 1. 问题陈述

Lumen 目前是**纯被动**的:用户发一条消息 → 主 Agent 在一个受限的 loop 里(`request_limit=12`, `tool_calls_limit=20`,见 `lib/chat/agent_runner.py:167`)回一条。它**不能自主拆解并推进一个多步骤任务**。

典型缺口场景:用户说「帮我调研 X」。这件事需要十几次搜索 + 读网页 + 综合,会产生海量中间 token。在当前主 loop 里做有两个问题:

1. 主对话的 `request_limit=12` 根本不够深度调研
2. 即便调够,十几轮搜索结果会把日常陪伴对话的上下文撑爆、污染人格

**目标:给 Lumen 一个通用的「委派」能力** —— 主 Agent 能起一个独立的 child Agent 去专注完成一个子任务(调研是第一个用例),child 在隔离的上下文里跑、有自己的高预算、进度实时流回用户,跑完只把**结果摘要**交还主 Agent。

---

## 2. 范围(已与用户确认)

### 做

- 一个**通用** `delegate` 工具,主 Agent 可调用
- child = 复用模型配置起一个**全新会话**的 Agent 实例,独立 deps、独立高 `UsageLimits`
- **上下文隔离**:child 不看父对话历史(`message_history=[]`);父只拿到 child 的**最终输出摘要**,看不到 child 的中间工具调用与推理
- **聚焦 prompt**:用 `goal + context` 构一个临时 worker system prompt,**不注入 Lumen 人格 / 记忆 / 技能目录**(child 只干活)
- **进度流回**:child 跑 `agent.run_stream_events`,进度通过现有 EventBus 发回,前端实时显示一个独立的「子任务进度」区块
- **工具受限**:child 工具集 = 调用方指定的 `toolsets` bundle(默认 `["web","files"]`),再剥掉黑名单
- **调研 = delegate 的用法**,不是单独功能:主 Agent 识别到调研意图 → `delegate(goal="调研 X 并产出带来源报告", toolsets=["web","files"])` → child 搜+读+综合+写报告 → 返回摘要。输出:聊天里给带来源摘要 + 本地落一份完整 Markdown 报告

### 不做(明确砍掉,避免长成 hermes 那种庞然大物)

- ❌ **递归委派** —— child 不能再 `delegate`(深度固定 = 1)
- ❌ orchestrator 多级角色
- ❌ 跨 provider 路由 / credential pool / ACP transport
- ❌ **批量并行 fan-out**(`tasks[]` 数组)—— v1 只做单任务,真有并行调研需求再加
- ❌ todo 工具(独立的 Spec 2,做完后 delegate 进度展示会更好看,但 delegate 不依赖它)

### 安全护栏(必须有)

- child 工具黑名单:不能写 `memory`(不污染共享 `memory.md`)、不能再 `delegate`、不能发主动消息
- child 失败被隔离:抛错被 handler 捕获,返回错误摘要给父 Agent,不崩主对话
- child 独立 `UsageLimits`(建议 `request_limit=40`, `tool_calls_limit=60`),封顶防失控

## 3. 设计参考与取舍

参考 `E:\OpenHub\hermes-agent/tools/delegate_tool.py`(2767 行的子 Agent 架构)。**只取四个真正解决问题的设计点**,其余(ACP、credential pool、orchestrator 嵌套、跨 provider 路由)对单用户陪伴型 Lumen 是过度设计,不采用。

| 采用的设计点 | hermes 怎么做 | Lumen 落地 |
|---|---|---|
| 上下文隔离 | child 全新会话,父只见摘要 | `message_history=[]` + 工具返回值 = 最终输出 |
| 独立高预算 | child 自己的 `max_iterations` | child 自己的 `UsageLimits` |
| 进度回传 | `progress_callback` → 父 `tool_progress_callback` | 新 `SubagentProgress` 事件 → 现有 EventBus → 渠道 |
| 工具受限 + 黑名单 | 父工具集交集 - 黑名单,深度上限 | 指定 toolsets - 黑名单,深度固定 1 |

Lumen 已有「后台起 Agent」的原语(`lib/memory/review_service.py:56` 的 `get_agent()` + `agent.run()`、`lib/memory/understanding.py` 同模式),delegate 是把这个原语**升级成带进度流 + 隔离上下文 + 受限工具**的版本。

## 4. 架构

```
主 Agent loop (AgentRunner._process_message)
  │
  │ LLM 决定调用 delegate(goal=..., context=..., toolsets=[...])
  ↓
delegate handler (lib/tools/delegate.py)
  │ 1. 解析 child 工具集 = (toolsets 或默认) - 黑名单
  │ 2. 构 worker Agent(聚焦 prompt + 固定受限 toolset,无人格/记忆)
  │ 3. 开独立 db session,构 child LumenDeps(带流式上下文)
  │ 4. agent.run_stream_events(goal, message_history=[], UsageLimits 高)
  │      └─ 每个事件 → 调 ctx.progress_emitter(phase, detail) 报进度
  │ 5. 收集 child 最终输出
  ↓
返回最终输出摘要(字符串)给主 Agent  ← 父只看到这个
  │
  ↓
主 Agent 把摘要整理成自然回复发给用户(+ child 已落地的报告文件)
```

## 5. 关键组件与改动

### 5.1 新增:`lib/tools/delegate.py`

沿用现有工具模式(`create_xxx_tools()` 返回 `list[ToolDef]`,见 `lib/tools/web_tools.py`):

- `create_delegate_tools() -> list[ToolDef]`
- `async def handle_delegate(args: dict, ctx: LumenDeps) -> ToolReturn`

工具 schema:

```
delegate(
  goal: str,           # 必填:交给 child 的明确目标
  context: str = "",   # 可选:背景信息(主 Agent 从对话里提炼,因为 child 看不到历史)
  toolsets: list[str] = None,  # 可选:bundle 名,默认 ["web","files"]
)
```

**Lumen 没有 hermes 那种 toolset 概念,工具是按单名注册的**(`web_search`/`web_extract`/`web_crawl`、`file_read`/`file_write`/`file_ls`/`file_grep`/`file_edit`、写记忆的 `memory` 等)。为了让主 Agent 传简单的 `"web"/"files"` 而不暴露一串工具名,在 `delegate.py` 内联一个小 bundle 映射(**不引入全项目的 toolset 抽象**):

```
_BUNDLES = {
  "web":   ["web_search", "web_extract", "web_crawl"],
  "files": ["file_read", "file_write", "file_ls", "file_grep", "file_edit"],
}
_BLOCKLIST = {"memory", "delegate"}  # + 主动发消息的工具(若有)
```

解析:`toolsets` → 展开成工具名 → 剥掉 `_BLOCKLIST` → 即 child 的固定工具集。

`ToolMeta`:`risk="write"`(child 可能写文件)、`always_on=False`(deferred,通过 tool_search 解锁)或 `True`(常驻)—— **待 review 决定**。

### 5.2 新增:`core/agent.py` 的 worker agent 构建

现有 `get_agent()` 构的是带 Lumen 人格 + 按 conversation 动态发现工具 + 技能目录的**主对话** Agent,不适合 child。新增一个构建函数:

```
def build_worker_agent(tool_names: list[str], system_prompt: str) -> Agent[LumenDeps, str]
```

- 复用 `_create_model()`(同一套 provider/model 配置)
- toolset = `build_pydantic_toolset([registry.get_tool(n) for n in tool_names])`(固定,**不走 conversation 动态发现**)
- `system_prompt` = 传入的聚焦 worker prompt
- **不挂** `@agent.system_prompt` 技能目录、**不用** Lumen 人格 prompt、**不注入** 记忆快照

worker system prompt 模板(handler 内构造):

```
你是一个专注的任务执行 Agent。完成以下目标后,用简洁的中文总结结果。
若涉及调研,综合多个来源并标注出处。

【目标】{goal}
【背景】{context}
```

### 5.3 改动:`core/agent.py` 的 `LumenDeps` —— 加 `progress_emitter`(方案 B)

**设计原则:工具不应感知传输层(EventBus / 渠道 / session)。** 当前 `LumenDeps`(`core/agent.py:24`)只带业务依赖(`db`/`user_id`/`conversation_id`...),不带 EventBus / 渠道信息——这是**正确的分层边界**,不是缺陷。

delegate 是第一个需要"主动往外报进度"的工具,但它**不需要整个 EventBus + 渠道坐标**,只需要"我能报一句进度"的能力。所以不下沉传输细节,只加一个**已绑定好渠道坐标的回调**:

```
# LumenDeps 新增一个字段:
progress_emitter: Callable[[str, str], None] | None = None
#   签名 (phase, detail) -> None;为 None 时工具静默(后台 review/understanding agent 无人看,就是 None)
```

`AgentRunner._process_message` 在构 `deps` 处(`lib/chat/agent_runner.py:139`)用闭包绑定渠道坐标后注入:

```
def _emit_progress(phase: str, detail: str) -> None:
    self._event_bus.emit(SubagentProgress(
        channel=msg.channel, session_key=session_key, chat_id=msg.chat_id,
        phase=phase, detail=detail,
    ))
deps = LumenDeps(..., progress_emitter=_emit_progress)
```

这样:
- delegate handler 只调 `ctx.progress_emitter("step", "正在搜索 X")`,**完全不知道 EventBus / channel / session_key 的存在**,分层边界保住
- child agent 的 deps **不需要** `progress_emitter`(child 工具不报进度;由 delegate handler 观察 child 的 stream 后代为 emit)

### 5.4 新增:`lib/bus/event_bus.py` 的 `SubagentProgress` 事件

```
@dataclass
class SubagentProgress:
    channel: str
    session_key: str
    chat_id: str
    phase: str       # "started" | "step" | "done" | "error"
    detail: str      # 给用户看的进度文本,如「正在搜索:X 的市场现状」
```

handler 在 child 的 `run_stream_events` 循环里,把 child 的工具调用/思考映射成有意义的 `detail` 文本发出(不是「调用了第 3 个工具」,而是「正在搜索 X / 已读取 3 篇来源 / 正在综合」)。

### 5.5 改动:渠道订阅 + 前端渲染

- `channels/web/web.py`、`channels/telegram/telegram.py`:`self._event_bus.on(SubagentProgress, handler)`,转成 SSE / Telegram 消息
- Web SSE formatter:新增 `subagent_progress` 事件类型
- 前端 `src/pages/Chat.tsx`:把 `subagent_progress` 渲染成独立的「子任务进度」区块(类似现有 thinking / tool trace 的展示),与 Lumen 正常回复区分开

### 5.6 注册

`lib/tools/factory.py:41` 的 `all_tools` 列表加 `*create_delegate_tools()`。

## 6. 上下文隔离机制(细节)

- **父 → child 不泄漏历史**:`run_stream_events(goal, message_history=[])`,child 起点干净。父需要传给 child 的背景,由主 Agent 显式放进 `context` 参数(主 Agent 从对话里提炼)。
- **child → 父只回摘要**:delegate handler 的返回值 = child 的最终 `output` 文本。父 Agent 的消息历史里只有 `delegate` 的调用与这个返回值,看不到 child 的中间步骤。
- **独立 db session**:child 用 `get_async_session_maker()()` 开自己的 session,不复用父的(父 session 此刻在 tool 执行中,事务态敏感)。child 主要用 web+files,几乎不碰 db。

## 7. 调研用例(端到端)

1. 用户:「帮我调研一下国内开源向量数据库的现状」
2. 主 Agent 识别调研意图,调 `delegate(goal="调研国内开源向量数据库现状,产出带来源的对比报告,并把完整报告写到 ~/.lumen/reports/ 下的 markdown 文件", context="<从对话提炼的背景>", toolsets=["web","files"])`
3. child worker Agent 启动:web 搜索 → 读取若干来源 → 综合 → 用 `file_write` 落一份完整 markdown 报告。过程中每步发 `SubagentProgress`,前端实时显示
4. child 返回摘要(含报告文件路径)给主 Agent
5. 主 Agent 用自己的人格把摘要整理成自然回复发给用户,并告知报告已存到本地路径

输出形式 = **聊天摘要 + 本地完整报告文件**(已确认)。

## 8. 测试计划

- `tests/test_delegate_tool.py`:
  - bundle 解析:`["web","files"]` → 展开成 8 个工具名(`web_search`/`web_extract`/`web_crawl` + 5 个 `file_*`)
  - 黑名单生效:resolved 工具集里**永不含** `memory` / `delegate`(防污染记忆、防递归);未知 bundle 名被忽略不报错
  - 默认 `toolsets=None` → 等价于 `["web","files"]`
  - child 抛错时 handler 返回错误摘要、不向上抛(主对话不崩)
  - child `message_history` 为空(隔离)
  - `progress_emitter=None` 时 handler 不报错(后台无人看场景)
- 用 mock model 跑一个最小 delegate:goal → child 返回固定文本 → 验证父拿到摘要、`SubagentProgress` 被 emit
- 验证门:真实配置下「帮我调研 X」能跑通,前端能看到进度区块,本地生成报告文件

## 9. 决策(已拍板 2026-05-29)

1. **`delegate` 工具可见性 = deferred**(`always_on=False`,靠 `tool_search` 解锁)。减少日常对话的 schema 体积,主 Agent 真要委派时搜出来。
2. **child `UsageLimits` = `request_limit=40` / `tool_calls_limit=60`**。
3. **报告文件目录 = `~/.lumen/reports/`**(与 Lumen 用户数据目录 `~/.lumen/` 一致;handler 负责确保目录存在)。
4. **`SubagentProgress.detail` 映射 v1 先粗**:工具名 + 简短参数即可(如「web_search: 国内开源向量数据库」),不做精细的「事件 → 人话」规则。

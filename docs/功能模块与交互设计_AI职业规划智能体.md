## 一、功能模块总览

### 1.1 一级模块划分

基于"计算机专业学生的AI职业规划伙伴"产品定位，结合4个阶段10个核心场景，系统将功能划分为 **5个一级模块**：

| 一级模块 | 模块定位 | 核心目标 | 覆盖用户旅程阶段 |
|---------|---------|---------|----------------|
| **职业认知与诊断** | 用户起点模块，帮助建立方向认知 | 解决"我是谁、适合什么方向" | 入学适应期 |
| **学习路径与成长** | 能力建设核心模块 | 解决"学什么、怎么学、学到什么程度" | 技能积累期 |
| **求职作战室** | 求职冲刺核心模块 | 解决"简历、面试、进度管理" | 实习求职期 |
| **AI对话伙伴** | 贯穿全旅程的交互入口 | 7x24小时答疑、陪伴、情感支持 | 全阶段 |
| **个人数据中心** | 用户资产沉淀模块 | 记录画像、进度、历史、成就 | 全阶段 |

## 二、核心功能详细设计

### 核心功能①：智能职业咨询对话

#### 功能定位与价值主张

**定位**: 系统的核心交互形态和流量入口，用户通过自然语言与AI智能体进行多轮对话，获取计算机行业方向、职业发展、学习建议等结构化、个性化回答。

**价值主张**: "像和一位懂技术的学长聊天" — 用大白话解释复杂概念，用故事和案例替代枯燥定义，主动追问以澄清需求，在对话中逐步帮用户理清思路。

#### 用户触发条件

| 触发场景 | 用户输入示例 |
|---------|------------|
| 方向迷茫 | "后端和前端有什么区别？我该选哪个？" |
| 信息求证 | "算法岗真的只招硕士吗？本科生有机会吗？" |
| 发展路径 | "做Java后端的话，未来3-5年怎么发展？" |
| 学习困惑 | "听说Go语言很火，我该从Java转Go吗？" |
| 行业趋势 | "大模型时代，普通开发还有前途吗？" |
| 情绪倾诉 | "投了20份简历都没回应，我是不是没戏了？" |

#### 输入与输出定义

| 维度 | 定义 |
|------|------|
| **用户输入** | 自然语言文本（支持追问、澄清、切换话题），可选附带用户画像上下文 |
| **系统输出** | 结构化回答（Markdown格式），包含：①核心观点摘要 ②详细解释（含类比/案例） ③个性化建议（基于用户画像） ④下一步行动推荐 ⑤相关追问引导 |
| **输出格式** | 对话气泡形式，长内容自动折叠"展开更多"，关键数据用信息卡片展示 |

#### 详细交互流程

**步骤1 - 用户发起对话**
- 用户操作：在首页输入框输入问题，或点击推荐问题卡片
- 系统响应：输入框下方显示"学长正在思考..."（带打字机动画），2秒内返回首句回应

**步骤2 - 意图识别与上下文加载**
- 系统操作：
  1. NLP解析用户意图（方向咨询/学习建议/情绪倾诉/事实查询）
  2. 加载用户画像（年级、目标方向、已知技能、历史对话偏好）
  3. 查询知识库（计算机行业知识图谱、最新趋势数据）
- 边界处理：意图模糊时，AI主动澄清："你是想了解技术层面的区别，还是就业前景的对比？"

**步骤3 - 知识检索与回答生成**
- 系统操作：
  1. RAG检索：从行业知识库召回相关知识点（方向定义、技能要求、薪资数据、发展趋势）
  2. 个性化融合：将检索内容与用户画像融合（如"基于你是大一学生，建议从...开始"）
  3. LLM生成：生成结构化回答，控制语气为"学长口吻"（亲切、有干货、不装腔）
- 输出格式示例：
  ```
  【一句话总结】后端开发主要负责服务器端的业务逻辑和数据处理。
  
  【详细解释】打个比方，一个电商App...（类比说明前后端分工）
  
  【给你的建议】考虑到你目前大一、C语言刚入门...（个性化）
  
  【下一步】想了解后端需要学哪些技术？或者看看后端 vs 前端的对比？
  ```

**步骤4 - 追问引导与对话延续**
- 系统操作：在回答末尾提供2-3个追问按钮（如"后端技术栈有哪些""后端薪资怎么样""我不爱写代码适合做后端吗"）
- 用户操作：点击追问按钮继续对话，或输入新问题

**步骤5 - 多轮对话状态管理**
- 系统操作：维护对话上下文树，记录当前话题分支（如"后端方向"子话题下的"技术栈->Spring Boot->微服务"深度）
- 状态切换：用户主动切换话题时（"换个话题，我想了解算法岗"），AI确认切换并保存原话题快照以便回溯

**步骤6 - 对话沉淀与行动转化**
- 系统操作：当对话涉及可执行内容时（如"大学四年规划"），提供"生成规划卡片"按钮
- 用户操作：点击后跳转至【四年规划生成】功能，预填对话中提取的参数
- 边界处理：用户未提供足够信息时，AI引导补充："为了给你更准确的建议，能告诉我你现在的编程基础吗？"

#### 边界情况与异常处理

| 边界情况 | 处理策略 |
|---------|---------|
| 用户问题超出计算机领域 | "这个问题超出了计算机职业规划的范围哦，不过我可以帮你分析XXX与计算机交叉的部分..." |
| 用户输入敏感/违规内容 | 温和拒绝并引导回正轨："咱们还是聊聊职业发展吧，你对哪个方向感兴趣？" |
| AI生成内容不确定 | 明确标注"这部分信息我了解有限，建议你再到XX（牛客/知乎）交叉验证一下" |
| 用户连续追问超过10轮 | 提示"对话有点长了，要不要我把刚才聊的内容整理成一份笔记？" |
| 网络异常/模型超时 | 显示"学长卡壳了，稍等2秒再试试"，自动重试3次后提示用户刷新 |

#### AI Agent角色与工作流

**Agent角色**: "职业学长Agent"
- 人设：研二学长，大厂实习经历，愿意分享，说话直接但不伤人，懂技术也懂职场
- 核心能力：行业知识问答 + 个性化建议 + 情绪识别与安抚

**工作流**: 
```
感知（用户输入）-> 意图分类（咨询类/情绪类/事实类）-> 
上下文加载（用户画像+对话历史）-> RAG知识检索 -> 
个性化内容生成（LLM+Prompt工程）-> 输出结构化回答 -> 
追问推荐生成 -> 对话状态更新
```

### 核心功能②：个性化学习路径生成

#### 功能定位与价值主张

**定位**: 将"碎片化收藏"转化为"可执行路线"，基于用户目标岗位、当前基础、可用时间，生成定制化的渐进式学习路径。

**价值主张**: "不是给你一张地图，而是给你一张根据你的体能和目标定制的训练计划" — 标注每个阶段学什么、学到什么程度、用多久、如何验收。

#### 用户触发条件

| 触发场景 | 用户操作 |
|---------|---------|
| 刚确定方向 | 在咨询对话中或画像设置里选择了"Java后端"方向 |
| 需要系统化 | 收藏了50篇博客但从未执行，主动寻求路径规划 |
| 目标明确 | 输入目标公司+岗位（如"字节跳动后端开发"） |
| 路径调整 | 已生成路径但想切换方向或调整难度 |

#### 输入与输出定义

| 维度 | 定义 |
|------|------|
| **用户输入** | 目标岗位（如"后端开发"）、当前基础（多选：零基础/学过C/做过项目）、可用时间（每天1h/2h/4h）、院校层级（985/211/双非）、目标公司层级（大厂/中厂/国企）、截止时间（如"大三暑假实习前"） |
| **系统输出** | 结构化学习路径，包含：①总览时间线（甘特图形式） ②阶段列表（每个阶段含节点名称、预计时长、验收标准、推荐资源） ③里程碑标记点 ④风险提醒（如"此阶段耗时较长，注意保持节奏"） |
| **输出格式** | 可视化时间线 + 可展开的卡片列表 + 一键导出日历/待办 |

#### 详细交互流程

**步骤1 - 信息收集（对话式表单）**
- 系统操作：以对话方式逐条询问关键参数，避免表单疲劳
  ```
  AI: "想给你定制一条学习路线，先聊几句~ 你目标的方向是什么？"
  选项卡：[Java后端] [Go后端] [前端] [算法] [大数据] [其他]
  
  用户: [点击 Java后端]
  
  AI: "好嘞！Java后端是个经典方向。你现在基础怎么样？"
  选项卡：[零基础，只学过C语法] [学过Java基础语法] [做过小项目] [较熟练，想冲大厂]
  ```
- 边界处理：用户不确定目标时，提供"帮我推荐"按钮，跳转方向测评

**步骤2 - 参数确认与路径计算**
- 系统操作：
  1. 将用户输入标准化为路径生成参数
  2. 调用路径规划引擎（基于岗位技能图谱 + 渐进式依赖图）
  3. 计算总时长、阶段拆分、资源匹配
- 系统响应：显示"正在为你规划路线..."进度动画（预计3-5秒）

**步骤3 - 路径可视化展示**
- 系统输出：顶部时间线（横向滚动）+ 下方详细卡片
  ```
  ━━━ 第1阶段：Java基础夯实（4周）━━━ 第2阶段：数据库核心（3周）━━━ ...
  
  [阶段卡片]
  阶段1：Java SE基础 -> 面向对象 -> 集合框架 -> 异常处理 -> IO流
  预计时长：4周（每天2小时）
  验收标准：能独立手写一个带连接池的JDBC工具类，理解多线程基础
  推荐资源：
    - 书籍：《Java核心技术 卷I》第3-8章
    - 视频：韩顺平Java（B站免费）
    - 练习：LeetCode简单题10道
  里程碑测验：[开始测验]（5道选择题+1道编程题）
  ```

**步骤4 - 用户确认与调整**
- 用户操作：可拖拽调整阶段顺序（系统校验依赖关系，如不能先学Spring再学Java基础），或点击"太紧了/太松了"调整每日时长
- 系统响应：实时重新计算并展示调整后的时间线

**步骤5 - 路径保存与日程同步**
- 用户操作：点击"保存到我的计划"，选择是否同步到系统日历/手机日历
- 系统操作：
  1. 保存路径到用户学习档案
  2. 生成每周学习任务提醒（如"本周目标：完成MySQL索引章节"）
  3. 在【个人数据中心-学习档案】创建条目

**步骤6 - 进度追踪入口**
- 系统操作：在学习路径页面常驻"标记进度"按钮，用户完成节点后打勾
- 联动效果：已完成的节点变绿，进行中的节点闪烁提示，未开始的节点置灰

#### 边界情况与异常处理

| 边界情况 | 处理策略 |
|---------|---------|
| 用户目标岗位不在库中 | "这个方向目前数据较少，我先按'后端通用'给你规划，其中XX部分你可以替换成目标技术" |
| 截止时间过紧（如2个月要掌握6个月内容） | 红色警告"时间非常紧张，建议：①延长截止时间 ②降低目标公司层级 ③聚焦核心技能放弃加分项" |
| 用户基础与目标跨度太大 | 弹出建议"你的基础和目标差距较大，建议分两步走：第一阶段打基础（6个月），第二阶段冲刺（3个月）" |
| 依赖冲突（用户调整顺序违反依赖） | 阻止调整并提示"Spring Boot需要先掌握Java基础，建议先完成阶段1" |

#### AI Agent角色与工作流

**Agent角色**: "学习规划Agent"
- 人设：刚拿到大厂offer的学长，擅长拆解目标、制定计划
- 核心能力：技能图谱检索 + 渐进式依赖计算 + 资源匹配

**工作流**:
```
参数收集 -> 目标岗位技能图谱查询 -> 当前基础匹配 -> 
缺失技能排序（按面试频率和依赖关系） -> 
渐进式路径编排（确保前置技能先学） -> 
学习资源匹配（书籍/视频/文档/练习题） -> 
时间分配计算 -> 生成可视化路径
```

### 核心功能③：岗位能力差距分析

#### 功能定位与价值主张

**定位**: 将"模糊的焦虑"转化为"清晰的行动清单"，客观评估用户当前技能水平与目标岗位要求的具体差距。

**价值主张**: "不是告诉你'你还差得远'，而是精确到'Redis持久化机制这道面试题你还答不上来，需要补这里'" — 用数据将焦虑转化为可执行的行动。

#### 用户触发条件

| 触发场景 | 用户操作 |
|---------|---------|
| 看到JD后自我怀疑 | 粘贴一份JD，让系统分析自己与岗位的匹配度 |
| 定期体检 | 每月主动进行一次能力盘点 |
| 路径生成后 | 学习路径中某阶段完成后进行自评 |
| 面试挂掉后 | 输入挂掉的面试环节，系统分析短板 |

#### 输入与输出定义

| 维度 | 定义 |
|------|------|
| **用户输入** | 目标岗位JD（文本粘贴）或选择预设岗位模板；用户自评（1-5分）或选择"通过测试评估" |
| **系统输出** | ① 岗位能力模型拆解（维度列表） ② 能力雷达图（当前vs目标） ③ 差距等级标注（达标/接近/差距大/缺失） ④ 补全建议（优先级排序） ⑤ 预计补全时长 |
| **输出格式** | 雷达图可视化 + 可排序的维度卡片 + 一键生成补全学习路径 |

#### 详细交互流程

**步骤1 - 目标岗位设定**
- 用户操作：
  - 方式A：粘贴一份JD文本
  - 方式B：选择预设模板（如"字节跳动-后端开发-2-1""阿里巴巴-Java开发-P5"）
  - 方式C：从已生成的学习路径直接关联目标岗位
- 系统操作：解析JD，提取关键技能要求（如"熟悉Redis"->拆解为"数据结构/持久化/集群/性能优化"）

**步骤2 - 评估方式选择**
- 系统操作：提供两种评估方式
  ```
  "想怎么评估？"
  [快速自评] 我对每个技能点自己打分（2分钟）
  [深度测评] 做题+问答，系统帮我评估（15-20分钟）
  ```
- 边界处理：首次使用建议深度测评，后续可用快速自评

**步骤3 - 深度测评流程（选择深度测评时）**
- 系统操作：按能力维度轮流出题
  ```
  【技术维度：Redis】
  Q1: Redis支持哪些数据结构？（多选）
  Q2: 场景题：如果Redis突然变慢，你的排查思路是什么？（简答，AI评分）
  Q3: 编程题：用Redis实现一个分布式锁。（在线编码）
  ```
- 每题完成后：即时反馈正确/部分正确/错误，附解析
- 全部完成后：显示"分析中..."，计算各维度得分

**步骤4 - 生成差距分析报告**
- 系统输出：
  ```
  【岗位：字节跳动-后端开发】能力差距分析报告
  
  [雷达图] 你的得分 vs 岗位要求
  
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  【达标】Java基础（85/100）✅
    -> 保持即可，面试前复习集合框架源码
  
  【接近】MySQL（62/100）⚠️ 需提升
    -> 差距点：索引优化、事务隔离级别、慢查询分析
    -> 建议学习：《MySQL实战45讲》第15-22讲
    -> 预计补全：2周
  
  【差距大】Redis（35/100）🔴 重点突破
    -> 差距点：持久化机制(RDB/AOF)、集群模式、缓存击穿/穿透/雪崩
    -> 建议学习：路径已生成 [查看补全路径]
    -> 预计补全：3周
  
  【缺失】分布式系统（0/100）❌ 从零开始
    -> 差距点：CAP理论、一致性协议、微服务架构
    -> 建议学习：路径已生成 [查看补全路径]
    -> 预计补全：6周
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  
  综合评估：当前匹配度 47%，预计11周可达80%（达到面试门槛）
  [生成补全学习路径] [保存报告] [分享报告]
  ```

**步骤5 - 补全路径生成**
- 用户操作：点击"生成补全学习路径"
- 系统操作：将差距维度按优先级排序（面试频率高->前置依赖->难度递增），生成针对性补全路径
- 联动：自动与现有学习路径合并，标注新增节点

**步骤6 - 定期复盘提醒**
- 系统操作：保存报告时询问"是否每月自动复测？"
- 用户确认后：每月推送复测提醒，对比历史雷达图展示进步曲线

#### 边界情况与异常处理

| 边界情况 | 处理策略 |
|---------|---------|
| JD解析失败（格式混乱） | "这份JD格式比较特别，我提取到以下关键要求：[列出]，如果遗漏了你手动补充一下？" |
| 用户自评明显偏高/偏低 | 与历史测评数据对比，偏差大时提示"你自评的Java基础和上次测试差距较大，建议用测评验证一下" |
| 目标岗位要求超出学生合理范围 | "这个岗位是社招P6级别，对在校生要求过高，建议换成'字节后端实习'或'校招P5'作为目标" |
| 测评中途退出 | 自动保存进度，下次进入提示"上次测评做到Redis部分，继续吗？" |

#### AI Agent角色与工作流

**Agent角色**: "能力诊断Agent"
- 人设：技术面试官视角，严格但公正，善于发现盲点
- 核心能力：JD解析 + 能力模型拆解 + 多维度测评 + 差距量化

**工作流**:
```
JD输入 -> JD解析引擎（提取技能关键词） -> 
岗位能力模型查询（技能->知识点映射） -> 
评估方式选择 -> 出题/自评收集 -> 
多维度评分计算 -> 差距等级判定 -> 
补全优先级排序（面试频率 x 提升空间 x 依赖关系） -> 
生成差距报告 + 补全路径
```

### 核心功能④：简历智能解析与优化

#### 功能定位与价值主张

**定位**: 将"一份简历投所有岗位"升级为"千人千面的精准投递"，通过JD匹配度分析、STAR法则优化、关键词补充，显著提升简历过筛率。

**价值主张**: "不只是改病句，而是帮你想清楚——这段经历想证明什么能力？面试官看到这句话会问什么？" — 从面试官视角重写简历。

#### 用户触发条件

| 触发场景 | 用户操作 |
|---------|---------|
| 准备投递 | 上传现有简历 + 粘贴目标岗位JD |
| 面试挂掉怀疑简历 | 上传简历询问"简历有什么问题" |
| 方向切换 | 已有简历但想换方向投递 |
| 定期迭代 | 每两周优化一次简历版本 |

#### 输入与输出定义

| 维度 | 定义 |
|------|------|
| **用户输入** | 简历文件（PDF/Word/图片）+ 目标岗位JD（文本） |
| **系统输出** | ① 结构化简历信息提取 ② JD匹配度评分 ③ 关键词缺失分析 ④ 逐条优化建议 ⑤ 优化后简历预览 ⑥ ATS友好性检测报告 |
| **输出格式** | 左右分栏：左侧原始简历预览，右侧分析报告 + 优化建议 |

#### 详细交互流程

**步骤1 - 简历上传**
- 用户操作：拖拽或点击上传简历文件，支持PDF/Word/PNG/JPG
- 系统响应：显示上传进度，完成后"正在解析简历..."（OCR+NLP提取）
- 解析结果展示：
  ```
  【解析结果确认】请检查识别是否准确：
  姓名：张三 ✓
  学校：XX大学 ✓
  专业：软件工程 ✓
  电话：138****8888 ✓
  经历1：XX公司实习 - 后端开发（2024.06-2024.09）✓
  项目1：电商系统（Spring Boot+MySQL）
  ...
  [有错误？手动修正]
  ```

**步骤2 - JD粘贴/选择**
- 用户操作：粘贴目标岗位JD文本，或从收藏夹选择已保存的JD
- 系统操作：解析JD提取核心要求（技能关键词、经验要求、软技能）

**步骤3 - 匹配度分析**
- 系统输出：
  ```
  【匹配度评分】67/100 — 有提升空间
  
  [匹配热力图]
  Java基础 ████████████ 90% ✅
  Spring框架 ████████░░ 65% ⚠️
  数据库优化 ████░░░░░░ 40% 🔴 缺失
  分布式经验 █░░░░░░░░░ 10% 🔴 严重缺失
  团队协作 █████████░░ 80% ✅
  
  【关键词缺失】以下JD关键词未在你的简历中体现：
  • "微服务架构" -> 建议补充位置：项目描述
  • "Redis缓存" -> 建议补充位置：技能列表/项目描述
  • "高并发" -> 建议补充位置：项目描述（量化指标）
  ```

**步骤4 - 逐条优化建议**
- 系统操作：遍历简历各模块，给出具体修改建议
  ```
  【教育背景】✅ 基本完整，建议补充：
    -> " GPA 3.6/4.0（前15%）" 如成绩好可加上
  
  【实习经历】⚠️ 需要优化：
    原文："负责后端模块开发，参与需求评审和代码编写"
    问题：过于笼统，无技术深度，无量化成果
    建议改为：
    "负责订单系统核心模块开发，使用Spring Boot + MyBatis实现订单状态机流转，
     设计并接入Redis缓存降低数据库查询量40%，参与Code Review提升团队代码规范覆盖率至90%"
    [一键应用此建议] [查看为什么这么改]
  
  【项目经历】🔴 严重需要优化：
    原文："做了一个电商系统，实现了用户登录、商品展示、购物车功能"
    问题：① "做了一个"显得随意 ② 纯功能罗列 ③ 无技术难点和成果
    建议改为：
    "独立设计并实现高并发电商秒杀系统，通过Redis预减库存+消息队列异步下单+数据库乐观锁，
     支撑1000 QPS并发场景下库存超卖率降至0，项目获课程设计优秀（前5%）"
    [一键应用此建议] [查看为什么这么改]
  ```

**步骤5 - 一键优化与预览**
- 用户操作：可选择"一键应用全部建议"或逐条确认
- 系统输出：优化后的简历预览，标注所有修改点（删除线+高亮）
- 提供多模板切换：大厂风格（简洁、关键词突出）/ 外企风格（STAR详细）/ 国企风格（稳重、全面）

**步骤6 - ATS检测与导出**
- 系统操作：模拟ATS系统扫描，检测：
  - 关键词密度是否合理（避免堆砌）
  - 格式是否会被解析错误（表格/图片/复杂排版）
  - 文件大小是否合适
- 用户操作：下载优化后的PDF/Word，或保存到【个人数据中心-求职档案】
- 版本管理：自动命名"张三_后端开发_字节跳动_20250601_v2"

#### 边界情况与异常处理

| 边界情况 | 处理策略 |
|---------|---------|
| 简历解析失败（扫描件质量差） | "简历图片清晰度不够，部分文字识别困难。建议重新上传或手动输入关键信息。" |

### 3.4 AI对话交互设计原则

#### 原则1："学长口吻" — 亲切专业不冰冷

| 场景 | 错误示例（冰冷） | 正确示例（学长口吻） |
|------|---------------|-------------------|
| 方向介绍 | "后端开发是负责服务器端业务逻辑处理的软件工程师岗位" | "后端就是处理你点外卖时'订单怎么创建、优惠券怎么算、支付怎么对接'这些事儿的。用户看不到后端，但没有后端，App就是空壳子。" |
| 学习建议 | "建议先学习Java SE基础，预计耗时4周" | "先别急着搞微服务，Java基础不牢地动山摇。老老实实4周把Java SE啃完，我当年就是跳过集合源码直接上Spring，面试被问到HashMap源码直接懵了。" |
| 焦虑安抚 | "请保持积极心态，继续投递简历" | "我秋招投了80多家才拿5个Offer，你这才20家急啥。不过咱们看看数据，是不是简历过筛率有点低？我给你分析分析。" |

#### 原则2：主动追问 > 被动回答

- 不直接回答模糊问题，而是先澄清："你是想了解前端技术栈有哪些，还是想知道前端未来的发展前景？"
- 回答后主动延伸："说到这儿，你可能还想知道..."
- 关键信息遗漏时主动确认："为了给你更准确的建议，你现在是大几？学过哪些编程语言？"

#### 原则3：结构化输出降低认知负荷

- 长回答必须分段：【一句话总结】【详细解释】【给你的建议】【下一步】
- 复杂内容用信息卡片承载（如路径卡片、雷达图卡片、对比表格卡片）
- 关键数据高亮（薪资范围、时间节点、成功率）
- 支持"展开更多"折叠长篇内容

#### 原则4：即时反馈与操作入口

- 回答中涉及的功能提供"一键跳转"按钮（如"生成学习路径""开始模拟面试"）
- 对话中可执行的结论提供"保存""导出""分享"操作
- 用户操作后给予即时确认（"已保存到你的学习档案"）

#### 原则5：渐进式披露降低门槛

- 首次对话不超过3轮即给出可见价值（如方向推荐、初步建议）
- 复杂功能分步引导（如测评分模块完成，不用一次做完）
- 高级功能默认隐藏，用户需要时展开（如"查看完整技术图谱"）

## 四、数据模型设计

### 4.1 核心实体定义

系统核心实体共 **10个**：

| 实体 | 英文名称 | 定义 | 核心作用 |
|------|---------|------|---------|
| 用户 | `User` | 系统注册用户 | 身份与权限基础 |
| 用户画像 | `UserProfile` | 用户职业相关信息 | 个性化推荐的数据基础 |
| 对话会话 | `Conversation` | 一次AI对话的完整记录 | 多轮对话状态管理 |
| 对话消息 | `Message` | 对话中的单条消息 | 内容存储与上下文构建 |
| 学习路径 | `LearningPath` | 用户的学习路线规划 | 学习进度追踪基础 |
| 路径节点 | `PathNode` | 学习路径中的单个学习单元 | 任务拆解与进度标记 |
| 简历 | `Resume` | 用户的简历数据 | 解析与优化的数据基础 |
| 求职记录 | `JobApplication` | 单次求职投递记录 | 求职进度追踪 |
| 面试记录 | `InterviewRecord` | 单次面试的详细记录 | 反馈分析与复盘 |
| 能力评估 | `SkillAssessment` | 用户的技能评估结果 | 差距分析基础 |

### 4.2 实体关系图（ER图文字描述）

```
+----------+       +--------------+       +----------------+
|   User   |<----->|  UserProfile |<----->|  SkillAssessment|
|  (用户)   | 1:1   |  (用户画像)   | 1:N   |   (能力评估)    |
+----------+       +--------------+       +----------------+
       |                 |
       | 1:N             | 1:N
       v                 v
+--------------+    +--------------+
| Conversation |    | LearningPath |
|  (对话会话)   |    |   (学习路径)  |
+--------------+    +--------------+
       | 1:N              | 1:N
       v                  v
+--------------+    +--------------+
|   Message    |    |   PathNode   |
|  (对话消息)   |    |   (路径节点)  |
+--------------+    +--------------+
                            |
       +--------------------+--------------------+
       | 1:N                | 1:N                | 1:N
       v                    v                    v
+--------------+    +--------------+    +--------------+
|    Resume    |    |JobApplication|    |   Resource   |
|    (简历)     |    |   (求职记录)  |    |   (资源)     |
+--------------+    +--------------+    +--------------+
                           | 1:N
                           v
                    +--------------+
                    |InterviewRecord|
                    |  (面试记录)   |
                    +--------------+
```

**关系说明**:
- `User` 1:1 `UserProfile`：每个用户有且仅有一个画像（注册时创建，持续更新）
- `User` 1:N `Conversation`：用户可有多段对话历史
- `Conversation` 1:N `Message`：每段对话包含多条消息
- `UserProfile` 1:N `LearningPath`：用户可有多条学习路径（不同方向/版本）
- `LearningPath` 1:N `PathNode`：每条路径包含多个学习节点
- `UserProfile` 1:N `SkillAssessment`：用户可进行多次能力评估
- `User` 1:N `Resume`：用户可有多版本简历
- `User` 1:N `JobApplication`：用户可投递多个岗位
- `JobApplication` 1:N `InterviewRecord`：每次求职可有多轮面试
- `PathNode` N:M `Resource`：学习节点关联多个学习资源

### 4.4 数据流转与状态机

#### 状态机1：学习路径状态流转

```
                    +-------------+
         +-------->|   draft     |<--------+
         |          |  （草稿）    |         |
         |          +------+------+         |
         |                 | 用户激活        |
         |                 v                |
         |          +-------------+         |
         |     +----|   active    |----+    |
         |     |    |  （进行中）  |    |    |
         |     |    +------+------+    |    |
         |     |           |           |    |
    放弃/过期 |     |    完成节点    暂停学习    |    | 重新激活
         |     |           |           |    |
         |     v           v           v    |
         |  +------+   +--------+   +-----+  |
         +--|abandoned|   |completed|   |paused|--+|
            | （已放弃）|   |（已完成）|   |（暂停）|
            +--------+   +---------+   +------+
```

**状态说明与流转条件**:
| 状态 | 说明 | 流转条件 |
|------|------|---------|
| draft | 用户创建但未确认激活 | 用户点击"开始执行" -> active |
| active | 用户正在执行中 | 完成所有节点 -> completed；用户点击暂停 -> paused；用户点击放弃 -> abandoned |
| paused | 用户暂停执行 | 用户点击继续 -> active |
| completed | 所有节点完成 | 不可流转（终态） |
| abandoned | 用户主动放弃或过期 | 不可流转（终态） |

#### 状态机2：求职进度状态流转

```
+----------+    +----------+    +------------+    +----------+
| applied  |--->| screening|--->|written_test|--->|1st_intervw|
| （已投递） |    |（简历筛选）|    |  （笔试）   |    |（一面）    |
+----------+    +----------+    +------------+    +-----+-----+
     |                                                   |
     |    +-----------------------------------------------+
     |    |
     |    v    +----------+    +----------+    +----------+
     |    +--->|2nd_intervw|--->|3rd_intervw|--->|hr_intervw|
     |         |（二面）    |    |（三面）    |    |（HR面）   |
     |         +----------+    +----------+    +-----+-----+
     |                                               |
     |                                               v
     |                                          +----------+
     |                                          |    oc     |
     |                                          | （Offer）  |
     |                                          +-----+-----+
     |                                                |
     |         +--------------------------------------+
     |         |
     v         v
+------------------------+
|      rejected            |
|      （已拒绝/已挂掉）    |
|  任一环节均可流转至此      |
+------------------------+
```

**关键流转规则**:
- 任意正向状态均可流转至 `rejected`（公司拒绝）或 `withdrawn`（用户主动撤回）
- `oc` 后可流转至 `accepted`（用户接受）或继续比较（保持在oc状态）
- `screening` 可能直接跳过 `written_test`（部分公司无笔试）
- `stage_history` 字段记录完整流转轨迹

#### 状态机3：路径节点状态流转

```
+---------+    +---------+    +---------+    +---------+
| locked  |--->|unlocked |--->|in_progress|--->|completed|
|（未解锁） |    |（已解锁） |    |（学习中）  |    |（已完成） |
+---------+    +---------+    +---------+    +---------+
                    |                              |
                    +------------------------------+
                           （前置节点完成后自动解锁）
```

**解锁规则**:
- 节点的 `prerequisite_nodes` 中所有节点状态为 `completed` 时，自动解锁
- 用户可手动标记 `skipped`（跳过）但系统会警告"跳过此节点可能影响后续学习"

## 五、AI Agent工作流设计

### 5.1 Agent核心工作流程

系统采用 **"感知->理解->规划->行动->反馈"** 五阶段Agent工作流，所有AI交互均遵循此范式：

```
+---------------------------------------------------------------+
|  1. 感知（Perception）                                         |
|     |-- 用户输入捕获（文本/语音/图片）                            |
|     |-- 上下文加载（用户画像 + 对话历史 + 系统状态）               |
|     |-- 环境感知（当前时间、用户活跃状态、外部事件）               |
+---------------------------------------------------------------+
|  2. 理解（Understanding）                                      |
|     |-- 意图识别（分类用户请求类型）                              |
|     |-- 情感分析（焦虑/积极/困惑/愤怒）                          |
|     |-- 实体抽取（方向名、技术栈、公司名、时间节点）               |
|     |-- 缺失信息检测（判断是否需要追问补充）                       |
+---------------------------------------------------------------+
|  3. 规划（Planning）                                            |
|     |-- 任务拆解（将复杂请求拆分为子任务）                        |
|     |-- 工具选择（判断是否需要调用外部工具）                       |
|     |-- 策略选择（直接回答 / 引导追问 / 推荐功能 / 情感安抚）       |
|     |-- 知识检索（RAG检索相关领域知识）                          |
+---------------------------------------------------------------+
|  4. 行动（Action）                                              |
|     |-- LLM生成（基于Prompt工程生成结构化回答）                   |
|     |-- 工具调用（如需要，调用简历解析/路径生成/题库查询等工具）     |
|     |-- 卡片渲染（将结构化内容转化为前端可渲染卡片）                |
|     |-- 记忆更新（更新对话上下文和用户短期记忆）                  |
+---------------------------------------------------------------+
|  5. 反馈（Feedback）                                            |
|     |-- 输出格式化（Markdown + 卡片 + 追问按钮）                  |
|     |-- 用户操作入口（保存/导出/跳转/一键执行）                   |
|     |-- 追问推荐（基于当前话题生成2-3个追问选项）                   |
|     |-- 状态同步（将关键信息写入用户画像/学习档案等持久存储）        |
+---------------------------------------------------------------+
```

### 5.2 多轮对话的状态管理

#### 对话状态树

系统将每段对话维护为一棵 **状态树**，支持话题分支、回溯、切换：

```
Conversation: "Java后端方向咨询"
|
|-- [Root] 用户："后端开发是做什么的？"
|   |-- [Branch A] 系统：解释后端职责
|   |   |-- [Node A1] 用户："需要学什么技术？" -> 进入技术栈话题
|   |   |   |-- [Node A1-1] 系统：列出技术栈 -> 用户："Spring Boot难吗？"
|   |   |   |-- [Node A1-2] 系统：解释Spring Boot -> ...
|   |   |-- [Node A2] 用户："薪资怎么样？" -> 进入薪资话题
|   |-- [Branch B] 用户："和前端比哪个好？"
|       |-- [Node B1] 系统：后端vs前端对比 -> ...
|
|-- 当前活跃话题栈：[Root -> Branch A -> Node A1 -> Node A1-1]
  （用户说"回到刚才那个问题" -> 可回溯到任意历史节点）
```

#### 状态管理策略

| 状态类型 | 存储位置 | 说明 |
|---------|---------|------|
| 对话级状态 | Redis（TTL=24h） | 当前对话的活跃话题栈、已确认的用户信息、待办追问 |
| 会话级状态 | Conversation.context_snapshot | 会话的摘要快照，用于对话列表预览和快速恢复 |
| 用户级状态 | UserProfile + 专用缓存 | 跨对话的持久记忆（目标方向、已知技能、偏好设置） |
| 全局级状态 | 系统配置 + 知识库 | 所有用户共享的行业知识、岗位数据、学习资源 |

**上下文窗口管理**:
- 短对话（<10轮）：全量上下文送入LLM
- 长对话（>=10轮）：采用滑动窗口 + 关键信息摘要，保留最近6轮 + 对话摘要
- 话题切换时：保存原话题摘要，加载新话题上下文

### 5.3 工具调用场景与流程

系统Agent具备调用 **7类外部工具** 的能力：

| 工具类别 | 工具名 | 调用场景 | 输入 | 输出 |
|---------|--------|---------|------|------|
| 知识检索 | `knowledge_search` | 用户询问方向/技术/岗位信息时 | 查询关键词 | 结构化知识片段 |
| 路径生成 | `path_generator` | 用户请求学习路径时 | 目标岗位、当前基础、时间参数 | 完整学习路径JSON |
| 简历解析 | `resume_parser` | 用户上传简历时 | 文件URL/二进制 | 结构化简历JSON |
| 简历优化 | `resume_optimizer` | 用户请求简历优化时 | 简历JSON + JD文本 | 优化建议 + 优化后简历 |
| 测评引擎 | `quiz_engine` | 用户进行能力测评时 | 测评维度、难度 | 题目列表 + 评分逻辑 |
| 题库查询 | `question_bank` | 模拟面试出题时 | 方向、模块、难度 | 题目 + 答案 + 追问链 |
| 数据看板 | `market_data` | 用户询问行业数据时 | 方向、城市、时间范围 | 岗位/薪资/趋势数据 |

**工具调用流程示例**（用户请求"帮我生成Java后端学习路线"）：

```
用户输入："帮我生成Java后端学习路线"
    |
[感知] 捕获文本输入
    |
[理解] 意图=path_generation, 实体={direction:"Java后端"}
    |
[规划] 发现缺少必要参数（当前基础、可用时间）
       策略选择：先引导补充信息，再调用path_generator
    |
[行动] LLM生成追问："好嘞！为了给你定制最合适的路线，先问几个小问题：
       你现在Java基础怎么样？每天能投入多少时间学习？"
       -> 用户回答后，调用path_generator工具
    |
path_generator(目标岗位="Java后端", 当前基础="零基础", 每日时间=2h)
    |
返回路径JSON -> Agent转化为可视化卡片
    |
[反馈] 输出学习路径卡片 + "保存到学习档案"按钮 + 追问选项
    |
记忆更新：保存路径到LearningPath表，更新UserProfile.target_direction
```

### 5.4 个性化记忆机制

#### 记忆分层架构

```
+---------------------------------------------+
|  L1: 工作记忆（Working Memory）               |
|     存储：当前对话中的临时信息                   |
|     时效：单轮对话内                           |
|     示例：用户刚才提到的目标公司是字节           |
+---------------------------------------------+
|  L2: 短期记忆（Short-term Memory）             |
|     存储：最近3-5段对话的关键摘要               |
|     时效：7天内活跃对话                        |
|     示例：用户上周确定了Java后端方向             |
+---------------------------------------------+
|  L3: 长期记忆（Long-term Memory）              |
|     存储：用户画像、历史路径、评估结果            |
|     时效：永久（持续更新）                      |
|     示例：用户是大三双非院校，目标中厂           |
+---------------------------------------------+
|  L4: 事实记忆（Factual Memory）               |
|     存储：行业知识、岗位数据、学习资源            |
|     时效：定期更新（非用户特定）                 |
|     示例：2025年Java后端技术栈优先级排序        |
+---------------------------------------------+
```

#### 记忆检索与融合策略

**检索触发条件**:
1. **用户画像自动注入**：每次对话开始时，将用户核心画像（年级、方向、基础）注入System Prompt
2. **相关历史检索**：用户提到某方向/技术时，检索该话题的历史对话摘要
3. **进度状态查询**：用户询问学习/求职进展时，查询最新状态数据

**记忆融合Prompt示例**:
```
【用户背景】
- 张浩，大三，双非院校软件工程
- 目标方向：Java后端（上周确定）
- 当前基础：学过Java SE，Spring Boot入门
- 已生成学习路径：Java后端冲刺路线，当前进度23%
- 上次对话：询问Redis持久化机制，已解答

【当前话题】用户询问"微服务架构是什么"
【回复策略】结合用户基础水平（入门级），用通俗类比解释，
          关联已学知识（Spring Boot），提示学习路径中的后续节点
```

#### 记忆更新机制

| 触发事件 | 更新内容 | 存储位置 |
|---------|---------|---------|
| 用户明确目标方向 | `target_direction`, `target_company_level` | UserProfile |
| 用户提及新技能 | 追加到 `current_skills` | UserProfile |
| 完成能力测评 | 创建SkillAssessment记录，更新画像能力标签 | SkillAssessment + UserProfile |

| 对话中确认信息 | 更新对话上下文快照 | Conversation.context_snapshot |

## 六、用户体验设计原则

### 6.1 视觉风格建议

#### 整体风格定位

**"科技蓝 + 活力橙 + 温暖灰"** — 专业但不冰冷，年轻但不轻浮。

```
主色调：
  Primary: #2563EB（科技蓝）—— 代表技术、专业、信任
  Secondary: #F59E0B（活力橙）—— 代表行动、激励、温暖
  Accent: #10B981（成长绿）—— 代表进步、完成、正向反馈
  
背景色：
  浅色模式：#F8FAFC（极浅灰蓝）
  深色模式：#0F172A（深蓝黑）

文字色：
  标题：#1E293B（深 Slate）
  正文：#475569（中 Slate）
  辅助：#94A3B8（浅 Slate）
```

#### 风格原则

| 元素 | 设计建议 |
|------|---------|
| **界面整体** | 留白充足，卡片式布局，圆角（8px-12px），阴影柔和（微悬浮感） |
| **AI对话界面** | 用户气泡白色/浅色，AI气泡淡蓝色渐变，头像用 friendly 的卡通学长形象 |
| **数据可视化** | 雷达图、环形进度图、漏斗图采用品牌色渐变，数据变化带平滑动画 |
| **学习路径** | 时间线采用纵向/横向节点图，已完成节点绿色+对勾，进行中节点脉冲动画 |
| **求职看板** | Kanban风格，不同状态列用轻微色区分（投递=灰，面试=蓝，Offer=绿） |
| **图标系统** | 线性图标为主，关键操作用面性图标强调，避免过度装饰 |

#### 字体建议

- 中文：系统默认无衬线字体（PingFang SC / Microsoft YaHei）
- 英文/代码：JetBrains Mono（等宽，技术感）
- 标题层级：H1 24px / H2 20px / H3 16px / 正文 14px / 辅助 12px

### 6.2 情感化设计点

#### 设计目标：缓解焦虑、建立信任、激发动力

| 场景 | 情感化设计策略 | 具体实现 |
|------|--------------|---------|
| **用户表达迷茫** | 共情回应 + 给予掌控感 | AI不直接给答案，而是先肯定"迷茫很正常，我大一也完全不知道方向"，再引导"咱们一起梳理一下" |
| **用户表达焦虑** | 数据安抚 + 正常化 | "我统计过，平均投40份简历拿1个Offer，你现在投了20份没回复其实很正常。不过咱们看看是不是简历需要优化" |
| **用户完成里程碑** | 庆祝反馈 + 成长可视化 | 完成学习节点时播放微动效（彩带/徽章弹出），显示"你已超越67%的同阶段用户" |
| **用户面试挂掉** | 接纳情绪 + 建设性复盘 | "挂了很难受吧，我当年三面挂过阿里。但挂掉不是终点，咱们把这次的经验变成下次的武器" |
| **用户长期未登录** | 温和召回 + 进度提醒 | "好久不见！你的Java路线还停留在第3章，要一起接着学吗？只需要15分钟" |
| **深夜时段使用** | 关怀提醒 + 适度幽默 | "这么晚还在学习，注意身体啊。不过既然来了，给你出一道简单的题保持手感，然后早点休息" |

#### 信任感建立机制

1. **透明度**：AI明确标注"这是基于2024年数据""这部分建议仅供参考，建议你到XX再验证"
2. **可验证性**：所有数据建议提供来源链接（如薪资数据来自OfferShow，面经来自牛客）
3. **渐进式承诺**：首次使用不给用户压力，先提供小价值（回答一个问题），再逐步深入
4. **用户掌控感**：所有AI生成的内容用户都可编辑、调整、拒绝（"这个建议不太适合我"->AI重新生成）

#### 成就感体系（游戏化设计）

```
成就徽章示例：
- # 初识方向 — 完成方向测评
- # 路在脚下 — 生成第一条学习路径
- # 连续7天 — 连续7天学习打卡
- # 第一份简历 — 完成简历优化
```

### 6.3 无障碍与包容性设计

| 设计维度 | 具体措施 |
|---------|---------|
| **视觉无障碍** | 所有文字对比度 >= WCAG 2.1 AA标准（4.5:1）；支持系统级字体缩放；关键信息不只依赖颜色传达（如状态用图标+文字） |
| **屏幕阅读器** | 核心功能支持ARIA标签；对话内容结构化（角色标注、卡片语义化）；图片提供alt文本 |
| **认知包容** | 复杂功能提供"简单模式"（减少选项）；专业术语首次出现时提供 tooltip 解释；支持对话历史回溯 |
| **输入包容** | 支持语音输入（方便不方便打字场景）；支持图片上传（简历/笔记拍照）；对话容错（错别字、口语化表达均可理解） |
| **情绪包容** | 检测到用户表达负面情绪时，AI优先安抚而非直接给建议；提供"仅倾听"模式（用户可倾诉而不被打断） |
| **经济包容** | MVP核心功能免费；付费功能提供清晰的"价值说明"；学生认证后可获得折扣 |

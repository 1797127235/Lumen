**文档版本**: v1.0
**撰写日期**: 2025年
**文档定位**: 目标用户需求分析与产品需求定义

## 一、用户画像分析

基于目标用户群体的细分，我们识别出以下 **6 类典型用户**，覆盖不同学业阶段、背景条件和职业规划状态。

---

### 用户画像 1：大一新生 — "迷茫的探索者"

| 维度 | 描述 |
|------|------|
| **基本信息** | 张浩，18岁，某211高校计算机科学与技术专业大一学生。高考成绩优异但对计算机行业缺乏实际认知。日常使用B站、小红书获取碎片信息，信息筛选能力弱。 |
| **核心诉求** | 1. 了解计算机行业有哪些真实的细分方向（前端/后端/算法/运维/产品等）；2. 明确大学四年应该如何安排；3. 判断自己适合哪个技术方向 |
| **典型场景** | 场景A：入学第一个月，听到学长谈论"Java后端""前端React""算法岗"等名词，完全不懂区别，打开B站搜索后信息过载更加迷茫。场景B：期中考试后，发现自己对编程兴趣一般，纠结是否转专业或辅修。 |
| **痛点难点** | 1. **信息过载与碎片化**：网络信息庞杂且质量参差，缺乏系统性梳理；2. **方向认知盲区**：不知道某些细分方向的存在（如SRE、DevOps、MLE）；3. **自我认知模糊**：不清楚自己的性格、能力适合什么方向；4. **缺少同辈参照**：身边同学同样迷茫，辅导员缺乏技术背景难以给出有效建议 |
| **期望解决方案** | 一个能"像学长一样聊天"的AI智能体，用通俗易懂的方式介绍各方向的真实工作内容、发展前景、技能要求；通过互动问答帮助自己逐步明确兴趣方向；生成一份可视化的"大学四年规划路线图" |

### 用户画像 2：大二/大三技术积累期学生 — "焦虑的追赶者"

| 维度 | 描述 |

| **核心诉求** | 1. 获得一条清晰、可执行的技术学习路径；2. 了解每个阶段应该掌握到什么程度；3. 通过实际项目验证学习成果；4. 知道自己与目标岗位的技能差距 |
| **典型场景** | 场景A：在牛客网看到一份字节跳动后端岗JD，要求"熟悉Spring Cloud、Redis、Kafka、MySQL优化"，对照后发现只学过基础Java SE，不知道需要多久才能补齐。场景B：跟着某培训机构视频学了一个"商城项目"，但面试时被问到"如何设计一个秒杀系统"完全答不上来，意识到学的只是CRUD。 |
| **痛点难点** | 1. **学习路径碎片化**：收藏了数十篇技术博客和路线图，但不知从何开始、学到什么程度算够；2. **知识浅层化**：能跑通教程代码，但不懂底层原理，面试一问就露馅；3. **缺乏反馈闭环**：自学过程中无人检验学习质量，不知道自己是否真正掌握；4. **技术更新焦虑**：新技术层出不穷（如AI辅助编程、云原生），担心学错方向或学了就过时；5. **院校背景自卑**：双非院校学生担心学历门槛，需要更清晰的差异化竞争策略 |
| **期望解决方案** | AI智能体根据个人基础（当前已掌握技能、可用时间、目标岗位）生成定制化学习路径，按周/月拆解学习任务；提供阶段性自测和模拟面试检验学习效果；在学习过程中随时答疑（如"为什么Redis单线程还这么快"）；根据学习进度动态调整路径 |

| **典型场景** | 场景A：投了20份简历全部石沉大海，不知道是简历问题还是学历/专业问题。场景B：面试某大厂时被问到"TCP三次握手详细过程""进程和线程区别"，虽然刷过题但底层原理回答不完整。场景C：看到同学大二就开始实习，自己研一才开始转码，时间紧迫感极强。 |
| **痛点难点** | 1. **基础课缺失**：未系统学习过操作系统、计算机网络、数据库等核心课程，面试高频考点覆盖不全；2. **项目经历薄弱**：缺少课程设计或实验室项目支撑，简历上难以写出有说服力的项目；3. **时间压力巨大**：相比科班学生少了3-4年积累时间，需要更高效的浓缩路径；4. **身份标签歧视**：部分企业HR筛选时对非科班简历直接过滤；5. **社交资本匮乏**：缺少计算机专业的同学网络，内推渠道有限 |
| **期望解决方案** | AI智能体为转码学生设计"极速补全计划"，优先覆盖面试最高频的基础知识；提供"转码友好型"公司和岗位清单；帮助包装和打磨项目经历（将原专业项目与计算机结合，如"土木工程+BIM数据分析"）；提供模拟面试并重点强化基础课问答 |

| **典型场景** | 场景A：在某招聘网站海投了50份简历，只收到3个面试邀请，不知道问题出在哪里。场景B：面试某大厂二面挂在了"设计一个微博热搜排行榜"，意识到系统设计是短板但不知从何补起。场景C：拿到2个offer（A公司ssp but 业务边缘，B公司sp but 核心业务），纠结如何取舍。 |
| **痛点难点** | 1. **简历同质化严重**：简历描述停留在"负责XX模块开发"，缺乏数据指标和差异化亮点；2. **面试准备缺乏体系**：算法靠刷题、八股靠背诵，但缺乏面试官视角的系统性准备框架；3. **信息差巨大**：不知道哪些组在扩招、哪些业务线在裁员、哪些面试官友好/严厉；4. **进度管理混乱**：投了哪些公司、哪些在等结果、哪些要准备面试，靠Excel手工管理易遗漏；5. **心态波动大**：面试挂掉后自我怀疑，长时间收不到回复焦虑加剧 |
| **期望解决方案** | AI智能体充当"求职管家"：一键解析JD并匹配个人经历生成定制化简历；提供面试全流程准备（算法模拟、八股问答、系统设计训练、行为面STAR法则指导）；整理目标公司的面经和面试真题；管理求职进度（投递-笔试-面试-offer的全流程跟踪）；在焦虑时提供心理疏导和策略调整建议 |

| **期望解决方案** | AI智能体分析个人科研成果（论文方向、代码实现质量）并映射到匹配的工业界岗位；提供"科研-求职"双轨时间管理方案；基于最新招聘数据给出算法岗/开发岗的取舍建议；帮助将论文工作包装成有业务价值的项目描述 |

## 二、需求场景拆解

按学生成长周期，我们将核心场景划分为 **4 个阶段、10 个具体场景**：

---

### 阶段一：入学适应期（专业认知、方向选择）

#### 场景 1：计算机行业全景认知

| 维度 | 描述 |
|------|------|
| **场景名称** | 计算机行业全景认知与方向初探 |
| **参与角色** | 大一/大二学生、AI职业规划智能体 |
| **用户目标** | 建立对计算机行业的系统性认知，了解各细分方向的工作内容、技能要求、发展前景、薪资区间 |
| **当前痛点** | 1. 信息来源碎片化（知乎、B站、小红书），质量参差不齐且充斥培训机构广告；2. 技术术语陌生（如"SRE""DevOps""MLE""数据工程师"），搜索后仍不理解差异；3. 信息时效性差，很多博客写于3-5年前，行业格局已变化（如2024年大模型崛起带动MLE岗位激增） |
| **系统机会点** | AI智能体以"对话式导览"形式，用大白话介绍各方向；支持"如果我数学好/喜欢可视化/不喜欢写代码"等个性化条件过滤推荐方向；展示各方向的真实工作日常（引用脉脉/知乎上的从业者分享）；提供各方向的"职业成长时间线"（1年/3年/5年做什么） |

#### 场景 2：自我评估与方向匹配

| 维度 | 描述 |
|------|------|
| **场景名称** | 多维自我评估与职业方向匹配 |
| **参与角色** | 低年级学生、AI职业规划智能体 |
| **用户目标** | 通过科学评估了解自己的性格特质、能力优势、兴趣偏好，并获得与计算机细分方向的匹配建议 |
| **当前痛点** | 1. 市面上的MBTI/霍兰德测试通用性强但缺乏计算机行业针对性；2. 测试后只有泛泛建议（"你适合技术类工作"），没有下一步行动指南；3. 学生倾向于自我美化或随机答题，导致结果失真 |
| **系统机会点** | 设计"计算机职业倾向评估"（融合霍兰德、大五人格、技术偏好问卷）；通过互动场景题替代枯燥问卷（如"看到一个bug，你更倾向于A.深挖根因 B.快速绕过继续推进"）；评估后生成"个人-方向匹配雷达图"并给出Top3推荐方向及差异化学习建议 |

#### 场景 3：大学四年规划制定

| 维度 | 描述 |
|------|------|
| **场景名称** | 个性化大学四年成长路径规划 |
| **参与角色** | 大一/大二学生、AI职业规划智能体 |
| **用户目标** | 获得一份可执行、可量化的大学四年规划，明确每学期应该完成什么目标 |
| **当前痛点** | 1. 网上流传的"学习路线图"千篇一律（如"大一学C++，大二学数据结构..."），未考虑个人基础、院校资源、目标公司层级；2. 规划缺少验收标准（"学完Java"学到什么程度？）；3. 缺少时间管理的具体建议（如何在课业和自学之间平衡） |
| **系统机会点** | 智能体基于目标方向（如"Java后端进大厂"）、当前基础（零基础/有编程课基础）、院校层级（985/211/双非）、可用时间（每天2小时/4小时），生成定制化学期规划；每个阶段设置"里程碑验收标准"（如"能独立手写一个带连接池的JDBC工具类"）；支持与学校课程表对接，推荐课余学习时段 |

---

### 阶段二：技能积累期（技术栈选择、学习路径）

#### 场景 4：技术栈选择与学习路径生成

| 维度 | 描述 |
|------|------|
| **场景名称** | 目标导向的技术栈选择与渐进式学习路径 |
| **参与角色** | 大二/大三学生、转码学生、AI职业规划智能体 |
| **用户目标** | 明确目标岗位需要掌握的技术栈（编程语言、框架、工具），并获得循序渐进的学习路径 |
| **当前痛点** | 1. 技术栈选择迷茫：Java还是Go？Vue还是React？MySQL还是PostgreSQL？每个选择都有大量争论帖；2. 学习路径跳跃：看完基础语法直接上框架，中间缺少衔接（如不懂HTTP协议直接学Spring Boot）；3. 盲目跟风：看到某技术登上GitHub Trending就开始学，与目标岗位无关 |
| **系统机会点** | 智能体解析目标岗位JD（如字节后端、美团前端），提取核心技术栈并标注"必须掌握""加分项""了解即可"三级优先级；生成"渐进式学习路径"（如"Java SE → 面向对象设计 → 集合框架源码 → JVM基础 → MySQL → Redis → Spring Boot → 微服务"），每个节点标注预计学习时长和验收标准；根据招聘市场趋势动态调整推荐（如2025年大模型应用开发兴起，推荐学习LangChain/AutoGen） |

#### 场景 5：项目实践与作品积累

| 维度 | 描述 |
|------|------|
| **场景名称** | 有竞争力的项目设计与代码实践指导 |
| **参与角色** | 技术积累期学生、AI职业规划智能体 |
| **用户目标** | 完成1-2个能写入简历、经得起面试深挖的真实项目 |
| **当前痛点** | 1. 教程项目同质化严重："电商系统""图书管理系统"面试官已看腻；2. 项目浅层化：跟着视频抄代码，遇到扩展需求（如"加秒杀功能""支持万级并发"）无从下手；3. 不知道如何包装项目：简历上只写"实现了XX功能"，缺少技术亮点和量化指标 |
| **系统机会点** | 智能体根据目标方向推荐"差异化项目"（如避免商城系统，推荐"分布式任务调度平台""实时日志分析系统"）；提供项目需求文档（PRD）和技术方案设计，由学生自主实现核心模块；在关键节点提供代码Review和架构建议；指导学生用"STAR+技术深度+量化指标"包装项目（如"设计并实现分布式锁，支撑秒杀场景下5000 QPS并发，库存超卖率降至0"） |

#### 场景 6：技能水平检测与差距分析

| 维度 | 描述 |
|------|------|
| **场景名称** | 岗位能力对标与技能差距诊断 |
| **参与角色** | 技术积累期学生/求职者、AI职业规划智能体 |
| **用户目标** | 客观评估自己当前技能水平，了解与目标岗位要求的具体差距 |
| **当前痛点** | 1. 自我评估偏差大：学生往往高估自己（"我学过Redis"但不懂持久化机制）或低估自己；2. 不知道岗位要求的具体深度：JD写"熟悉MySQL"，但面试会问"InnoDB索引结构""慢查询优化"到多深算"熟悉"；3. 缺少"能力对标"工具，不知道大厂P5/字节2-1的能力模型是什么 |
| **系统机会点** | 智能体提供"岗位能力模型拆解"（如字节后端2-1需要：算法中等、Java基础扎实、熟悉1个存储组件、有分布式系统概念、能独立完成需求）；通过在线编程测试+知识问答+系统设计题多维度评估；生成"能力雷达图"与目标岗位对比，标注每个维度的差距等级（达标/接近/差距大/缺失）和补全建议 |

## 四、非功能需求

### 4.1 性能要求

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **响应时间** | 常规对话响应 ≤ 2秒；复杂路径生成（如学习路径规划）≤ 8秒 | 用户与AI智能体对话时，2秒内响应是良好体验的上限；超过5秒会产生明显等待焦虑 |
| **并发支持** | 支持至少 5000 用户同时在线，高峰期（如秋招季晚8-11点）不降级 | 秋招季用户需求集中，系统需能承受脉冲式流量 |
| **吞吐量** | 简历解析模块支持单用户5秒内完成解析；测评模块支持1000人/小时同时测评 | 简历解析涉及OCR/NLP，需优化为异步处理+进度通知 |
| **可用性** | 系统可用率 ≥ 99.5%（月度），计划维护窗口提前公告 | 求职期用户依赖度高，系统宕机可能导致错过关键时间节点 |
| **数据容量** | 支持百万级用户档案存储，单用户学习/求职数据保留至少3年 | 满足用户从入学到毕业后早期的全周期数据沉淀 |

### 4.2 可用性要求

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **多端适配** | 支持Web端（PC）、移动端H5、微信小程序三端，核心功能在移动端完整可用 | 学生群体移动端使用时长远高于PC，求职场景（如面试前临时复习）高度依赖手机 |
| **交互设计** | 对话界面采用"聊天式"设计，降低使用门槛；复杂输出（如学习路径、能力雷达图）采用可视化卡片 | 避免传统表单填写式的复杂交互，让用户"像聊天一样做规划" |
| **无障碍支持** | 核心功能支持屏幕阅读器，文字对比度符合WCAG 2.1 AA标准 | 确保视障学生群体的可访问性 |
| **新手引导** | 首次使用提供3-5步引导，帮助用户完成基础信息录入和首次对话 | 降低新用户流失率，快速建立"这产品能帮我"的认知 |
| **容错设计** | 用户输入模糊/错误时，AI智能体主动澄清（如"你是指Java后端开发还是Android开发？"）而非直接报错 | 学生用户表达能力参差不齐，系统需具备"引导式纠错"能力 |
| **离线缓存** | 已生成的学习路径、简历模板、测评报告支持本地缓存，弱网环境下可查看 | 学生群体在校园网/公共场所网络不稳定场景常见 |

### 4.3 安全性要求（涉及学生隐私数据）

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **数据分级** | 用户数据分为三级：公开级（昵称、学校）、敏感级（手机号、邮箱）、机密级（简历内容、测评结果、求职进度） | 简历中可能包含个人项目经历、联系方式等高度敏感信息 |
| **加密存储** | 敏感级和机密级数据采用AES-256加密存储；传输过程全链路TLS 1.3加密 | 防止数据库泄露导致的大规模隐私暴露 |
| **访问控制** | 用户数据严格隔离，任何管理员/运营人员查看用户原始数据需经审批留痕；AI模型训练使用脱敏数据 | 避免内部人员滥用数据，符合《个人信息保护法》要求 |
| **最小授权** | 第三方组件/服务（如云存储、AI API）仅授予最小必要权限，定期审计API调用日志 | 减少供应链攻击面 |
| **数据删除** | 用户可随时导出并删除全部个人数据，删除操作在72小时内物理清除（含备份） | 满足用户"被遗忘权"，符合GDPR/个保法合规要求 |
| **未成年人保护** | 对未满18岁用户（如大一新生中部分未成年），默认开启最高隐私保护模式，对话记录不上传至模型训练 | 符合未成年人网络保护相关法规 |
| **安全审计** | 每季度进行一次第三方安全渗透测试，漏洞修复SLA：高危24小时、中危7天、低危30天 | 建立持续安全运营机制 |

### 4.4 扩展性要求

| 需求项 | 具体要求 | 说明 |
|--------|----------|------|
| **模块化架构** | 系统采用微服务架构，核心模块（对话引擎、路径规划、简历解析、测评系统）独立部署、独立扩展 | 便于后续迭代和故障隔离 |
| **模型可替换** | AI对话引擎层与底层LLM解耦，支持在不同模型（GPT-4/Claude/国产大模型）之间切换，模型切换不改动业务逻辑 | 平衡成本、性能和合规性（如国内部署需用国产模型） |
| **插件化扩展** | 学习路径中的"学习资源推荐"、测评题库支持插件化接入，新资源/题库可通过配置化方式添加 | 便于对接第三方课程平台、快速扩充题库 |
| **多租户支持** | 数据模型支持未来向高校/培训机构输出SaaS版本（如某大学购买服务给学生使用），租户间数据完全隔离 | 为未来B端扩展预留架构能力 |
| **多语言预留** | 核心数据模型预留多语言字段，为后续面向港澳台/海外计算机留学生扩展做准备 | 全球化扩展的技术预埋 |
| **数据接口开放** | 提供标准REST API和Webhook，支持未来与高校教务系统、第三方招聘平台的数据打通 | 建立生态连接能力 |

## 五、需求优先级总览

### 5.1 MVP 版本（第一阶段：0-3个月）

**目标**：验证核心用户价值，覆盖"最痛的痛点"

功能范围：M-01 ~ M-06（全部Must Have）

**关键成功指标**：
- 用户首次对话完成率 ≥ 70%
- 学习路径生成后7日内用户回访率 ≥ 40%
- 简历优化功能使用率（上传简历用户数/总注册用户）≥ 25%

### 5.2 增强版本（第二阶段：4-6个月）

**目标**：完善核心场景闭环，提升用户粘性

功能范围：S-01 ~ S-06（全部Should Have）+ C-03（校招日历）+ C-04（报告导出）

**关键成功指标**：
- 用户周活跃率 ≥ 35%
- 模拟面试功能人均使用次数 ≥ 3次/月
- 求职进度管理功能在秋招季渗透率 ≥ 30%

### 5.3 生态版本（第三阶段：7-12个月）

**目标**：构建用户社区和生态连接

功能范围：C-01 ~ C-02（社区/导师对接）+ C-05（面经库）+ C-06（多模态）+ 多租户SaaS支持

**关键成功指标**：
- 社区功能日活跃用户占比 ≥ 15%
- 导师对接服务GMV（如有）达到自给自足
- 高校合作SaaS版本签约 ≥ 5 家

## 六、关键假设与风险

### 6.1 关键假设

1. **用户付费意愿假设**：学生群体付费能力有限，MVP阶段应以免费增值模式为主，高级功能（如1v1模拟面试、深度测评）可设置付费门槛
2. **AI回答质量假设**：大模型生成的职业规划建议需要达到"比学长学姐更系统、比辅导员更懂技术"的质量门槛，否则用户不会持续使用
3. **数据获取假设**：能够合法获取并持续更新互联网招聘数据、面经数据、公司信息，用于市场分析和岗位匹配

### 6.2 主要风险

| 风险 | 影响等级 | 缓解策略 |
|------|----------|----------|
| **AI幻觉（Hallucination）**：模型生成错误的技术信息或过时数据 | 高 | 建立技术知识库+RAG检索增强，关键数据人工审核；对AI输出添加"仅供参考，请交叉验证"提示 |
| **隐私泄露**：用户简历、测评数据被非法获取 | 高 | 严格的数据加密和访问控制；定期安全审计；制定应急响应预案 |
| **用户留存不足**：工具用完即走（如简历优化完就不再来） | 中 | 构建学习路径的持续追踪和动态调优，创造"陪伴式"使用场景 |
| **竞品压力**：牛客、LeetCode、脉脉等大平台快速跟进AI功能 | 中 | 聚焦"全链路职业规划"差异化定位，而非单点工具；深耕个性化和情感陪伴 |
| **合规风险**：AI生成内容涉及就业歧视建议或虚假承诺 | 中 | 建立内容审核机制；AI输出添加免责声明；避免给出绝对化承诺（如"保证进大厂"） |

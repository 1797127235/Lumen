## 1. 系统整体架构设计

### 1.1 架构风格选择：模块化单体 → 渐进式微服务

**选型决策**：MVP 阶段采用 **模块化单体（Modular Monolith）**，内部按微服务边界严格模块化，预留平滑拆分路径。

| 架构风格 | 适用阶段 | 选择理由 |
|---------|---------|---------|
| **模块化单体（MVP）** | 0-6个月 | 1. 团队规模有限（通常5-10人），避免分布式复杂度；2. 需求仍在快速验证期，单体迭代速度更快；3. 减少网络开销，满足≤2秒响应要求更容易；4. 内部通过代码级模块隔离（Python包/Go模块），后期拆分成本低 |
| **微服务（演进）** | 6-12个月后 | 1. 用户量突破10万或团队规模扩大后拆分；2. 面试服务、简历解析服务对资源需求差异大，需独立扩缩容；3. 多租户SaaS版本需要独立部署隔离 |

**核心原则**："先跑起来，再拆开"。模块化单体≠混乱的单体，而是内部具备清晰边界、可通过配置切换为独立服务的架构。

**关键交互流程示例（智能对话式职业咨询）**：

```
用户输入 → Kong网关(认证+限流) → 对话服务(WebSocket维持) 
  → Agent编排引擎(意图识别: "咨询Java后端方向")
    → LLM路由(选择模型: 对话类→DeepSeek-V3)
    → RAG检索(查询知识库: Java后端方向介绍)
    → 记忆管理(读取用户画像: 大一新生/零基础)
    → LLM生成个性化回答
  → 流式返回SSE → 用户界面
  → 记忆管理(保存对话摘要)
```

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|

## 2. AI Agent 架构设计

### 2.1 Agent 设计模式选择：Multi-Agent + ReAct

**选型决策**：采用 **Multi-Agent 协作架构**，每个子 Agent 内部使用 **ReAct（Reasoning + Acting）** 模式进行推理。

**理由**：
1. **职责隔离**：职业规划涉及多种截然不同的任务（对话咨询、路径生成、简历优化、模拟面试），单一 Agent 会导致提示词过度复杂、指令冲突。
2. **模型差异化路由**：不同任务对模型能力要求不同（对话需创意性，面试需严谨性，简历需结构化），Multi-Agent 允许为每个子 Agent 配置最优模型。
3. **ReAct 成熟度高**：ReAct 是当前最成熟的单 Agent 推理模式，在工具调用和步骤分解上表现稳定。
4. **预留演进空间**：未来可扩展为 Plan-and-Execute（复杂多步规划）和 AutoGen 式多 Agent 对话。

#### 主控 Agent 调度逻辑（伪代码）

```python
# 主控 Agent 意图识别 & 路由
def main_agent_dispatch(user_input, user_context):
    # Step 1: 意图识别（使用轻量级模型，如 DeepSeek-V3 / Qwen-14B）
    intent = classify_intent(user_input)
    # 意图类别: CONSULTATION, PATH_PLANNING, RESUME, INTERVIEW, ANALYSIS, GENERAL
    
    # Step 2: 任务复杂度评估
    complexity = assess_complexity(intent, user_input)
    
    # Step 3: 路由到子 Agent
    if intent == "CONSULTATION":
        return dialog_agent.run(user_input, user_context)
    elif intent == "PATH_PLANNING":
        return path_agent.run(user_input, user_context)
    elif intent == "RESUME":
        return resume_agent.run(user_input, user_context)
    elif intent == "INTERVIEW":
        return interview_agent.run(user_input, user_context)
    elif intent == "ANALYSIS":
        return analysis_agent.run(user_input, user_context)
    else:
        # 通用对话由对话Agent兜底
        return dialog_agent.run(user_input, user_context)
```

### 2.2 长期记忆机制设计

│              │                │    这类语义回忆                   │

#### 记忆写入与检索流程

**写入流程**：
1. **对话进行中**：每轮对话写入 Redis 短期记忆
2. **对话结束时**：触发摘要生成（轻量模型）→ 提取关键事件 → 写入 PostgreSQL 中期记忆 + Milvus 向量记忆
3. **定期归档**：每日批处理，将中期记忆中的重要事实合并到长期画像（PostgreSQL）

**检索流程**：
1. **对话开始时**：加载用户长期画像 → 注入系统提示词
2. **对话进行中**：从 Redis 加载短期上下文
3. **需要回忆时**：
   - 精确事实 → 查询 PostgreSQL 结构化数据
   - 模糊语义 → Milvus 向量相似度检索（如"用户之前提到过对分布式感兴趣"）
4. **多轮关联**：若用户说"继续刚才的问题"，通过会话ID从 Redis/PostgreSQL 恢复上下文

### 2.4 主动推送机制

├──────────────────┬──────────────────┬─────────────────────────────┤

├──────────────────┼──────────────────┼─────────────────────────────┤
│  定时任务         │  Celery Beat     │  • 每日学习提醒（按用户设置   │
│  (Scheduled)      │  + Quartz        │    的时间）                 │
│                  │                  │  • 校招节点预警（如"字节     │
│                  │                  │    提前批3天后截止"）        │
│                  │                  │  • 周/月学习进度报告         │
├──────────────────┼──────────────────┼─────────────────────────────┤
│  事件触发         │  Kafka +         │  • 用户完成里程碑（"恭喜     │
│  (Event-driven)   │  Webhook         │    完成Redis学习！"）        │
│                  │                  │  • 进度落后预警（"本周计划   │

│  Agent主动决策    │  Agent内部       │  • 检测到用户连续3天未登录   │

#### 多模型路由策略

```python
# 模型路由决策矩阵
ROUTING_MATRIX = {
    # 任务类型 -> (默认模型, 备选模型, 成本上限)
    "general_chat":       ("deepseek-v3",    "qwen-max",     0.005),  # 元/1k tokens
    "career_planning":    ("claude-3.5-sonnet", "gpt-4o",    0.015),
    "resume_optimize":    ("gpt-4o",         "claude-3.5",   0.015),
    "mock_interview":     ("claude-3.5-sonnet", "gpt-4o",    0.02),
    "skill_analysis":     ("deepseek-v3",    "gpt-4o-mini",  0.005),
    "path_generation":    ("claude-3.5-sonnet", "gpt-4o",    0.015),
    "content_safety":     ("gpt-4o-mini",    "qwen-turbo",   0.001),
    "memory_summarize":   ("deepseek-v3",    "qwen-turbo",   0.002),
}
```

**路由规则**：
1. **按任务类型**：不同任务对模型能力要求不同（创意/推理/编码/安全）
2. **按用户等级**：免费用户→轻量模型；付费用户→高端模型
3. **按合规要求**：国内用户→国产模型；海外用户→GPT/Claude
4. **按成本配额**：接近成本上限时自动降级到轻量模型
5. **按模型可用性**：主模型故障时自动切换到备选模型（熔断机制）

#### 简历与岗位匹配向量化方案

```
【简历向量化】
简历文本 → 分段（基本信息/教育/技能/项目/实习）
  → 技能段提取关键词（Java, Spring Boot, Redis, MySQL...）
  → 项目段生成摘要
  → 整体生成技能描述文本
  → Embedding → 用户技能向量 (1024维)

【JD向量化】
JD文本 → 解析（岗位名称/公司/技能要求/经验要求）

  → Embedding → 岗位需求向量 (1024维)

【匹配计算】
匹配度 = cosine_similarity(用户技能向量, 岗位需求向量) * 0.6
       + 关键词精确匹配率 * 0.3
       + 经验年限匹配 * 0.1

结果：匹配度评分 (0-100) + 缺失技能列表 + 建议补充项
```

### 5.2 AI 内容安全架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      【AI 内容安全防护链】                                 │
│                                                                         │

│                                                                         │
│  【第一层：输入过滤】                                                    │
│    • 敏感词过滤：政治、暴力、色情关键词库                                │
│    • Prompt注入检测：检测试图覆盖系统提示词的输入                        │
│    • 毒性检测：调用阿里云绿网API                                        │
│    • 未成年人保护：18岁以下用户限制部分话题                               │
│                                                                         │
│  【第二层：LLM处理安全】                                                 │
│    • 系统提示词加固：明确AI角色边界，禁止医疗/法律/投资建议               │
│    • 温度参数控制：事实类查询 temp=0.1-0.3，创意类 temp=0.5-0.7         │
│    • RAG优先：事实类问题必须检索知识库，禁止纯模型幻觉                    │
│                                                                         │
│  【第三层：输出审核】                                                    │
│    • 事实一致性检查：对关键数据（如薪资、公司信息）与知识库交叉验证        │
│    • 免责声明自动附加：AI输出末尾附加"建议仅供参考..."                   │
│    • 敏感内容二次过滤：输出关键词扫描                                    │
│    • 置信度标注：对不确定的信息标注"据部分用户反馈"等                    │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**幻觉缓解具体措施**：
1. **RAG强制绑定**：职业规划相关问题必须检索知识库，模型只能在检索结果基础上组织语言
2. **关键数据校验**：涉及薪资、公司HC、技术趋势等数据，必须标注数据来源和时效
3. **自洽性检查**：对复杂回答进行"事实一致性"二次验证（轻量模型判断回答是否与检索内容矛盾）
4. **用户反馈闭环**：用户可标记"回答不准确"，运营团队审核后更新知识库

### 5.3 合规方案

## 7. 关键技术风险与缓解策略

### 7.1 风险矩阵

| 风险项 | 影响等级 | 发生概率 | 风险等级 | 缓解策略 |
|--------|---------|---------|---------|---------|
| **LLM调用成本失控** | 高 | 中 | 🔴 高 | 模型路由、响应缓存、Token配额、成本监控 |
| **模型幻觉输出** | 高 | 中 | 🔴 高 | RAG强制绑定、事实校验、用户反馈闭环、免责声明 |
| **高并发响应延迟** | 高 | 中 | 🔴 高 | 流式输出、缓存预加载、异步处理、弹性扩容 |
| **数据隐私泄露** | 高 | 低 | 🟡 中 | AES-256加密、KMS密钥管理、访问审计、最小授权 |
| **LLM服务商故障** | 中 | 低 | 🟡 中 | 多模型冗余、熔断降级、本地轻量模型兜底 |
| **向量数据库性能瓶颈** | 中 | 中 | 🟡 中 | 分区策略、索引优化、缓存热点、水平扩展 |
| **知识库更新滞后** | 中 | 高 | 🟡 中 | 自动化采集、运营审核流程、时效性标注 |
| **用户留存不足** | 高 | 中 | 🔴 高 | 产品策略（非技术）：持续陪伴机制、社区运营 |
| **合规审查风险** | 中 | 低 | 🟡 中 | 算法备案、内容审核、数据脱敏、法律咨询 |

### 7.2 详细缓解策略

#### 风险1: LLM调用成本与配额管理

**问题**：5000用户日活下，若全部使用GPT-4o，月均LLM成本可达¥30,000+；秋招季可能翻倍。

**缓解策略**：
1. **分层模型路由**：
   - 简单对话/摘要 → DeepSeek-V3 / Qwen-Turbo（¥1-2/百万tokens）
   - 复杂推理/路径生成 → Claude 3.5 / GPT-4o（¥15-30/百万tokens）
   - 内容安全/过滤 → GPT-4o-mini / 自研规则（¥0.5/百万tokens）

2. **Token配额管理**：
   - 免费用户：月Token上限500万（约200次长对话）
   - 付费用户：月Token上限5000万
   - 系统级Token池：实时监控，超过阈值触发告警+自动降级

3. **响应缓存**：
   - 相同/相似问题直接返回缓存（Redis存储问答对Embedding索引）
   - 预期缓存命中率：常见问题30-40%

4. **流式输出优化**：
   - 首字延迟优先（用户感知快）
   - 非关键内容分块生成，减少等待

#### 风险2: 模型幻觉问题

**问题**：LLM可能生成错误的技术信息（如"Redis 7.0支持多线程"实际上是6.0），或过时的招聘数据。

**缓解策略**：
1. **RAG强制绑定**：职业规划类问题必须检索知识库，禁止纯模型编造
2. **时效性标注**：所有回答涉及时间敏感信息时，标注"截至2025年X月"+数据来源
3. **置信度分级**：
   - 🔒 知识库有明确答案 → 高置信度
   - ⚠️ 知识库有相关参考 → 中置信度+补充说明
   - ❓ 知识库无参考 → 低置信度+明确提示"以下建议仅供参考"
4. **人工审核飞轮**：运营团队每周抽查AI回答，发现幻觉→更新知识库→减少同类问题幻觉
5. **用户纠错机制**：每个回答提供"不准确"反馈按钮，触发人工复核

#### 风险3: 高并发下的响应延迟

**问题**：秋招季晚8-11点，5000+用户同时在线，可能出现对话响应>5秒。

**缓解策略**：
1. **流式输出（SSE）**：LLM生成过程中逐步返回，首字延迟<1秒，用户感知快
2. **预加载机制**：用户打开页面时预建立WebSocket连接，预加载用户画像到内存
3. **热点数据缓存**：用户画像、常用知识库结果常驻Redis，减少数据库查询
4. **异步处理**：简历解析、复杂路径生成等走异步任务，即时返回"处理中"状态+进度推送
5. **弹性扩容**：HPA + 定时预扩容，提前30分钟准备资源
6. **数据库优化**：对话表按user_id分区 + 索引优化 + 读写分离

#### 风险4: 数据隐私风险

**问题**：用户简历包含大量个人信息，求职进度涉及敏感数据；未成年人需要特殊保护。

**缓解策略**：
1. **加密存储**：机密级数据AES-256-GCM加密，密钥由阿里云KMS管理，轮换周期90天
2. **访问控制**：
   - 用户数据严格按user_id隔离
   - 管理员查看需审批留痕（审计表记录who/when/what/why）
   - AI模型训练仅使用脱敏数据（去标识化+差分隐私）
3. **数据生命周期**：
   - 用户注销后72小时内物理删除（含备份）
   - 3年以上冷数据归档至加密存储
4. **未成年人保护**：
   - 年龄识别：实名认证/微信授权获取年龄
   - 未成年用户：对话记录不上传模型训练池；限制部分功能；家长通知机制

### 7.3 应急预案

| 场景 | 应急措施 | 恢复目标 |
|------|---------|---------|
| 主LLM服务商故障 | 自动切换至备选模型（<30秒）；若全部不可用，启用本地轻量模型（Qwen-7B）兜底，仅处理简单对话 | 核心功能可用 |
| 数据库主库故障 | RDS自动主备切换（<2分钟）；只读流量切换至备库 | 数据零丢失 |
| K8s集群故障 | ACK多可用区自动漂移；关键服务在预留的ECS上启动应急实例 | RTO < 10分钟 |
| DDoS攻击 | WAF自动清洗 + 速率限制降级 + CDN抗DDoS | 服务可用 |
| 数据泄露事件 | 立即启动应急响应：1. 断开泄露源；2. 通知受影响用户；3. 上报监管部门；4. 安全审计 | 24小时内控制 |

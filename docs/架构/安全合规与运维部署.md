# CareerOS — 安全与运维部署

> 文档版本：v2.0 | 2026年5月
> 适用范围：本地优先自托管部署

---

## 1. 安全模型

CareerOS 的安全边界与传统 SaaS 根本不同：**数据在用户本机，安全由本地文件系统权限保障**，没有网络攻击面、没有多用户隔离需求、没有云端数据泄露风险。

### 1.1 数据安全

| 数据类型 | 存储位置 | 保护机制 |
|---------|---------|---------|
| 对话记录 | `~/.careeros/career_os.db` | 本地文件系统权限（用户目录） |
| API Key | SQLite（加密字段）| 内存使用时脱敏，日志不输出原始 Key |
| 用户画像 | SQLite | 不出本机，不传输到任何第三方 |
| Mem0 记忆 | `~/.careeros/chroma_db/` | 本地文件，随 `~/.careeros/` 一起备份/删除 |
| 简历原文 | `~/.careeros/` 文件系统 | 本地文件，用户可直接访问 |

**不存在的安全风险**（无需处理）：
- 无多用户隔离需求（单用户应用）
- 无认证/授权机制（本地文件权限即隔离）
- 无 DDoS 风险（不对外暴露服务）
- 无备份数据加密需求（用户自己的机器）

### 1.2 API Key 保护

```
用户在 Settings 页填写 API Key
    ↓
存入 SQLite（字段在内存读写时脱敏处理）
    ↓
LiteLLM SDK 调用 Provider API 时注入 Authorization Header
    ↓
日志系统过滤：不记录原始 Key，只记录 Key 前缀（如 sk-xxxx...）
```

### 1.3 AI 内容安全

| 措施 | 实现 |
|------|------|
| 建议标注 | AI 所有输出隐含"仅供参考"语境，关键决策明确提示需自行判断 |
| 幻觉缓解 | Mem0 记忆注入提供用户个人上下文，减少 AI 无依据编造 |
| 记忆纠错 | 用户在记忆管理页可删除/修改错误记忆，下次对话立即生效 |

## 2. 部署架构

### 2.1 Docker Compose 单容器

```
docker compose up -d
    ↓
┌─────────────────────────────┐
│   CareerOS 容器              │
│                             │
│   FastAPI :8001             │
│     ├── /api/* (REST + SSE)  │
│     └── /* (静态文件 SPA)    │
│                             │
│   Volume mount:             │
│   ~/.careeros/ → /data/     │
└─────────────────────────────┘
        ↕ 网络
   localhost:8001（仅本机访问）
```

**关键设计**：
- 单容器，单端口（8001），无 nginx/网关
- Volume 挂载 `~/.careeros/` 保证数据在容器重建后持久化
- 生产环境 FastAPI 直接 serve 前端 `dist/`，无 CORS 问题

### 2.2 本地开发

```bash
# 后端
pip install -r requirements.txt
uvicorn app.backend.main:app --reload --port 8001

# 前端（独立终端）
cd app/frontend && npm install && npm run dev
# Vite dev server :5173，/api/* 自动 proxy 到 :8001
```

### 2.3 数据目录结构

```
~/.careeros/
├── career_os.db      # SQLite — 对话、画像、技能、项目、API Keys
├── chroma_db/        # Chroma — Mem0 向量索引
│   └── ...           # ChromaDB 内部文件
└── config.json       # 用户配置（Provider 选择、数据目录路径等）
```

### 2.4 端口冲突处理

默认端口 8001 被占用时：

```bash
PORT=8080 docker compose up -d
```

或在 `.env` 中设置 `PORT=8080`。

## 3. 数据管理

### 3.1 备份

用户只需备份 `~/.careeros/` 目录，包含所有数据：

```bash
cp -r ~/.careeros/ ~/careeros-backup-$(date +%Y%m%d)/
```

### 3.2 数据迁移（版本升级）

版本升级后 `~/.careeros/` 数据必须 100% 兼容：
- SQLAlchemy migration 脚本处理 Schema 变更（向前兼容）
- Chroma 数据格式兼容：升级后按需重建索引（不影响 SQLite 数据）

### 3.3 数据重置

```bash
# 清空对话记忆索引（保留画像/技能/项目数据）
# 在 Settings 页操作，或调用 POST /api/memory/reset

# 完全重置（删除所有数据）
rm -rf ~/.careeros/
docker compose up -d   # 重新初始化
```

## 4. 运维

### 4.1 日志

| 环境 | 日志级别 | 位置 |
|------|---------|------|
| 开发 | DEBUG | stdout |
| Docker 生产 | INFO | stdout（`docker compose logs -f`） |

关键日志事件：
- LLM Provider 调用失败（含错误码）
- Mem0 记忆提取成功/失败
- SQLite 写入失败
- 容器启动完成

### 4.2 健康检查

`GET /api/health` — 返回服务状态，Docker Compose healthcheck 使用。

```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8001/api/health"]
  interval: 30s
  timeout: 10s
  retries: 3
```

### 4.3 常见问题排查

| 问题 | 排查步骤 |
|------|---------|
| 白屏 / `/api/config` 404 | 检查 FastAPI 是否启动；开发模式确认 Vite proxy 配置 |
| 对话无响应 | 检查 Settings 页 API Key 是否填写；`docker compose logs` 看 LLM 调用错误 |
| 端口被占用 | `PORT=8080 docker compose up -d` |
| 记忆不准确 | 进入记忆管理页删除错误条目，或 `/api/memory/reset` 重建索引 |
| 容器数据丢失 | 确认 `docker-compose.yml` 中 volume 挂载正确指向 `~/.careeros/` |

## 5. 技术风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| Mem0 本地部署配置复杂 | 中 | 提供默认配置模板，开箱即用；Settings 页引导填写 |
| LLM Provider 故障 | 中 | 用户可在 Settings 页一键切换 Provider；Ollama 本地模型作为兜底 |
| Docker 镜像体积过大 | 中 | 多阶段构建（node build + python runtime），镜像分层缓存 |
| SQLite 文件损坏 | 低 | WAL 模式保证写入原子性；提供备份文档引导用户定期备份 |
| 大一新生 Docker 门槛 | 高 | 提供 Windows `.bat` 一键启动脚本 + 图文教程 |
| 长期不使用记忆失效 | 低 | Mem0 记忆不主动过期，数据持久存储在本地 |

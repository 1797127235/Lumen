# CareerOS — 系统架构设计

> 文档版本：v2.0 | 2026年5月
> 文档定位：本地优先自托管架构，面向单人全栈开发

---

## 1. 架构概览

CareerOS 是**本地优先的单用户应用**，通过 Docker Compose 一键启动，所有数据存储在用户本地 `~/.careeros/`，无云端依赖。

**核心约束驱动架构决策**：
- 目标用户在本地机器上自己部署，不是多用户 SaaS
- 单人开发，不引入分布式复杂度
- 数据主权：数据绝不出本机

### 1.1 架构分层图

```
┌─────────────────────────────────────────────────────────────┐
│                        【前端层】                             │
│  React 19 + Vite + Tailwind CSS v4 (SPA)                    │
│  路由：react-router-dom | 组件：shadcn/ui                    │
│  开发模式：Vite dev server(:5173) → proxy → FastAPI(:8001)   │
│  生产模式：FastAPI StaticFiles 直接 serve dist/              │
├─────────────────────────────────────────────────────────────┤
│                        【API 层】                             │
│  FastAPI (async) — 单端口 8001                               │
│  路由前缀：/api/*（优先）/ 静态文件 fallback index.html       │
│  SSE 流式输出（对话接口）                                    │
├─────────────────────────────────────────────────────────────┤
│                       【业务逻辑层】                          │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐        │
│  │  对话服务     │ │  画像服务     │ │  配置服务     │        │
│  │ chat_service │ │profile_service│ │config_router │        │
│  └──────────────┘ └──────────────┘ └──────────────┘        │
│  ┌──────────────┐ ┌──────────────┐                         │
│  │  技能服务     │ │  项目服务     │                         │
│  │skill_service │ │project_service│                         │
│  └──────────────┘ └──────────────┘                         │
├─────────────────────────────────────────────────────────────┤
│                       【AI 核心层】                           │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐        │
│  │  Skill 编排   │ │  LLM 路由    │ │  Mem0 记忆    │        │
│  │ orchestrator │ │  llm_router  │ │  自动提取+检索 │        │
│  └──────────────┘ └──────────────┘ └──────────────┘        │
├─────────────────────────────────────────────────────────────┤
│                       【数据层】                              │
│  SQLite (~/.careeros/career_os.db)  — 结构化数据             │
│  Chroma (~/.careeros/chroma_db/)    — Mem0 向量存储后端      │
│  config.json (~/.careeros/)         — 用户配置               │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心组件职责

| 组件 | 技术实现 | 职责 |
|------|---------|------|
| **前端 SPA** | React 19 + Vite | 对话、画像、技能/项目 CRUD、记忆管理、Settings |
| **FastAPI** | Python async | REST API + SSE 流式对话，单端口 serve 前后端 |
| **Skill 编排器** | Python | 自发现 `skills/` 目录 → 意图分类 → 拼装 system prompt |
| **LLM 路由** | LiteLLM SDK | 多 Provider 统一路由（DashScope/OpenAI/DeepSeek/Anthropic/Ollama/OpenRouter） |
| **Mem0 记忆层** | Mem0 + Chroma | 从对话中自动提取关键记忆，语义检索注入上下文 |
| **SQLite** | SQLAlchemy 2.0 async | 对话历史、用户画像、技能记录、项目经历 |
| **Chroma** | PersistentClient | Mem0 的向量存储后端，存储在本地文件系统 |

### 1.3 对话请求完整流程

```
用户发送消息
    ↓
FastAPI /api/chat (SSE 端点)
    ↓
Skill 编排器
  → 意图分类（轻量 LLM 调用）
  → 加载对应 SKILL.md 的 system prompt
    ↓
Mem0 记忆检索
  → 根据当前消息语义检索相关记忆
  → 将记忆片段注入 system prompt 上下文
    ↓
LiteLLM → 用户选择的 Provider（DashScope / OpenAI / Ollama 等）
    ↓
SSE 流式返回 → 前端打字机效果
    ↓
Mem0 后台异步
  → 对话结束后自动提取新记忆写入 Chroma
```

## 2. Mem0 记忆架构

记忆层是 CareerOS 的核心差异化。

### 2.1 工作原理

| 阶段 | 行为 |
|------|------|
| **提取（写）** | 对话结束后，Mem0 分析本轮对话，自动识别关键信息（技能、目标、项目、焦虑点等）并向量化存入 Chroma |
| **检索（读）** | 每次对话开始前，用用户消息语义检索 Chroma，返回 top-k 相关记忆注入 system prompt |
| **管理** | 用户在记忆管理页面查看所有记忆，可删除/修改；删除后下次对话不再检索到 |

### 2.2 存储位置

```
~/.careeros/
├── career_os.db      # SQLite — 对话记录、画像、技能、项目
├── chroma_db/        # Chroma — Mem0 向量索引
└── config.json       # 用户配置（API Key、Provider 选择等）
```

### 2.3 用户数据类型

| 数据类型 | 存储 | 说明 |
|---------|------|------|
| 对话记录 | SQLite | 按 conversation_id 分组，消息历史 |
| 用户画像 | SQLite | 学校、年级、目标方向等结构化字段 |
| 技能记录 | SQLite | 表单 CRUD + 简历自动同步 |
| 项目经历 | SQLite | 表单 CRUD + 简历提取 |
| Mem0 记忆 | Chroma | 从对话自动提取，可在记忆管理页查看 |
| API Key | SQLite | 各 Provider Key，仅存本地，不传输 |
| 用户配置 | JSON | Provider 选择、模型选择、数据目录路径 |

## 3. LLM 路由架构

### 3.1 多 Provider 设计

```
用户配置 Provider（Settings 页）
    ↓
LiteLLM SDK（非 Proxy 模式）
    ↓
┌─────────────────────────────────────────┐
│  DashScope (通义千问)                    │
│  OpenAI / OpenAI-compatible             │
│  DeepSeek                               │
│  Anthropic (Claude)                     │
│  Google Gemini                          │
│  Ollama (本地模型)                       │
│  OpenRouter                             │
│  Custom (自定义 base_url)               │
└─────────────────────────────────────────┘
```

### 3.2 Skill 意图分类

- `skills/` 目录自发现：每个子目录代表一个意图（如 `career_planning`、`resume_polish`）
- 每个意图目录包含 `SKILL.md`，frontmatter 定义 `task_type`（决定用哪档模型）
- 意图分类用轻量 LLM 调用（低成本），主对话调用高质量模型

## 4. 架构决策记录（ADR）

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|
| 部署模型 | Docker Compose 单容器 | K8s / 云服务 | 目标用户是个人自托管，单容器零运维 |
| 数据库 | SQLite | PostgreSQL | 单用户无并发压力，SQLite 零运维，文件直接备份 |
| 记忆层 | Mem0 + Chroma | 手写 RAG / LlamaIndex | Mem0 自动提取免去手写意图识别逻辑，Chroma 本地持久化 |
| LLM 接入 | LiteLLM SDK | LiteLLM Proxy / 单一 Provider | 用户自选厂商和模型，SDK 直调减少一层网络开销 |
| 前端架构 | React SPA | Next.js / SSR | 本地部署无 SEO 需求，SPA 更简单，生产环境 FastAPI 直接 serve |
| 认证 | 无 | JWT / Session | 单用户本地应用，文件系统权限即隔离 |
| 流式输出 | SSE | WebSocket | SSE 单向推送满足需求，实现更简单 |
| 状态管理 | React Context + useReducer | Redux / Zustand | 单用户无复杂共享状态，避免引入额外依赖 |

## 5. 开发 vs 生产环境

| 环境 | 前端 | 后端 | 说明 |
|------|------|------|------|
| **开发** | Vite dev server `:5173` | FastAPI `:8001` | Vite proxy 转发 `/api` 到后端，HMR 热重载 |
| **生产（Docker）** | FastAPI StaticFiles serve `dist/` | FastAPI `:8001` | 单端口，无 CORS 问题，`/api/*` 优先路由 |

### 环境变量优先级

```
config.json > .env > 代码默认值
```

关键配置项：
- `USER_DATA_DIR`：默认 `~/.careeros/`，可自定义数据目录
- `PORT`：默认 8001，端口冲突时覆盖
- `DEBUG`：开发 `true`，Docker 生产 `false`

## 6. 性能指标与约束

| 指标 | 目标 | 说明 |
|------|------|------|
| 容器启动时间 | ≤ 30 秒 | `docker compose up -d` 到可访问 |
| 对话首 token | ≤ 3 秒 | 用户发送到 AI 开始回复（网络正常） |
| 流式间隔 | ≤ 100ms/token | SSE 推送，感知流畅 |
| Idle 内存 | ≤ 500MB | 含 SQLite + Chroma + FastAPI |
| 首屏加载 | ≤ 3 秒 | 本地部署，无 CDN |
| 前端包体积 | ≤ 1MB gzip | SPA 初始包 |
| 数据库查询 | ≤ 100ms | SQLite 单文件，无并发压力 |

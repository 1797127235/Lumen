## 1. 系统整体架构设计

### 1.1 架构风格选择：模块化单体 → 渐进式微服务

**选型决策**：MVP 阶段采用 **模块化单体（Modular Monolith）**，内部按微服务边界严格模块化，预留平滑拆分路径。

| 架构风格 | 适用阶段 | 选择理由 |
|---------|---------|---------|
| **模块化单体（MVP）** | 0-6个月 | 1. 团队规模有限（通常5-10人），避免分布式复杂度；2. 需求仍在快速验证期，单体迭代速度更快；3. 减少网络开销，满足≤2秒响应要求更容易；4. 内部通过代码级模块隔离（Python包/Go模块），后期拆分成本低 |
| **微服务（演进）** | 6-12个月后 | 1. 用户量突破10万或团队规模扩大后拆分；2. 面试服务、简历解析服务对资源需求差异大，需独立扩缩容；3. 多租户SaaS版本需要独立部署隔离 |

**核心原则**："先跑起来，再拆开"。模块化单体≠混乱的单体，而是内部具备清晰边界、可通过配置切换为独立服务的架构。

### 1.3 核心组件清单及交互关系

| 组件 | 技术实现 | 职责 | 上下游关系 |
|------|---------|------|-----------|
| **Kong Gateway** | Kong 3.x (Open Source) | API网关：认证、限流、路由、TLS终止 | ← CDN/WAF → 应用服务 |
| **用户服务** | Python FastAPI | 用户注册/登录、画像管理、权限控制 | ← Kong → PostgreSQL, Redis |
| **对话服务** | Python FastAPI + SSE | WebSocket/SSE 长连接管理、多轮对话状态机 | ← Kong → Agent编排引擎, Redis |
| **路径服务** | Python FastAPI | 学习路径生成、进度追踪、动态调整 | ← Agent编排引擎 → PostgreSQL |
| **简历服务** | Python FastAPI + Celery | 简历解析（OCR+NLP）、JD匹配、优化建议 | ← 异步任务 → PostgreSQL, Milvus |
| **面试服务** | Python FastAPI | 题库管理、AI模拟面试、评分反馈 | ← Agent编排引擎 → PostgreSQL |
| **求职服务** | Go Gin | 求职进度管理、时间节点追踪（高性能CRUD） | ← Kong → PostgreSQL |
| **通知服务** | Go Gin + Kafka Consumer | 消息推送（微信/短信/站内信）、定时提醒 | ← Kafka → 微信SDK/短信网关 |
| **Agent编排引擎** | Python + LangChain 0.2 | Agent调度、意图识别、工作流编排 | ← 对话/面试/路径服务 → LLM路由/工具编排 |
| **LLM路由** | Python | 模型选择策略、请求分发、响应缓存、流式处理 | ← Agent编排引擎 → GPT-4/Claude/DeepSeek |
| **RAG检索** | Python + Milvus | 向量索引构建、语义检索、重排序 | ← Agent编排引擎 → Milvus + PostgreSQL |
| **记忆管理** | Python + Redis + PostgreSQL | 短期上下文、会话摘要、长期画像记忆 | ← Agent编排引擎 → Redis/PostgreSQL |
| **工具编排** | Python | 外部API调用编排（JD解析、面经检索、日历） | ← Agent编排引擎 → 外部服务 |

**关键交互流程示例（智能对话式职业咨询）**：

```
用户输入 → Kong网关(认证+限流) → 对话服务(WebSocket维持) 
  → Agent编排引擎(意图识别: "咨询Java后端方向")
    → LLM路由(选择模型: 对话类→DeepSeek-V3)
    → RAG检索(查询知识库: Java后端方向介绍)
    → 记忆管理(读取用户画像: 大一新生/零基础)
    → LLM生成个性化回答
  → 流式返回SSE → 用户界面
  → 记忆管理(保存对话摘要)
```

### 1.4 架构决策记录（ADR）

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|
| 架构风格 | 模块化单体→微服务 | 纯微服务(MVP) | 避免过早拆分，降低初期复杂度 |
| 通信协议 | REST + WebSocket/SSE | gRPC | WebSocket适合AI流式输出；REST降低前端接入门槛 |
| 会话状态 | Redis + 数据库 | 纯内存 | 满足5000并发+故障恢复需求 |
| 文件存储 | MinIO/S3 | 本地NAS | 高可用、弹性扩展、CDN友好 |

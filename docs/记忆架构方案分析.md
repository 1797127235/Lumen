# CareerOS 记忆架构方案分析

## 1. 问题背景

CareerOS 是一个对话式 AI 职业规划助手，使用 PydanticAI + DashScope (qwen-plus) 构建。当前存在以下根本性问题：

### 1.1 现状

```
每次 LLM 调用拿到的上下文:
  system_prompt（静态）
  + dynamic_prompt:
      UserProfile（DB 查询）
      growth_events 最近 3 条（按时间序，无语义检索）
  + user_input（当前这条消息）

缺失：
  ❌ 会话历史（上一轮说了什么，AI 看不到）
  ❌ 会话摘要（conversation.summary 只写不读）
  ❌ 跨会话记忆（Cognee 链路完全断裂，growth_events 从未被写入）
```

### 1.2 核心矛盾

AI 每一轮对话都是全新启动，没有任何历史上下文。用户说"我是大三的"，AI 回复"好的"，用户接着问"那我该学什么"，AI 已经不知道用户是大三的了。

### 1.3 涉及文件

| 文件 | 职责 |
|------|------|
| `app/backend/services/chat_service.py` | 对话入口，调 agent.run_stream()，**没传 message_history** |
| `app/backend/agent/pydantic_agent.py` | Agent 定义，system_prompt + dynamic_prompt（仅加载 UserProfile） |
| `app/backend/agent/pydantic_tools.py` | 2 个工具：get_profile / update_profile（JD 诊断仅保留 REST `/api/jd/diagnose`，不在 Agent 工具中） |
| `app/backend/services/cognee_service.py` | 记忆层封装，recall() 降级为 SQLite 时间序查询 |
| `app/backend/services/cognee_projector.py` | 事件投影到 Cognee，**从未被调用** |
| `app/backend/models/conversation.py` | Conversation 模型，有 summary 字段但未被读取 |
| `app/backend/models/growth_event.py` | GrowthEvent 模型，**从未有业务流程写入** |

---

## 2. 需要解决的三层记忆

| 层级 | 名称 | 含义 | 举例 |
|------|------|------|------|
| L1 | 短期记忆（会话内） | 同一次对话中上一轮说了什么 | "用户上一句说了他是大三的" |
| L2 | 结构化状态 | 用户画像的持久化字段 | school_name="双非"，grade="junior" |
| L3 | 长期记忆（跨会话） | 跨多次对话的事实和偏好 | "用户上周诊断过字节 JD，匹配度 45 分" |

---

## 3. PydanticAI 框架约束

> **关键陷阱**：PydanticAI 官方文档明确说明 —— 当 `message_history` 非空时，`@agent.system_prompt` 不会再执行。

这意味着：
- 如果传了 `message_history`，`dynamic_prompt`（包含 UserProfile 加载）不会执行
- 需要使用 `ReinjectSystemPrompt` capability 才能让 system_prompt 重跑
- 但即使重跑，如果 UserProfile 在上一轮被 update_profile 更新了，system_prompt 的内容也可能过时

相关文档：https://ai.pydantic.dev/message-history

---

## 4. 短期记忆方案对比（L1）

### 方案 A：在 dynamic_prompt 里注入历史文本

**做法**：不使用 PydanticAI 的 `message_history`，而是在 `dynamic_prompt` 里从 DB 加载最近 N 条消息，格式化为文本段落注入 system prompt。

```python
@agent.system_prompt
async def dynamic_prompt(ctx: RunContext[CareerOSDeps]) -> str:
    # 1. 加载 UserProfile（已有）
    # 2. 加载 conversation.summary（如果有）
    # 3. 加载最近 20 条消息 → 格式化为:
    #    user: 我是大三的...
    #    assistant: 好的，已记录...
    # 4. 拼接返回
```

**优点**：
- 实现简单，改动最小（只改 pydantic_agent.py）
- UserProfile 每次都从 DB 读最新值，不存在过时问题
- 不触发 PydanticAI 的 "message_history 非空则跳过 system_prompt" 陷阱

**缺点**：
- 工具调用历史变成纯文本，AI 看不到结构化的工具调用和返回值
- 全部塞进 system prompt，token 开销更大
- 不是 PydanticAI 推荐的标准用法

**适合场景**：MVP / 快速修复，工具调用不频繁的对话场景

---

### 方案 B：存储 PydanticAI 原生消息 JSON + ReinjectSystemPrompt

**做法**：每轮对话结束后，用 `ModelMessagesTypeAdapter` 把 `response.all_messages()` 序列化存入 DB。下轮加载回来作为 `message_history` 传入 agent，同时启用 `ReinjectSystemPrompt` capability。

```python
from pydantic_ai import ModelMessagesTypeAdapter
from pydantic_ai.capabilities import ReinjectSystemPrompt

# 创建 agent 时添加 capability
agent = Agent(
    model=model,
    capabilities=[ReinjectSystemPrompt()],
    ...
)

# 对话时
history = load_message_history_from_db(conversation_id)  # list[ModelMessage]
async with agent.run_stream(
    user_input,
    deps=deps,
    message_history=history,
) as response:
    ...

# 流结束后
new_messages = response.all_messages()
save_message_history_to_db(conversation_id, new_messages)
```

**优点**：
- PydanticAI 标准用法，工具调用历史完整保留（包括 tool call + tool return 配对）
- AI 能看到之前调用过哪些工具、返回了什么结果
- 对话追踪、调试更方便

**缺点**：
- 需要新增存储列或表来存 PydanticAI 格式的消息 JSON
- 与现有 `messages` 表存在数据重叠（现有表存的是 role/content 纯文本，PydanticAI 格式包含 parts/tool_calls 等结构）
- `ReinjectSystemPrompt` 是否能让 `dynamic_prompt` 每次重跑需要验证
- 工具调用消息体可能很大，需要做 token 管理

**适合场景**：正式实现，工具调用是核心交互方式

---

### 方案 C：混合方案（推荐探索）

**做法**：`message_history` 传最近 N 条（保留工具调用结构），同时把 UserProfile + 会话摘要作为最新的 system prompt 注入。

```python
# 1. 从 DB 加载最近 20 条 PydanticAI 格式消息
history = load_native_messages(conversation_id, limit=20)

# 2. 构造一条 system-level 的画像注入
#    放在 message_history 的开头，作为第一条 ModelRequest
profile_context = await build_profile_context(db, user_id, conversation_id)

# 3. 传给 agent
async with agent.run_stream(
    user_input,
    deps=deps,
    message_history=[profile_context_message] + history,
) as response:
    ...
```

**优点**：
- 工具调用结构保留
- UserProfile 每次都是最新的（手动注入，不依赖 dynamic_prompt）
- 摘要可以放在 profile_context_message 里一起注入

**缺点**：
- 需要手动构造 ModelRequest/ModelResponse 对象
- 消息列表管理逻辑更复杂
- 需要处理 token 超限时的裁剪策略

---

### L1 方案决策矩阵

| 维度 | 方案 A（文本注入） | 方案 B（原生消息） | 方案 C（混合） |
|------|---|---|---|
| 实现复杂度 | 低 | 中 | 中高 |
| 工具调用保真度 | ❌ 丢失结构 | ✅ 完整 | ✅ 完整 |
| UserProfile 实时性 | ✅ 每次读最新 | ⚠️ 依赖 ReinjectSystemPrompt | ✅ 手动注入 |
| Token 效率 | ⚠️ 全在 system prompt | ✅ 分散在 history | ✅ 分散在 history |
| 改动范围 | 仅 pydantic_agent.py | chat_service + 新存储 | chat_service + context_builder |
| PydanticAI 标准度 | 非标准 | 标准 | 半标准 |

## 5. 长期记忆方案对比（L3）

### 现状：Cognee

**设计意图**：知识图谱 + 向量搜索，事件投影到图谱实体

**实际状态**：
- `growth_events` 表从未被任何业务流程写入（chat/profile/tools 均不写）
- `cognee_projector.py` 有完整投影逻辑但从未被调用
- `cognee_service.recall()` 直接走 `_recall_from_sqlite()`，Cognee 语义搜索被绕过
- 整条链路等于不存在

**如果要修复 Cognee**：
1. 在 update_profile 工具成功后写 GrowthEvent（若恢复 Agent 侧 JD 工具或 REST 诊断后写事件，可另增 `jd_diagnosed` 等）
2. fire-and-forget 调用 cognee_projector.project_event()
3. 修复 cognee_service.recall() 的两阶段逻辑（Cognee 优先，SQLite 降级）
4. 确保 Cognee 服务本地可用且能正常运行

**选型说明**：本项目长期记忆以 **Cognee + growth_events + 投影** 为主线，**不引入**第三方通用记忆服务；L3 仅在「修复完整链路」与「降级简化」之间取舍。

---

### L3 内部选项对比（均在现有栈内）

| 维度 | 修复 Cognee 全链路 | 简化降级（仅 SQLite） |
|------|-------------------|----------------------|
| 语义 / 图谱能力 | 具备（需服务可用） | 无，仅时间序或关键词过滤 |
| 工作量 | 中（写事件 + 投影 + recall） | 低（写事件 + SQL 注入上下文） |
| 依赖 | Cognee 运行时 | 仅现有 DB |
| 适用 | 希望保留图谱与语义检索设计 | 先跑通数据流，再决定是否启用 Cognee 检索 |

---

### L3 方案决策矩阵

| 选项 | 描述 | 工作量 | 风险 |
|------|------|--------|------|
| 修复 Cognee | 打通 growth_events → projector → recall 链路 | 中 | 高（Cognee 本身稳定性需验证） |
| 暂时搁置 | 先解决 L1（短期记忆），L3 后续再做 | 低 | 无（不影响核心对话体验） |
| 简化保留 | 保留 growth_events 表，改为 SQL 查询注入上下文，暂不启用 Cognee 语义检索 | 低 | 低（放弃图谱语义，但事件可追溯） |

---

## 6. 建议优先级

```
P0: 解决短期记忆（L1）→ 用户在同一对话内不再失忆
    选方案 A/B/C 之一

P1: 解决 L2（update_profile 字段补全）→ 已完成大部分

P2: 决定 L3 方向（修复 Cognee 全链路 / 简化仅 SQLite / 暂时搁置）

P3: 前端反馈（profile_saved 事件、摘要可见性等）
```

**建议**：先用最小成本解决 P0，让对话体验能正常工作，再考虑 L3。

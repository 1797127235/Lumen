# CareerOS 记忆层架构变更

## 1. 变更目标

本次变更的目标不是合并 `.md` 文件、SQLite 和 Cognee，而是建立正确的写入与投影关系：

- `growth_events` 作为唯一真相源，记录所有长期记忆事件。
- `.md` 文件作为用户画像与实体档案的可读投影，用于同步注入 prompt 和前端展示。
- Cognee 作为语义索引与图谱检索层，用于长期记忆召回。

旧架构的问题是两条写入链路彼此独立：

```text
Agent 工具 -> .md 文件

对话提取 -> growth_events -> Cognee
```

新架构统一为：

```text
所有写入 -> growth_events
                |
     +----------+----------+
     |                     |
     v                     v
.md 投影器            Cognee 投影器
 同步阻塞              异步后台
     |                     |
     v                     v
memory.md              Cognee 索引
entities/*.md
```

## 2. 存储职责

| 存储 | 定位 | 是否真相源 | 主要用途 |
|---|---|---:|---|
| `growth_events` | 事件日志 | 是 | 记录用户成长轨迹、画像变化、技能、经历、偏好、目标、状态、决策 |
| `memory.md` / `entities/*.md` | Markdown 投影 | 否 | prompt 同步注入、人工可读、前端画像展示 |
| Cognee | 语义索引 / 图谱索引 | 否 | 长期记忆语义召回、相关事件检索 |

核心原则：

- 所有写入入口都必须先写 `growth_events`。
- `.md` 不允许作为长期记忆写入入口，只能由投影器生成。
- Cognee 不允许作为真相源，只能从 `growth_events` 重建。
- `.md` 投影必须同步完成，保证下一轮 `dynamic_prompt` 读取到最新画像。
- Cognee 投影可以异步，失败不回滚 SQLite，但必须可重试、可重建。

## 3. 当前架构问题

| 问题 | 当前表现 | 影响 |
|---|---|---|
| Agent 工具只写 `.md` | `memory_update` / `memory_add` 直接改 Markdown | 长期记忆事件缺失，Cognee 无法召回 |
| 对话提取只写 events | extractor 写 `growth_events`，不更新 `.md` | prompt 里的画像可能是旧的 |
| 简历上传绕过 events | 简历解析后直接覆盖 `memory.md` | 画像变化没有完整事件轨迹 |
| `clear_user_index()` 占位 | reset 只删 SQLite，Cognee 旧索引还在 | reset 后仍可能召回旧记忆 |
| rebuild 追加重复 | 重放 events 时没有清旧索引或 upsert | Cognee 里同一事件重复 |
| 7 天窗口误去重 | 只按 `event_type/entity_type/entity_id` 判断 | 不同目标、经历、偏好可能被误删 |
| 5 分钟投影窗口 | commit 后投影最近 5 分钟所有事件 | 并发和连续对话会重复投影 |
| `memory_search` 只搜 `.md` | 不查 `growth_events` | 搜不到事件时间线和未投影信息 |

## 4. 数据模型变更

修改 `app/backend/models/growth_event.py`。

建议新增字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `dedupe_key` | `String(128), nullable=True, index=True` | 精确去重键，由事件类型、实体、payload 关键字段生成 |
| `payload_hash` | `String(64), nullable=True, index=True` | payload 规范化后的哈希，用于判断内容是否重复 |
| `projected_md_at` | `DateTime, nullable=True` | `.md` 投影成功时间 |
| `projected_cognee_at` | `DateTime, nullable=True` | Cognee 投影成功时间 |

可选后续字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `conversation_id` | `String(36), nullable=True` | 追踪对话来源 |
| `confidence` | `Float, default=1.0` | LLM 提取置信度 |
| `source_event_id` | `String(36), nullable=True` | 支持更复杂的修正/撤销关系 |

## 6. 关键函数

### 6.1 `create_growth_event_with_dedup()`

职责：创建事件，并进行精确去重。

建议接口：

```python
async def create_growth_event_with_dedup(
    db: AsyncSession,
    *,
    user_id: str,
    event_type: str,
    entity_type: str | None = None,
    entity_id: str | None = None,
    payload: dict | None = None,
    source: str = "system",
    dedupe_key: str | None = None,
) -> tuple[GrowthEvent, bool]:
    ...
```

返回值：

- `GrowthEvent`：新建或已存在事件。
- `bool`：是否实际新建。

去重规则：

- 优先使用调用方传入的 `dedupe_key`。
- 未传时由 `event_type + entity_type + entity_id + payload_hash` 生成。
- 不再使用粗粒度 7 天窗口作为主去重逻辑。

### 6.2 `create_event_and_project_md()`

职责：统一写入入口。创建事件后同步投影 `.md`，保证下一轮 prompt 可读到最新状态。

建议接口：

```python
async def create_event_and_project_md(
    db: AsyncSession,
    *,
    user_id: str,
    event_type: str,
    entity_type: str | None = None,
    entity_id: str | None = None,
    payload: dict | None = None,
    source: str = "system",
    dedupe_key: str | None = None,
    project_cognee: bool = True,
) -> GrowthEvent:
    ...
```

处理顺序：

1. 写入 `growth_events`。
2. `db.commit()` 或确保调用方已提交。
3. 同步调用 `project_user_to_md(user_id)`。
4. 后台触发 Cognee 精确投影。

注意：如果函数内部负责 commit，则调用方不要再复用未提交事务做其他强一致写入；如果由调用方负责 commit，则函数名应明确为 `create_event_then_project_md_after_commit()`。

### 6.3 `project_user_to_md()`

职责：从 `growth_events` 全量重建 `.md` 快照。

建议接口：

```python
async def project_user_to_md(db: AsyncSession, user_id: str) -> None:
    ...
```

投影规则 MVP：

| 实体 | 规则 |
|---|---|
| `profile` | 按字段取最新值 |
| `status` | 按字段取最新值 |
| `preference` | 按 `entity_id` 或偏好类别取最新值 |
| `goal` | 按 `entity_id` 合并，状态取最新 |
| `skill` | 按 `entity_id` 合并，level 取最新 |
| `experience` | 追加列表，按 `dedupe_key` 去重 |
| `decision` | 追加时间线，按创建时间倒序或正序展示 |
| `relationship` | 按 `entity_id` 合并 |

投影输出：

```text
~/.careeros/memory/memory.md
~/.careeros/memory/entities/skills.md
~/.careeros/memory/entities/experiences.md
~/.careeros/memory/entities/preferences.md
~/.careeros/memory/entities/goals.md
~/.careeros/memory/entities/decisions.md
~/.careeros/memory/entities/relationships.md
~/.careeros/memory/entities/status.md
```

## 7. 写入入口改造

### 7.1 Agent 工具

修改 `app/backend/agent/pydantic_tools.py`。

变更前：

```text
memory_update / memory_add -> memory_service -> .md
```

变更后：

```text
memory_update / memory_add
  -> create_event_and_project_md()
  -> growth_events
  -> 同步 .md 投影
  -> 后台 Cognee 投影
```

要求：

- 工具返回前 `.md` 必须已投影完成。
- 工具返回内容应说明“已记录”，但不能承诺 Cognee 已同步完成。

### 7.2 简历上传

修改 `app/backend/services/profile_service.py`。

变更前：

```text
简历解析 -> 直接 write_memory(markdown)
```

变更后：

```text
简历解析 -> 结构化事件列表 -> growth_events -> .md 投影
```

MVP 可先写：

- `resume_uploaded`
- `profile_updated`

后续增强：

- 从简历结构化提取 `skill_added`
- 从项目、实习、获奖提取 `experience_added`
- 从求职方向提取 `goal_updated`

### 7.3 对话后提取

修改 `app/backend/services/memory_extractor.py`。

变更点：

- 不再直接调用低层 `create_growth_event()`。
- 改用统一入口。
- 去重使用 `dedupe_key/payload_hash`。
- Cognee 投影按新建 event ids 精确触发，不再使用最近 5 分钟窗口。

### 7.4 `/api/memory/*`

修改 `app/backend/routers/memory.py`。

接口语义：

| 接口 | 新语义 |
|---|---|
| `GET /api/memory/stats` | 返回 `growth_events` 数量、待投影数量、Cognee 状态 |
| `GET /api/memory/list` | 返回事件列表，而不是 Cognee 内部列表 |
| `GET /api/memory/search` | 同时查 `growth_events` 和 `.md` 投影 |
| `POST /api/memory/reset` | 单用户模式下清 SQLite、重建空 `.md`、清 Cognee 数据集 |
| `POST /api/memory/rebuild` | 从 SQLite 重建 `.md` 和 Cognee |

单用户模式下，Cognee 不支持按用户删除时，可以采用 drop dataset 或删除本地 Cognee 存储目录的策略。该操作必须只在单用户模式启用。

## 8. Cognee 投影策略

当前限制：Cognee 暂不可靠支持按 `user_id` 删除。CareerOS 当前是单用户应用，因此可采用单用户清空策略。

要求：

- 投影按 `event_id` 精确执行。
- 投影成功后更新 `projected_cognee_at`。
- 投影失败不影响主流程，但必须记录日志。
- rebuild 前必须先清理 Cognee 单用户索引，否则会重复追加。

建议新增：

```python
async def project_event_by_id(event_id: str) -> bool:
    ...

async def rebuild_cognee_from_events(user_id: str) -> dict:
    ...

async def reset_cognee_single_user() -> bool:
    ...
```

## 9. 竞态与一致性

### 9.1 `.md` 投影必须同步

原因：`dynamic_prompt` 每轮直接读取 `memory.md`。如果 Agent 工具写入后 `.md` 投影异步执行，下一轮对话可能读到旧画像。

规则：

- `growth_events` 写入成功后，同步执行 `.md` 投影。
- `.md` 投影失败时，写入接口应返回失败或至少明确告警。
- 不允许出现“事件已写入，但 prompt 画像仍旧”的静默状态。

### 9.2 Cognee 投影可以异步

原因：Cognee 是召回增强，不是同步画像上下文的唯一来源。

规则：

- 异步投影失败不回滚事件。
- 需要 `projected_cognee_at` 支持后续补偿。
- `/api/memory/rebuild` 可作为人工修复入口。

## 10. 实施阶段

### Phase 1：补强事件模型和统一写入

交付：

- `GrowthEvent` 新字段。
- `create_growth_event_with_dedup()`。
- `create_event_and_project_md()`。
- 基础单元测试。

验收：

- 同一 `dedupe_key` 不重复创建事件。
- 写入事件后能同步触发 `.md` 投影。

### Phase 2：实现 `.md` 投影器

交付：

- `services/md_projector.py`。
- 从 events 全量重建 `memory.md/entities/*.md`。
- 投影规则覆盖 profile、skill、experience、preference、goal、decision、status。

验收：

- 删除 `.md` 后调用 rebuild 可完整恢复。
- 同一技能多次更新后，Markdown 只展示最新状态。
- 经历和决策按事件时间线展示。

### Phase 3：改造所有写入入口

交付：

- Agent 工具改造。
- 简历上传改造。
- 对话提取改造。

验收：

- Agent 工具不再直接写 `.md`。
- 简历上传不再绕过 `growth_events`。
- 对话提取不再使用 5 分钟投影窗口。

### Phase 4：修复管理面和 Cognee 重建

交付：

- reset 清空 SQLite、`.md`、Cognee 单用户索引。
- rebuild 从 SQLite 重建 `.md` 和 Cognee。
- memory search 同时查 events 与 `.md`。

验收：

- reset 后不会召回旧 Cognee 记忆。
- rebuild 连续执行两次不会制造重复召回。
- search 能搜到未出现在核心画像里的事件。

### Phase 5：测试和可观测性

交付：

- 关键链路测试。
- 投影失败日志。
- 待投影事件统计。

验收：

- `pytest` 覆盖写入、去重、`.md` 投影、reset、rebuild。
- `/api/memory/stats` 能看到 `md_pending`、`cognee_pending`。

## 11. 测试清单

必须覆盖：

- Agent `memory_add` 创建 `growth_events`，并同步更新 `.md`。
- Agent `memory_update` 更新同一实体时不重复膨胀。
- 对话提取两个不同目标不会被 7 天窗口误去重。
- 同一事件不会被 5 分钟窗口重复投影。
- 简历上传至少产生 `resume_uploaded` 和 `profile_updated`。
- 删除 `.md` 后 rebuild 可恢复。
- reset 后 `growth_events` 为空，`.md` 回到默认模板，Cognee 不再召回旧内容。
- Cognee 不可用时，核心写入和 `.md` 投影仍可用。

## 13. 非目标

本次不解决：

- 多用户隔离目录设计。
- 复杂事件溯源 UI。
- Cognee 多租户按用户删除。
- Markdown 手工编辑后的反向同步。
- 完整认证系统。

这些可以等单用户主链路稳定后再做。

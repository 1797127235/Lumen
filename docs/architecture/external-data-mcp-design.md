## Problem Statement

Lumen Phase 1 的记忆系统只知道「用户告诉它的」——对话中 Agent 通过 memory_save 工具保存的信息。但用户的真实画像散布在笔记、代码仓库、文件系统中。Phase 2 让 Lumen 主动扫描外部数据源，构建比用户自述更完整的认知。

## What Makes This Cool

**Lumen 主动洞察。** 不是用户问「我笔记里提过什么」，而是 Lumen 看到 Obsidian 里多了三篇 Rust 笔记，主动说「你在学 Rust？之前你提过想做系统编程，需要我帮你记住什么吗？」

**三分钟了解你。** 新用户部署 Lumen，配置好本地目录和 GitHub token，三分钟后 Lumen 比用户更清楚自己学过什么、做过什么、关心什么。

**开源社区可部署。** 不是一个需要注册的 SaaS，是别人 clone 下来就能用的开源项目。

## Constraints

- 所有数据本地处理，不依赖云端
- Cognee 保留（Phase 1 已断开 GrowthEvent 连接）
- Phase 1 双管线架构兼容（Profile/Narrative 管线不变，新增 External 管线）
- 一个月内出可用版本（Markdown + GitHub + 文件监听）

## Premises

1. 第一个数据源：本地 Markdown 文件（Obsidian/笔记），不需要网络/认证
2. Cognee 是外部数据的正确索引层（多源异构 → 图数据库 + 向量库匹配）
3. Phase 2 目标：「能搜到外部数据」；主动洞察是 Phase 3 的事
4. TG/QQ、24小时在线、Agent 自主行动 → 10x 版本，不在 Phase 2 范围

## Approaches Considered

### Approach A: 最小可用（FTS5 only，单源 Markdown）
- 改动：~80行，Markdown → FTS5
- 优势：零依赖，最快 demo
- 劣势：无语义搜索，无跨文件关联

### Approach B: 理想架构（多源 + Cognee）
- 改动：~200行，Markdown + GitHub + 文件监听 → Cognee
- 优势：语义搜索、跨源实体链接、完整功能
- 劣势：复杂度高，一个月内可能做不完全

### Approach C: 渐进取径（先 FTS5，后 Cognee）
- Phase 2a (1-2周): Markdown + 文件监听 → FTS5
- Phase 2b (1周): GitHub 接入 → FTS5
- Phase 2c (1-2周): 全量切 Cognee

## Recommended Approach

**方案 C（渐进取径）。** 每步都有可 demo 的东西——前两周就能搜 Markdown 笔记，后两周加 GitHub + Cognee 语义搜索。三阶段复用同一套 Ingestion Pipeline，FTS5 阶段验证管道正确性，切 Cognee 时只换索引层。

## Architecture

**数据接入：MCP Servers（替代自定义 ingestion pipeline）**

```
外部数据源              MCP Server           Agent Tool         索引层
───────────           ───────────           ──────────         ──────
本地 .md 目录 ──→  MCP filesystem   ──→  search_files()  ──→ Cognee
GitHub repos ──→  MCP GitHub       ──→  search_repos()  ──→ Cognee
                                       list_commits()       (Phase 2c)
                                       get_file_contents()

Agent 直接调 MCP tool 读取外部数据 → Cognee Pipeline 建立索引
```

**为什么用 MCP：**
- 零自定义数据接入代码——filesystem/GitHub MCP server 已是标准化实现
- Agent 可以实时搜外部数据（调 MCP tool），也可以后台批量索引
- 开源社区部署只需配置 MCP server，不需要写 Python 管道
- 和 PydanticAI Agent 天然兼容：MCP tool 就是 tool

**三阶段：**

```
Phase 2a (1-2周):
  MCP filesystem server → Agent tool 调 search_files/read_file
  → 提取文本块 → FTS5 索引（新表 external_items）
  → L2 搜索：keyword 模式搜 Narrative + External

Phase 2b (1周):
  MCP GitHub server → Agent tool 调 search_repos/list_commits
  → 同上 FTS5 索引路径

Phase 2c (1-2周):
  同上 MCP 数据源 → Cognee Pipeline → cognee.add() + cognify
  → KùzuDB（图谱，跨源实体链接）+ LanceDB（向量，语义搜索）
  → FTS5 降级为关键词兜底
  → L2 新增 search_mode="semantic"（Cognee）
```

**召回路径（统一语义搜索）：**
```
facade.recall()
  ├── search_mode="keyword"  → FTS5（Narrative + External）
  ├── search_mode="grep"     → time-filtered SQL
  └── search_mode="semantic" → DocumentIndexProvider.prefetch()（可插拔：Cognee/LanceDB/HRR）
```

**主动洞察（Phase 3 路线图）：**
```
MCP filesystem → 定期扫描变更 → 新内容 → Cognee Pipeline
→ 对比已有记忆 → 发现新信息 → Agent 主动通知用户
```

## Next Steps

1. Phase 2a：配置 MCP filesystem server → Agent 注册 `search_files`/`read_file` tool → FTS5 索引层
2. Phase 2b：配置 MCP GitHub server → Agent 注册 `search_repos`/`list_commits` tool → 同上
3. Phase 2c：Cognee Pipeline 层，MCP tool 结果喂入 `cognee.add()` + `cognify`，语义召回
4. Agent 层：`tool_memory_search.py` 加 `search_mode="semantic"` + `scope="external"`
5. 配置层：`.env` 或设置页管理 MCP server 地址 + GitHub token

## Open Questions

- MCP filesystem server 的文件监听能力？是否需要额外 watchdog 做变更检测？
- MCP GitHub server 是否有 rate limit 处理？批量索引时需要本地缓存策略？
- Cognee cognify 对大批量外部数据（上百个 MCP tool 返回的文件）性能如何？
- PydanticAI Agent 注册 MCP tool 的方式？直接 MCP client 还是包装成 `@agent.tool`？

## Success Criteria

- 用户配置一个 Obsidian 目录后，Lumen 能搜索到笔记内容（keyword）
- 用户连接 GitHub 后，Lumen 知道用户的 repos、技术栈
- 用户在 Obsidian 里新建笔记，1 分钟内 Lumen 能搜到（文件监听生效）
- Cognee 阶段：搜「设计模式」能关联到笔记里的理论文章和代码仓库里的实现

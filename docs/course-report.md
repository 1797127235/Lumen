# Lumen：基于 ReAct 推理与文件优先记忆架构的长期 AI 伙伴系统

## 1 项目概述

### 1.1 项目背景

当前主流 AI 助手（ChatGPT、Claude 等）在单轮对话中已展现出很强的语言能力，但在实际使用中存在三个核心痛点：

**记忆断裂。** 每次对话独立开始，用户需要反复描述自己的背景信息。即使模型支持长上下文窗口，也无法跨会话保留信息。用户与 AI 的关系始终停留在"初次见面"的状态。

**推理不可靠。** LLM 的幻觉问题在需要精确信息的场景中尤为突出。单次推理无法保证准确性，需要通过多轮工具调用和事实验证来提升可靠性。

**缺乏个性化。** 通用 AI 助手不理解用户个体，无法根据用户的性格、偏好和沟通风格调整交互方式。

Lumen 正是针对这些问题设计的——一个真正认识你的 AI 伙伴，具备长期记忆、可靠推理和个性化交互能力。

### 1.2 技术栈

| 层 | 选型 |
|---|---|
| 后端 | FastAPI + SQLAlchemy + SQLite (Python) |
| 前端 | React 19 + Vite + Tailwind CSS 4 |
| 桌面 | Tauri v2 (Rust) |
| Agent | 自研 ReAct Loop + MCP 工具协议 |
| 记忆 | MEMORY.md 文件优先 + MemoryProvider 插件 |
| 调度 | APScheduler + asyncio 短轮询 |

## 2 系统总体架构

### 2.1 分层架构

系统分为四层：接入层、消息总线层、业务逻辑层和数据层。

```
┌─────────────────────────────────────────────────────┐
│                   接入层 (Channels)                   │
│   WebChannel (SSE)  │  TelegramChannel  │  CLI TUI  │
└─────────────────────┼───────────────────────────────┘
                      ↓
┌─────────────────────┼───────────────────────────────┐
│              消息总线层 (MessageBus + EventBus)        │
│   InboundMessage → AgentRunner → OutboundMessage     │
└─────────────────────┼───────────────────────────────┘
                      ↓
┌─────────────────────┼───────────────────────────────┐
│                  业务逻辑层                            │
│   Agent推理 │ 记忆系统 │ 工具系统 │ 调度 │ 触发 │ 指标  │
└─────────────────────┼───────────────────────────────┘
                      ↓
┌─────────────────────┼───────────────────────────────┐
│                    数据层                              │
│   SQLite (对话/消息) │ ~/.lumen/memory/ (MEMORY.md)   │
└─────────────────────────────────────────────────────┘
```

接入层负责将不同渠道（Web、Telegram、CLI）的消息统一为 `InboundMessage`，通过消息总线交给业务逻辑层处理。业务逻辑层是核心，包含 Agent 推理、记忆系统、工具系统等模块。数据层负责持久化。

### 2.2 关键设计决策

**文件优先 vs 数据库优先。** 项目早期采用数据库事件投影方案——用户输入存入事件表，通过投影生成 Markdown 文件。这种方案引入了事件表、FTS 索引、投影缓存等组件，维护成本高。重构后直接以 Markdown 文件作为长期记忆的唯一真相源，带来了零运维、用户可编辑、版本友好等优势。

**单用户模型。** 身份固定为 `"me"`，大幅简化了用户管理和权限逻辑，适合个人 AI 伙伴的产品定位。

**SQLite 选择。** 单文件数据库零运维，FTS5 原生支持全文搜索，单用户场景下数据量 < 100MB 完全够用。

## 3 Agent 推理机制

### 3.1 为什么选 ReAct

ReAct（Reasoning + Acting）的核心思想是让 LLM 在推理过程中交替执行"思考"和"行动"。与纯 Chain-of-Thought 不同，ReAct 允许模型在推理中调用外部工具获取真实信息。

选择 ReAct 而非其他方案（如 Reflexion、Plan-and-Execute）的原因：

1. **简单直接**：推理和行动交替进行，不需要额外的规划层或反思层
2. **工具原生**：OpenAI 兼容 API 原生支持 function calling，与 ReAct 天然契合
3. **可控性好**：通过轮次限制和失败检测，可以精确控制推理过程

### 3.2 推理循环设计

Lumen 的 ReAct 循环核心逻辑如下：

```
用户消息 → LLM 推理 → 需要工具？
                        ├─ 是 → 执行工具 → 结果追加到消息 → 继续推理（最多12轮）
                        └─ 否 → 输出最终回答
```

每轮迭代中，系统会重新计算当前可见的工具集合（因为 `tool_search` 可能在运行过程中解锁了新工具），然后将工具 schema 连同消息一起发给 LLM。

### 3.3 三重安全机制

ReAct 循环中最关键的设计不是推理本身，而是如何防止推理失控。Lumen 设计了三重安全机制：

**第一重：工具调用次数上限（20次）。** 防止 Agent 在一个 turn 内无限调用工具。通过上下文中的计数器实现，每次工具执行后递增，达到上限返回错误信息。

**第二重：最大迭代轮次（12轮）。** ReAct 循环本身有轮次上限。达到上限后，系统注入一条消息要求 LLM 基于已有信息直接回答，不再调用工具。

**第三重：连续失败检测（2次）。** 当连续 2 次工具调用失败时（返回错误标记或未知工具），系统注入一条系统消息强制 LLM 停止工具调用。这避免了 LLM 在工具不可用时反复尝试的死循环。

这三重机制层层兜底，确保 Agent 不会陷入无限循环，同时在工具失败时仍能给出有意义的回复。

### 3.4 渐进式裁剪策略

LLM 调用的常见失败是上下文长度超限。随着对话轮次增加，消息序列不断增长，总 token 数超过模型上下文窗口时 API 会报错。

传统方式是简单截断，但可能丢失重要上下文。Lumen 的做法是渐进式裁剪——先尝试完整消息，失败后逐步缩减历史：

| 层级 | 策略 | 适用场景 |
|------|------|----------|
| L1 | 完整消息序列 | 正常情况 |
| L2 | system prompt + 最近 50% 消息 | 中度溢出 |
| L3 | system prompt + 最后一条 user message | 重度溢出 |

关键设计：始终保留第一条消息（system prompt），因为它是 Agent 身份和行为规范的定义；每层裁剪后重新运行完整的 ReAct 循环，而非在同一循环内重试。

### 3.5 System Prompt 设计

System prompt 采用两层结构，这是整个推理机制中最重要的设计之一：

**稳定前缀层。** 完全静态的内容：身份声明、行为规范、输出格式要求、记忆使用规则等。这些内容不随用户或对话变化，byte-identical 的前缀可以最大化 LLM API 的 prefix cache 命中率，降低延迟和成本。

**上下文后缀层。** 半稳定的用户相关内容：用户称呼、L0 用户画像（USER.md / MEMORY.md）、PARTNER.md 协作规则、PERSONA.md 人格设定。按 (user_id, conversation_id) 缓存。

为什么分两层？因为 LLM API 的 prefix cache 机制——相同前缀的请求可以跳过预填充阶段。如果 system prompt 每轮都变，cache 完全失效。分层后，稳定前缀跨轮次 byte-identical，cache 命中率可达 80% 以上。

### 3.6 Conversation 级冻结

一个关键的设计决策：当用户首次进入 conversation 时，系统构建完整的 system prompt 并持久化。此后该 conversation 内所有轮次复用同一份 system prompt，即使期间有新的记忆写入。

为什么这样设计？

1. **一致性保证**：同一对话内使用相同的理解基础，避免中途记忆更新导致行为突变
2. **Cache 保护**：记忆写入后只清除其他 conversation 的缓存，当前 conversation 继续使用冻结版本，保护 prefix cache
3. **语义合理**：用户在对话中途写入的记忆，不应该立刻改变 AI 在当前对话中的行为——那是"下一段对话"才会生效的新理解

### 3.7 人格系统

Lumen 的人格（性格、语气、说话方式）通过 PERSONA.md 文件定义，是用户可编辑的 Markdown 文档。

设计原则：人格来源唯一——要么是用户编写的 PERSONA.md，要么是默认人格常量，两者二选一，绝不叠加。默认人格的核心特征是"先接住再展开"、"有知识但不无所不能"、"会轻轻吐槽"、"情绪要看得见"，营造伙伴感而非客服播报器的交互体验。

## 4 记忆系统

### 4.1 设计目标

记忆系统需要回答三个问题：

1. **存什么**：用户的偏好、经历、关系、习惯等稳定信息
2. **怎么存**：持久化格式、过期策略、安全防护
3. **怎么用**：如何在对话中自然地利用记忆，而不让用户感觉"记忆"这个东西的存在

### 4.2 三层上下文模型

Lumen 的记忆分为三层，分别解决不同时间尺度的需求：

```
┌─────────────────────────────────────────────────┐
│  L0 — 冻结快照（注入 system prompt）              │
│  USER.md + MEMORY.md 的内容                      │
│  进入对话取一次，对话内多轮复用                    │
├─────────────────────────────────────────────────┤
│  L1 — 近期对话上下文                              │
│  最近几轮消息，本就在消息历史中                    │
├─────────────────────────────────────────────────┤
│  L2 — 动态召回（按需注入消息序列）                 │
│  外部 Provider 的语义检索结果                      │
│  每轮动态生成，用 <memory-context> 围栏包裹        │
└─────────────────────────────────────────────────┘
```

**L0 冻结快照**是记忆的核心。它在对话开始时从 MEMORY.md 和 USER.md 读取一次，注入 system prompt 的上下文后缀层。MEMORY.md 是长期记忆的唯一真相源，USER.md 是由 MEMORY.md 通过 LLM 自动生成的 evidence-based 用户画像（每条带来源追溯）。

**L2 动态召回**通过外部 Provider 插件实现。MemoryManager 向所有已注册的外部 Provider 并发发起 prefetch 请求，结果按 provider 标注后拼接，用 `<memory-context>` XML 围栏包裹注入消息序列。围栏标签附带系统说明，告诉 LLM 这是"回忆起来的上下文，不是用户输入"。

为什么 L2 要用围栏？因为如果不加标识，LLM 可能把记忆内容误认为用户刚说的话，导致混淆。围栏标签提供了明确的语义边界。

### 4.3 文件存储设计

#### 4.3.1 条目格式

MEMORY.md 中的每条记忆采用统一格式，带时间戳和分类标签：

```
- 2026-06-22 14:30 — [fact] 用户的名字是张三
- 2026-06-21 — [preference] 喜欢简洁的回复风格
- 2026-06-20 — [intent] 打算学习 Rust 编程
- 2026-06-19 — [transient] 最近在赶项目，比较忙
```

五种分类标签对应不同的过期策略：

| 标签 | 含义 | 过期策略 |
|------|------|----------|
| `fact` | 稳定事实 | 永不过期 |
| `preference` | 长期偏好 | 永不过期 |
| `intent` | 意图/计划 | 30 天后标记待确认，再 30 天删除 |
| `transient` | 临时状态 | 7 天后自动删除 |
| `correction` | 纠正旧记忆 | 永不过期 |

这种设计的好处是：用户不需要手动管理记忆的生命周期，系统自动清理过期信息。`fact` 和 `preference` 永远保留，`transient` 自然消亡，`intent` 超时后提醒确认。

#### 4.3.2 安全防护

记忆写入前必须经过安全扫描，检测三类风险：

- **Prompt Injection**：防止恶意指令通过记忆注入 system prompt（如 "ignore all previous instructions"）
- **数据外泄**：防止 API key 等敏感信息被写入记忆（如 "sk-xxx" 模式）
- **隐形 Unicode**：防止零宽字符等不可见字符注入

扫描不通过则拒绝写入，返回错误信息。

#### 4.3.3 原子写入与并发锁

文件写入采用临时文件 + rename 的原子操作，避免写入过程中断导致文件损坏。并发控制使用双重锁：asyncio.Lock 保证进程内串行化，文件锁（msvcrt/fcntl）保证跨进程安全。

### 4.4 记忆写入流程

Agent 通过 `memory` 工具写入记忆。整个流程分为五步：

1. **安全扫描**：检测 prompt injection、数据外泄、隐形 Unicode
2. **写入 MEMORY.md**：追加带时间戳和分类标签的条目，超限时丢弃最旧条目
3. **后台刷新 USER.md**：调用 LLM 从 MEMORY.md 生成 evidence-based 画像（5 分钟防抖）
4. **镜像到外部 Provider**：将写入事件转发给所有已注册的外部 Provider
5. **缓存失效**：使其他 conversation 的 system prompt 缓存失效，当前 conversation 保持冻结

第 3 步的防抖机制很重要——用户可能连续写入多条记忆，如果每次都触发 USER.md 刷新会产生大量 LLM 调用。5 分钟防抖确保在活跃写入期间只在最后一次写入后触发一次刷新。

### 4.5 AI 画像生成

USER.md 的生成过程是：读取 MEMORY.md 全文（过滤 transient 条目）+ 现有 USER.md，调用 LLM 生成 evidence-based 的用户画像。

生成规则很严格：
- 只用 bullet 列表，不写散文
- 每条必须带 `[来源: 日期 category]` 追溯
- 禁止推测，只陈述 MEMORY.md 中明确出现的事实
- 宁缺毋滥，不确定的不写

画像末尾还附带一个 JSON 数组，包含 0-3 条跨维度的模式洞察（如"技术导向型用户，偏好直接解决方案"）。

### 4.6 记忆定期整理

`MemoryHousekeeper` 是后台定时任务，每 24 小时扫描所有用户的记忆文件，执行过期清理。整理逻辑是纯文本操作，不依赖 LLM——通过正则解析条目格式，按分类标签和时间戳判断是否过期。

### 4.7 MemoryProvider 插件系统

记忆系统采用插件架构，MemoryManager 作为编排器，内置 BuiltinMemoryProvider 始终存在，同时最多挂载一个外部 Provider（如 Akasha 或 Honcho）：

```
MemoryManager（编排器）
    │
    ├─ BuiltinMemoryProvider（内置，始终存在）
    │   └─ 直接读写 MEMORY.md / USER.md
    │
    ├─ HonchoProvider（外部，语义召回）
    │   └─ 通过 Honcho AI 服务进行语义检索
    │
    ├─ AkashaProvider（外部，图记忆）
    │   └─ 基于知识图谱的向量检索 + 涟漪激活
    │
    └─ 自定义 Provider（用户插件）
        └─ 放在 ~/.lumen/plugins/memory/ 下自动发现
```

插件需要实现的核心接口：

| 方法 | 说明 |
|------|------|
| `is_available()` | 可用性检查 |
| `system_prompt_block()` | 返回 L0 快照内容 |
| `prefetch()` | L2 动态语义召回 |
| `sync_turn()` | 轮次同步 |
| `on_memory_write()` | 记忆写入事件镜像 |

外部 Provider 故障时，MemoryManager 静默降级到内置文件记忆，不打断聊天。启动时不可用的 Provider 加入 pending 队列，后台按指数退避（2 分钟→30 分钟封顶）定期重试。

### 4.8 Akasha 图记忆引擎

Akasha 是项目自研的内置图记忆插件，负责基于向量的语义召回。它的核心思想来自神经科学——用图结构模拟人脑的联想记忆。

#### 4.8.1 设计动机

MEMORY.md 的关键词匹配只能处理"用户明确说过什么"，但无法处理"哪些记忆之间有关联"。比如用户提到了"咖啡"，系统应该能联想到上周聊过的"下午三点的习惯"和上个月的"买了新磨豆机"——这些记忆之间没有关键词重叠，但在语义和时间上有关联。

Akasha 的做法是：把每轮对话都变成图上的节点，节点之间通过共激活建立边，形成一个自组织的记忆网络。

#### 4.8.2 三层召回机制

Akasha 的检索不是简单的向量搜索，而是三层递进的召回：

**第一层：Dense 向量检索。** 将查询文本 embedding 后，与所有历史 turn 的 embedding 做余弦相似度，找到语义最接近的种子节点。同时用 FTS BM25 做文本检索，两路结果取并集，Dense ∩ FTS 的节点获得额外 boost。

**第二层：RWR 涟漪激活。** 从种子节点出发，沿图的共激活边做 Random Walk with Restart（RWR）扩散。扩散过程中，节点的能量受三个因素调节：salience（与全局重心的偏离度，越不寻常的记忆越重要）、strength（长期强度，随时间衰减）、resource（短期资源，刚被回忆过的节点暂时抑制，鼓励多样性）。扩散 2 轮后，能量较高的节点被激活。

**第三层：Graph-expand 图扩展。** 对 Dense 种子节点的一跳邻居做补充，找到那些语义不直接匹配、但通过共激活边与种子节点紧密关联的历史记忆。

#### 4.8.3 自组织图生长

图的边不是预先定义的，而是在使用过程中自动生长的。设·1计灵感来自神经科学中的三个机制：

**Hebbian 学习。** "一起激活的节点连接在一起"——当一次查询同时激活了节点 A 和 B，A 和 B 之间就会建立或增强共激活边。下次查 A 时，B 更容易被涟漪激活。

**STDP（脉冲时序依赖可塑性）。** 因果方向的边获得更大增益（A 比 B 先激活 → A→B 的边更强），反因果方向增益较小。

**时间衰减。** 节点的 strength 按 7 天半衰期指数衰减，边按 14 天半衰期衰减。这意味着旧记忆逐渐淡化，除非被再次回忆（回忆会刷新 strength）。

#### 4.8.4 轮次同步

每次 assistant turn 结束后，MemoryManager 调用 Akasha 的 `sync_turn()` 方法，将本轮对话的 user 和 assistant 消息分别 embedding，写入图中作为新节点。写入前先对现有图做一次激活（用当前 user message 作为查询），生成共激活边，让新节点自然地接入已有记忆网络。

整个过程对用户完全透明——不需要手动触发，每轮对话自动同步。

#### 4.8.5 存储

Akasha 使用独立的 SQLite sidecar DB（`~/.lumen/memory/{user_id}/akasha.db`），存储节点、边、turn 原文、FTS 索引和查询日志。节点 embedding 以 float32 BLOB 形式存储在节点表中。FTS 使用 trigram tokenizer 支持中文全文搜索。

## 5 工具系统

### 5.1 核心问题

工具系统需要解决的核心问题是：LLM 的 context window 有限，不可能每轮都传入所有工具的 schema。但用户可能需要的工具很多（文件操作、搜索、Shell、MCP 外部工具等），如何在有限的 context 空间内提供最大的工具覆盖？

### 5.2 三层可见性设计

Lumen 的解决方案是三层可见性：

| 层级 | 说明 | 示例 |
|------|------|------|
| Always-on | 始终可见，每轮都传入 schema | `memory`, `memory_search`, `tool_search` |
| Preloaded | 通过 tool_search 解锁后可见 | `file_read`, `shell`, `web_search` |
| Deferred | 未加载，仅在 system prompt 中列出目录 | MCP 工具、低频工具 |

`tool_search` 是关键——它是一个元工具，让 Agent 按需搜索并加载工具。当 Agent 需要某个能力时，先调用 `tool_search(query="文件读取")` 搜索匹配的工具，系统将匹配的工具加入当前 conversation 的预加载缓存，下一轮 ReAct 迭代中这些工具的 schema 就会包含在 LLM 请求中。

这种设计的效果：always-on 工具只占少数核心能力（3-5 个），大部分工具按需加载。首次调用需要多一轮 tool_search，但后续轮次直接可用。

### 5.3 动态发现缓存

每个 conversation 维护独立的工具可见性缓存（LRU，容量 10 个工具）。缓存是进程内存态，重启后清空——这是有意为之的，因为 always-on 工具覆盖了核心能力，重新解锁只需一轮 tool_search。

缓存还支持 Skill scope 限制：激活特定 Skill 后，该 conversation 只能使用 Skill 声明的工具 + 核心工具，避免工具污染。

### 5.4 MCP 工具桥接

Lumen 通过 MCP（Model Context Protocol）协议接入外部工具服务器。配置在 `~/.lumen/config.json` 中，启动时自动连接。MCP server 暴露的工具会自动注册到工具注册表，与内置工具共享同一执行流程，对 Agent 完全透明。

一个典型的使用场景：用户启动一个 RSS 订阅服务（lumen-feed），配置为 MCP server，Agent 就能直接调用该服务的工具来管理订阅和获取更新。

### 5.5 工具执行与可观测性

工具执行通过统一的注册表入口，自动合并会话上下文并记录执行指标（耗时、状态）。每次工具调用都会异步写入 `metric_events` 表，支持后续的监控和分析。

## 6 主动行动能力

### 6.1 设计理念

Lumen 默认是被动应答——用户说话才有反应。主动行动能力让 AI 能自己到点做事或有事件时主动找用户，是"伙伴感"的核心来源。

统一概念模型：**触发源 × 动作**。

```
        触发源              动作
  ┌──────────────┐    ┌──────────────────────────┐
  │ 时间(scheduler)│    │ silent → 跑工具,只写日志   │
  │ 事件(triggers) │    │ notify → 注入对话,主动触达 │
  └──────────────┘    └──────────────────────────┘
```

### 6.2 时间触发

基于 APScheduler，支持 cron/interval/date 三种调度方式。典型场景：每天早上 8 点查天气并推送、每周五下午提醒周报、一次性定时提醒。

任务持久化到 JSON 文件，重启后自动恢复。每次执行结果追加记录到对应的 JSONL 文件，支持历史回溯。

### 6.3 事件触发

监听 MCP server 通知，匹配订阅规则后注入对话。典型场景：RSS 订阅有新文章时通知、外部服务状态变化时提醒。

通过 asyncio 短轮询实现，订阅持久化到 JSON 文件。

### 6.4 渠道触达差异

当前只有 Telegram 渠道能真正主动触达用户（Bot API 主动推消息，无需用户在线）。Web 渠道的 SSE 连接是瞬态的——用户没正开着页面，主动消息会被静默丢弃（但回复仍持久化进历史，事后打开可见）。真正的 Web 推送需要持久化收件箱/通知表，是独立的后续工程。

## 7 可观测性体系

### 7.1 设计思路

所有指标写入同一张 `metric_events` 统一时序表，避免为每类指标建独立表。新增指标只需加一个 `record()` 调用，无需改 schema。

### 7.2 埋点覆盖

系统在 7 个关键路径设置了埋点，覆盖 LLM 调用、Agent 推理循环、工具执行、HTTP 请求、记忆写入、主动推送等。所有埋点采用 fire-and-forget 模式，绝不阻塞业务流程。

### 7.3 观测站

独立的单文件 HTML 页面（Chart.js + Tailwind CDN，零构建），挂在 `/observatory` 路由。每 30 秒自动刷新，展示 Token 用量、延迟时序、工具调用排行、KPI 摘要等图表。与主前端完全解耦，即使主前端不维护也能用。

## 9 总结

### 9.1 核心工作

本项目围绕"如何让 AI 真正认识用户"这一问题，在三个维度进行了设计和实现：

**推理机制**：基于 ReAct 的渐进式裁剪推理引擎，三重安全机制（调用上限 + 轮次上限 + 连续失败检测）保障可靠性，conversation 级 system prompt 冻结在记忆更新与 prefix cache 之间取得平衡。

**记忆架构**：文件优先的 Hermes-Pure 架构，Markdown 文件作为唯一真相源。三层上下文模型实现跨会话连续性，五种分类标签配合自动过期清理保证时效性，可插拔 Provider 插件支持外部语义召回。自研的 Akasha 图记忆引擎实现了基于 Hebbian 学习和 RWR 扩散的联想记忆，让系统能发现记忆之间的隐含关联。

**工具系统**：注册表 + 动态发现的三层可见性设计，tool_search 元工具实现按需加载，MCP 协议桥接实现外部工具零成本接入。

### 9.2 不足与改进方向

1. **Web 推送不完整**：主动行动仅 Telegram 真正触达
2. **缺乏智能摘要**：固定消息窗口控制上下文，无分层摘要策略

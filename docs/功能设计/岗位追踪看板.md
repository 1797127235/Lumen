# 岗位追踪看板 · V1 实施方案

> 定位：从"分析工具"升级为"持续陪跑工具"。用户粘贴 JD，系统自动诊断 + 建卡，看板管理投递进度，每张卡片有 LLM 生成的行动建议。
>
> 关联设计：`docs/功能设计/模拟面试与求职管理.md` 中"核心功能⑥：求职进度管理"是完整版愿景，本文档是 V1 落地范围。

---

## 一、核心流程

```
粘贴 JD → 诊断打分 → 建卡片 → 看板展示 → 拖拽改状态 → LLM 给下一步建议
                                    ↑
                              可选手动建卡（不诊断）
```

**V1 不做的事**：
- 不做面试反馈记录（V2）
- 不做数据复盘周报（V2）
- 不做主动提醒/倒计时（V2）
- 不做从看板触发模拟面试/学习路径（V2）
---

## 二、数据模型

新增 `job_targets` 表，复用已有 `jd_diagnoses` 表：

```python
# app/backend/models/job_target.py

class JobTarget(Base):
    __tablename__ = "job_targets"

    target_id: Mapped[str] = mapped_column(
        String(36), primary_key=True, default=lambda: str(uuid.uuid4())
    )
    user_id: Mapped[str] = mapped_column(String(36), ForeignKey("users.user_id"), index=True)
    company: Mapped[str] = mapped_column(String(100))           # 公司名
    title: Mapped[str] = mapped_column(String(100))             # 岗位名
    location: Mapped[str | None] = mapped_column(String(50))    # 城市
    salary: Mapped[str | None] = mapped_column(String(50))      # 薪资
    jd_text: Mapped[str | None] = mapped_column(Text)           # JD 原文
    jd_url: Mapped[str | None] = mapped_column(String(500))     # JD 链接
    status: Mapped[str] = mapped_column(String(20), default="interested")
    interview_round: Mapped[str | None] = mapped_column(String(50))  # 面试轮次，用户自填
    diagnosis_id: Mapped[str | None] = mapped_column(
        String(36), ForeignKey("jd_diagnoses.diagnosis_id", ondelete="SET NULL")
    )
    match_score: Mapped[int | None] = mapped_column(Integer)    # 冗存，创建时从诊断结果写入
    agent_advice: Mapped[str | None] = mapped_column(Text)      # LLM 行动建议
    notes: Mapped[str | None] = mapped_column(Text)             # 用户备注
    sort_order: Mapped[int] = mapped_column(Integer, default=0) # 同列排序
    status_changed_at: Mapped[datetime | None] = mapped_column(
        DateTime(timezone=True)
    )  # 仅 status 变更时更新。创建时写入 = created_at（为 V2 留干净数据）
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), onupdate=func.now()
    )
```

**状态枚举**（字符串，不用 Enum，方便扩展）：

| 值 | 含义 | 说明 |
|----|------|------|
| `interested` | 感兴趣 | |
| `applied` | 已投递 | |
| `test` | 笔试 | |
| `interview` | 面试 | 具体轮次由 `interview_round` 字段区分 |
| `offer` | Offer | |
| `rejected` | 被拒 | |
| `abandoned` | 放弃 | |

**面试轮次**（`interview_round`，仅 `status="interview"` 时有值，用户自填）：

自由文本，常见值如：一面、二面、三面、HR面、终面、技术面、交叉面等。不做枚举限制，用户按实际情况填写。

**设计决策**：
- `diagnosis_id` 是外键，`ondelete="SET NULL"`——诊断记录被清理时岗位卡保留，只是不显示匹配分。
- `match_score` 冗存——创建时从诊断结果写入，避免每次 GET 做 N 次 JOIN。V1 不支持重新诊断，所以不会 drift。
- `agent_advice` 生成时机：① 创建卡片时 ② 状态变更时。用 FastAPI `BackgroundTasks`，不阻塞响应。**task 内部用 `get_async_session_maker()` 自建 session**，因为 endpoint 的 `get_db` 在返回后已关闭。
- `sort_order` V1 只做追加（拖拽到目标列末尾），不支持精确位置插入。已知 V1 限制：并发拖拽可能出现同列 sort_order 重复，单用户场景可接受。
- `status` 数据库用字符串（扩展性），但 Pydantic 层用 `Literal` 校验合法值。
- `status_changed_at` 仅在 status 变更时更新，`updated_at` 会被 notes/interview_round 等字段污染。创建时写入 `= created_at`，为 V2 留干净数据。
- `interview_round` 与 `status` 分离——粗状态控制看板列，细轮次在详情页手动填写。字段长度 String(50)。
- `company`、`title` 可修改（不触发重新诊断）。`jd_text`、`diagnosis_id` 创建后锁定。

### 3.2 `POST /api/targets`

**请求体**：
```json
{
  "company": "字节跳动",
  "title": "后端开发工程师",
  "location": "北京",
  "salary": "30-50K",
  "jd_text": "...（可选，有则触发诊断）",
  "jd_url": "https://...",
  "notes": "提前批"
}
```

**流程**：
```
1. 创建 JobTarget 记录（status="interested", sort_order=该列 max+1, status_changed_at=now()）
2. 如果有 jd_text：
   a. 同步调用 jd_service.diagnose_jd() 生成诊断（10-30s，用户等待）
   b. 关联 diagnosis_id，match_score 从诊断结果写入
3. 返回完整卡片数据（此时 diagnosis 和 match_score 已就绪）
4. [BackgroundTasks] 异步调用 LLM 生成 agent_advice，写回 DB
   ⚠️ task 内部用 get_async_session_maker() 自建 session，不能用 endpoint 传入的 db

无 jd_text 时跳过步骤 2 和 4，直接返回卡片（match_score=null, agent_advice=null）。
```

**设计选择**：诊断同步（用户等一次），建议异步（不阻塞响应）。诊断结果是卡片核心数据，必须就绪后才返回；建议是锦上添花，可以后补。

**响应**：
```json
{
  "target_id": "uuid",
  "company": "字节跳动",
  "title": "后端开发工程师",
  "status": "interested",
  "match_score": 78,
  "agent_advice": "匹配度较高，建议今天投递。简历中可以突出网络库项目的性能指标。",
  "created_at": "2025-05-03T10:00:00Z"
}
```

**`stats.common_gaps` 逻辑**：
- 遍历所有关联了诊断的 target
- 聚合 `skill_gaps`，按出现频次排序
- 返回 TOP 5 高频缺口

**`stats.avg_score` 边界**：如果没有关联诊断的 target，`avg_score` 返回 0，不除零。

### 3.4 `PATCH /api/targets/{id}`

**请求体**：
```json
{
  "company": "字节跳动",        // 可改（修正 typo）
  "title": "后端开发工程师",     // 可改
  "status": "interview",        // 状态变更（拖拽）
  "interview_round": "二面",     // 面试轮次（详情页手动填）
  "notes": "已内推"              // 备注
}
```

**流程**：
```
1. 更新 JobTarget 字段
2. 如果 status 变更：
   a. 重新计算 sort_order = 目标列 max(sort_order) + 1
   b. 更新 status_changed_at = now()
   c. [BackgroundTasks] 异步重新生成 agent_advice
      ⚠️ task 内部用 get_async_session_maker() 自建 session
3. 返回更新后的卡片数据
```

**可修改的字段**：`company`、`title`、`status`、`interview_round`、`notes`、`location`、`salary`。修改 `company`/`title` 不触发重新诊断。

**锁定的字段**：`jd_text`、`diagnosis_id`、`match_score`（创建时确定）

## 四、Schema 定义

```python
# app/backend/schemas/target.py

from typing import Literal
from pydantic import BaseModel, Field

VALID_STATUSES = Literal[
    "interested", "applied", "test", "interview", "offer", "rejected", "abandoned"
]

class TargetCreate(BaseModel):
    company: str = Field(..., min_length=1, max_length=100)
    title: str = Field(..., min_length=1, max_length=100)
    location: str | None = None
    salary: str | None = None
    jd_text: str | None = None
    jd_url: str | None = None
    notes: str | None = None

class TargetUpdate(BaseModel):
    company: str | None = None          # 可改（修正 typo）
    title: str | None = None            # 可改
    status: VALID_STATUSES | None = None
    interview_round: str | None = None
    location: str | None = None
    salary: str | None = None
    notes: str | None = None

class TargetCard(BaseModel):
    target_id: str
    company: str
    title: str
    status: str
    interview_round: str | None = None
    match_score: int | None = None  # 冗存，从诊断结果写入
    agent_advice: str | None = None
    location: str | None = None
    created_at: str

class TargetDetail(TargetCard):
    jd_text: str | None = None
    jd_url: str | None = None
    salary: str | None = None
    notes: str | None = None
    diagnosis: dict | None = None  # JDDiagnoseResponse 的 dict 形式

class BoardStats(BaseModel):
    total: int
    avg_score: float
    common_gaps: list[str]

class BoardResponse(BaseModel):
    columns: dict[str, list[TargetCard]]
    stats: BoardStats
```

**LLM 建议生成**（`_generate_advice`）：
```
触发时机：① 创建卡片时 ② 状态变更时
输入：用户画像 + JD 诊断结果 + 当前状态 + 距创建天数
输出：一句话行动建议（≤50字）
模型：qwen-plus，temperature=0.3
⚠️ task 内部用 get_async_session_maker() 自建 session，不能依赖 endpoint 的 db
```

**Prompt 模板**（实现时直接复制）：
```python
_ADVICE_PROMPT = """你是求职陪跑教练。基于以下信息，给一句话行动建议（≤50字）。

【用户画像】{profile_summary}
【岗位】{company} - {title}
【当前状态】{status}（{interview_round or "无轮次"}）
【距投递】{days_since_create}天
【诊断要点】匹配{match_score}/100，主要缺口：{top_gaps}

只输出建议本身，不要解释。"""
```

**agent_advice 兜底**：LLM 调用失败时 `agent_advice` 保持 null。前端 null 时显示"生成中，刷新试试"。

**共性缺口聚合**（`_aggregate_gaps`）：
```python
def _aggregate_gaps(diagnoses: list[JDDiagnosis]) -> list[str]:
    """统计所有诊断的 skill_gaps，按出现频次返回 TOP 5。"""
    from collections import Counter
    counter = Counter()
    for d in diagnoses:
        for gap in d.get_skill_gaps():
            # 归一化：strip + lower，避免 "K8s" 和 "kubernetes" 算两个
            counter[gap["skill"].strip().lower()] += 1
    return [skill for skill, _ in counter.most_common(5)]
```

**JDDiagnosis JSON 访问 helper**（在已有 `models/jd_diagnosis.py` 中补充）：
```python
# app/backend/models/jd_diagnosis.py — 在 JDDiagnosis 类中加

def get_skill_gaps(self) -> list[dict]:
    """类型安全地读取 skill_gaps，避免业务代码直接掏 dict。"""
    return self.result_data.get("skill_gaps", []) if self.result_data else []
```

## 九、智能体体现

| 场景 | 做法 | 成本 |
|------|------|------|
| 新增岗位时有 JD | 自动诊断 + LLM 生成 `agent_advice` | 1 次 LLM（诊断）+ 1 次 LLM（建议） |
| 状态变更时 | 重新生成 `agent_advice` | 1 次 LLM 调用 |
| 看板顶部 | 聚合 `common_gaps`（纯 Python，零成本） | 无 |
| 每张卡片 | 显示 `agent_advice` | 已预生成，读取即可 |
| 详情页 | 展示完整诊断报告 | 已预生成，读取即可 |

V1 的智能体是"轻量"的：诊断 + 建议 + 聚合分析。不依赖复杂 Agent 编排，不依赖 `orchestrator.py` 的 Skill 系统。

## 十、已知 V1 限制

| 限制 | 说明 | V2 计划 |
|------|------|---------|
| `sort_order` 并发 race | 同列拖拽可能出现 sort_order 重复，单用户可接受 | 精确位置插入 |
| JD 重复创建无去重 | 同一 JD 粘贴两次会创建 2 张卡 | jd_text hash 去重 |
| `user_id` 来源 | 跟随 `jd.py` 模式：`Query("demo_user")`，无认证 | JWT 认证 |
| `agent_advice` 失败 | LLM 调用失败时显示"生成中，刷新试试" | 重试 + 手动重新生成按钮 |
| `common_gaps` 同义词 | "K8s" 和 "Kubernetes" 算两个，未归一 | skill 名称归一化 |
| `interview_round` 自由文本 | 用户可能写出"1面""一面""第一面"，V2 聚合时需处理 | 前端下拉 + 自填"其他" |

# 模拟面试与求职管理

> 源自：功能模块与交互设计_AI职业规划智能体.md

### 核心功能⑤：技术八股/算法模拟面试

#### 功能定位与价值主张

**定位**: 将"被动背诵"转化为"主动输出+即时反馈"，提供沉浸式面试训练环境，覆盖技术八股和算法两大核心模块。

**价值主张**: "不是让你背答案，而是让你体验被面试官追问到答不上来的感觉——因为真正的面试就是这样的" — 模拟真实追问，训练临场思维。

#### 用户触发条件

| 触发场景 | 用户操作 |
|---------|---------|
| 面试前突击 | "我明天有美团面试，帮我模拟一下Java后端" |
| 系统性训练 | "我想每周练3次八股，每次30分钟" |
| 短板突破 | "Redis是我的薄弱项，专门练Redis相关题目" |
| 算法保持手感 | "帮我出2道中等难度的链表题" |

#### 输入与输出定义

| 维度 | 定义 |
|------|------|
| **用户输入** | 选择面试模式（八股/算法/综合）、目标方向/岗位、题目数量/时长、难度偏好 |
| **系统输出** | 面试题目 -> 用户回答 -> AI追问/评判 -> 评分反馈 -> 改进建议 -> 知识延伸 |
| **输出格式** | 分屏界面：上方题目卡片，下方输入区，右侧可选"思路提示""相关知识" |

#### 详细交互流程

##### 子功能A：技术八股模拟面试

**步骤1 - 面试配置**
- 系统操作：提供快速配置选项
  ```
  "来场模拟面试？"
  方向：[Java后端] [前端] [算法] [测试] [运维] [自定义]
  模块：[操作系统] [计算机网络] [数据库] [Redis] [消息队列] [Spring] [全部]
  难度：[基础] [中等] [困难] [混合]
  题数：[5道快练] [10道标准] [20道深度]
  模式：[练习模式-即时看答案] [考试模式-答完再看]
  ```

**步骤2 - 逐题面试**
- 系统输出题目：
  ```
  【第1/10题】Redis
  面试官："说说Redis为什么这么快？"
  
  [计时器 00:00] [语音输入] [思路提示]
  
  你的回答：__________
  [提交回答]
  ```

**步骤3 - 回答评判与追问**
- 用户提交回答后，AI评判：
  ```
  【评判】你的回答涵盖了：内存存储、单线程模型
  遗漏关键点：IO多路复用、高效数据结构（SDS/跳表）、持久化策略的取舍
  
  【追问】面试官继续问：
  "你说Redis是单线程，那它如何处理多个客户端并发请求？"
  
  （追问逻辑：基于你回答的漏洞或深度触发追问，最多追问2-3层）
  ```
- 用户可继续回答追问，或选择"这题我认了，看标准答案"

**步骤4 - 单题复盘**
- 系统输出：
  ```
  【参考答案】
  Redis高性能的核心原因：
  1. 内存操作：数据存储在内存，读写速度比磁盘快数个数量级
  2. 单线程模型：避免多线程上下文切换和锁竞争（注：Redis 6.0+引入多线程IO，但命令执行仍为单线程）
  3. IO多路复用：基于epoll/kqueue实现单线程处理万级并发连接
  4. 高效数据结构：SDS动态字符串、跳表、压缩列表等专为性能设计
  5. 持久化策略取舍：RDB快照和AOF日志的灵活配置，在性能和 durability 间权衡
  
  【你的表现】⭐⭐⭐☆☆（3/5）
  优点：抓住了核心点（内存+单线程）
  不足：缺少底层机制（IO多路复用）和具体数据结构
  建议：面试前重点复习Redis底层网络模型和数据结构实现
  ```

**步骤5 - 面试结束与总评**
- 全部题目完成后：
  ```
  【模拟面试报告】
  总评分：68/100（中等水平）
  
  各维度表现：
  Redis ████████░░ 4.2/5
  MySQL █████░░░░░ 3.1/5
  操作系统 ███████░░░ 3.8/5
  
  薄弱知识点TOP5：
  1. MySQL索引优化（连续2题答偏）
  2. Redis持久化机制（未区分RDB和AOF适用场景）
  3. ...
  
  【下一步】[针对薄弱点生成复习计划] [再来一场] [保存报告]
  ```

##### 子功能B：算法模拟面试

**步骤1 - 题目生成**
- 系统操作：根据用户水平和目标生成题目（从LeetCode类似题库）
- 题目展示：题目描述 + 示例输入输出 + 约束条件

**步骤2 - 编码环境**
- 系统输出：在线代码编辑器（支持Java/Python/C++）+ 运行按钮
- 用户操作：编写代码，可运行测试用例

**步骤3 - 自动判题**
- 系统操作：运行用户代码 against 隐藏测试用例
- 结果反馈：AC（通过）/ WA（答案错误）/ TLE（超时）/ RE（运行错误）
- 边界测试：额外展示用户未考虑的边界用例

**步骤4 - 思路讲解评估**
- 系统操作：要求用户讲解思路（文本/语音）
- AI评判：评估时间复杂度分析、空间复杂度分析、优化思路的完整性

**步骤5 - 标准题解**
- 系统输出：最优解代码 + 复杂度分析 + 相似题目推荐

#### 边界情况与异常处理

| 边界情况 | 处理策略 |
|---------|---------|
| 用户回答过于简短 | "能展开说说吗？面试官希望听到你的思考过程。" |
| 用户完全不会 | "这道题确实有难度，要直接看答案吗？或者我给你个提示？" |
| 算法题超时 | "运行超时了，想想时间复杂度是否可以优化？当前是O(n^2)，目标可以做到O(n log n)" |
| 用户连续3题表现差 | "今天的难度可能不太适合你，要不要降低难度，或者先补补基础？" |
| 用户输入代码有语法错误 | 实时标红提示语法错误位置 |

#### AI Agent角色与工作流

**Agent角色**: "面试官Agent"
- 人设：大厂技术面试官，严肃但不刻薄，善于引导，追问有深度
- 核心能力：题库管理 + 追问逻辑链 + 回答评估 + 思路引导

**工作流（技术八股）**:
```
面试配置 -> 题库检索（按方向+模块+难度） -> 
出题 -> 用户回答 -> 
回答分析（关键点提取 vs 标准答案对比） -> 
追问决策（遗漏点/深度不够 -> 生成追问） -> 
追问最多3层 -> 
评分计算（覆盖度40% + 深度30% + 表达清晰度20% + 应变能力10%） -> 
生成面试报告
```

**工作流（算法）**:
```
题目生成 -> 在线编码 -> 自动判题 -> 
思路讲解收集 -> 思路评估 -> 
标准题解展示 -> 相似题推荐
```

### 核心功能⑥：求职进度管理

#### 功能定位与价值主张

**定位**: 将"Excel手工管理"升级为"可视化作战室"，全链路追踪投递-笔试-面试-Offer全流程，基于数据驱动策略调整。

**价值主张**: "不只是记你投了哪些公司，而是告诉你'投了50家只收到2个面试，问题出在简历过筛率上'" — 从记录工具升级为策略分析师。

#### 用户触发条件

| 触发场景 | 用户操作 |
|---------|---------|
| 开始求职 | 秋招季创建求职看板 |
| 投递后记录 | "我刚投了字节跳动" |
| 状态更新 | "字节约了三面，下周二" |
| 定期复盘 | "这周求职情况怎么样？" |
| 面试提醒 | 系统主动提醒"明天上午10点有美团面试" |

#### 输入与输出定义

| 维度 | 定义 |
|------|------|
| **用户输入** | 公司名、岗位、投递方式（官网/内推/BOSS直聘）、当前状态、时间节点、面试反馈 |
| **系统输出** | ① 可视化看板（漏斗图+状态列表） ② 数据复盘报告 ③ 节点提醒 ④ 策略建议 |
| **输出格式** | 看板视图（可切换列表/看板/日历）+ 数据仪表盘 |

#### 详细交互流程

**步骤1 - 创建求职战役**
- 用户操作：点击"开启秋招作战室"，设置求职目标
  ```
  目标岗位：[Java后端]
  目标城市：[北京] [上海] [杭州] [多选]
  目标公司层级：[大厂优先] [中厂也考虑] [国企也考虑]
  开始时间：2025-08-01
  期望截止：2025-11-30
  ```
- 系统操作：生成空白看板，预置目标公司列表（可选从校招日历导入）

**步骤2 - 记录投递**
- 用户操作：点击"+ 记录投递"，输入：
  ```
  公司：字节跳动
  岗位：后端开发工程师
  部门：抖音电商（可选）
  投递渠道：官网直投 / 内推（内推人：张三） / BOSS直聘
  投递时间：2025-08-15
  简历版本：张三_后端_字节_v2（关联简历档案）
  备注：提前批，感觉匹配度80%
  ```
- 系统操作：在看板上创建卡片，状态设为"已投递"

**步骤3 - 状态流转**
- 看板列：已投递 -> 待笔试 -> 待一面 -> 待二面 -> 待三面 -> HR面 -> Offer/OC -> 已拒/已接受
- 用户操作：拖拽卡片到下一列，或点击卡片更新状态
- 状态变更触发：
  - "待笔试" -> 自动设置笔试倒计时提醒
  - "待一面" -> 触发面试准备建议（"距离一面还有3天，建议复习XX知识点"）
  - "Offer/OC" -> 触发Offer决策分析入口

**步骤4 - 面试反馈记录**
- 用户操作：面试后点击"记录反馈"
  ```
  面试轮次：二面
  面试形式：视频面试
  面试时长：45分钟
  问到的问题：
    1. 项目中的分布式锁怎么设计的？
    2. MySQL慢查询怎么优化？
    3. ...
  自我感觉：[很好] [一般] [较差]
  哪里挂了：（如已知）
  ```
- 系统操作：保存到面试档案，关联到对应公司卡片

**步骤5 - 数据复盘与策略建议**
- 系统操作：每周/每月自动生成复盘报告
  ```
  【求职周报】8月第3周
  
  [漏斗图]
  已投递：15  ->  笔试邀请：5（转化率33%）  ->  面试：2（40%）  ->  Offer：0
  
  [数据洞察]
  ⚠️ 简历过筛率33%，低于平均45%，建议优化项目描述
  ✅ 笔试通过率80%，表现良好
  ⚠️ 一面通过率50%，低于平均60%，模拟面试数据显示MySQL是薄弱项
  
  [本周行动建议]
  1. 优化简历项目描述（重点：量化指标+技术深度）
  2. 针对MySQL索引和优化专题训练（预计3小时）
  3. 投递节奏建议：本周再投5-8家，保持pipeline健康
  
  [下周面试提醒]
  • 周二 10:00 美团一面（倒计时2天）[查看准备建议]
  • 周五 14:00 京东二面（倒计时5天）[查看准备建议]
  ```

**步骤6 - 主动提醒**
- 系统操作：关键节点自动推送
  - "XX公司3天后截止投递，要投吗？"
  - "明天上午10点有美团面试，准备得怎么样？"
  - "距离上次投递已过7天，该投下一批了"
  - "XX公司你的内推人发了进度更新：已通过简历筛选"

#### 边界情况与异常处理

| 边界情况 | 处理策略 |
|---------|---------|
| 用户忘记更新状态 | 主动询问"上周投的字节有更新吗？点击更新状态" |
| 同一公司多岗位投递 | 支持子卡片嵌套，或分别记录标注"岗位A/岗位B" |
| 长时间无进展（如30天0面试） | 触发策略调整建议"可能需要：①降低目标层级 ②优化简历 ③扩大投递量" |
| 用户拿到多个Offer | 在看板中标记，自动推荐【Offer决策分析器】 |
| 用户被拒后心态崩了 | 触发【求职心理疏导】Agent："被拒是求职常态，看看数据：平均投递30+才拿1个Offer" |

#### AI Agent角色与工作流

**Agent角色**: "求职管家Agent"
- 人设：秋招拿到5个Offer的学长，擅长时间管理、进度把控、数据分析
- 核心能力：进度追踪 + 漏斗分析 + 节点提醒 + 策略建议

**工作流**:
```
投递记录 -> 看板状态更新 -> 节点触发（提醒/建议） -> 
定期数据汇总（漏斗转化率计算） -> 
与历史数据/平均水平对比 -> 
薄弱环节识别 -> 生成行动建议 -> 
主动推送提醒
```
